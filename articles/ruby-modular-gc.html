01

前提知識 — GCとは何か

ガベージコレクション（GC）はプログラムが使用しなくなったメモリを自動的に回収するメカニズムです。Rubyのようなマネージド言語では、GCはランタイムの中核を成す機能です。

GCプロセスは大きく3つのフェーズに分けられます：

  • Object Allocation — 新しいオブジェクトをメモリに割り当てる
  • Liveliness判定 — どのオブジェクトがまだ「生きている」（使用可能）かを判定
  • Reclaim（回収） — 不要なオブジェクトをメモリから解放

Liveliness判定の起点はRoot objectです。プログラムのスタック、グローバル変数、その他の参照可能なオブジェクトから始まって、参照グラフを辿っていき、到達不可能なオブジェクトを特定します。

なぜGCが必要か？ 手動メモリ管理（C/C++など）では、開発者が明示的にfree()やdeleteを呼び出す必要があります。これは以下のようなバグの温床となります：

  • 解放忘れ（メモリリーク）
  • 二重解放（ダブルフリーエラー）
  • 使用後解放（Use-After-Free）

Rubyはすべてがオブジェクトのため、GCなしには運用できません。メモリ安全性の確保と開発の効率性がGCの必須要件となっているのです。

02

Ruby の現行GC — Mark-Sweep-Compact

Free List Allocator

Ruby の標準的なメモリ割り当て方式はFree List Allocatorです。これは、解放されたメモリスロットを連結リストとして管理し、新規割り当て時に再利用するシステムです。

RubyのオブジェクトはRVALUE構造体として表現され、基本的には40バイトの大きさを持ちます：

  • 8バイト — flags（オブジェクトのメタ情報）
  • 8バイト — klass（クラスポインタ）
  • 24バイト — データ領域

Variable Width Allocation (VWA)は Ruby 3.2で導入された最適化です。従来はすべてのオブジェクトが40バイト割り当てられていましたが、VWAでは40B / 80B / 160B / 320Bというスロットサイズプールが導入されました。例えば、文字列は従来の23バイト制限から615バイトまで直接埋め込めるようになり、外部メモリ割り当てのコストが削減されています。

Mark-Sweep-Compact フェーズ

Ruby の Mark-Sweep-Compact GCは以下の3つのステップで構成されます：

  1. Mark Phase — Rootから参照グラフを深さ優先で辿り、到達可能なすべてのオブジェクトにマークを付与します。この過程で、参照先のオブジェクトが見つかると、再帰的にそれらもマークされます。
  2. Sweep Phase — ヒープ全体をスキャンし、マークされなかったオブジェクト（到達不可能と判定されたもの）のメモリを解放し、Free Listに登録します。
  3. Compact Phase — メモリ断片化を解消するために、カーソルを左右から走査してfreeスロットとオブジェクトを交換します。移動したオブジェクトにはT_MOVEDというセンチネルを配置し、参照を新しい位置に更新します。

詳細な実装解説は Peter Zhu の個人ブログ「blog.peterzhu.ca/notes-on-ruby-gc」で公開されており、Ruby GC理解の参考資料となっています。

03

他のGCアルゴリズム — トレードオフの世界

Ruby 標準GC以外にも、様々なGCアルゴリズムが存在し、それぞれ異なるトレードオフを持っています：

アルゴリズム 仕組み メリット デメリット
Reference Counting 参照追加/削除のたびにカウント更新 インクリメンタル、局所的コスト 循環参照に弱い、トラッキング負荷
Semispace Copying 2つの空間を用意、生存オブジェクトをコピー コンパクション自動、シンプル メモリ2倍必要、移動コスト
Immix ブロック→ライン階層、バンプポインタ割り当て メモリ効率とスループットのバランス 実装が複雑、研究段階
LXR Reference Counting + Immixのハイブリッド 低レイテンシ、高スループット 最新研究、検証中

重要な指摘として、Ruby自体はGCに最適化された設計ではないという点があります。C拡張機能との互換性や、既存のGCアルゴリズムへの依存性から、Reference Countingのような方式は不向きとされています。また、「One size fits all」のGCは存在しない

04

現行GCの限界

Parallelism（並列化）の課題

Ruby の現行GCはシングルスレッドに設計されています。Mark フェーズではオブジェクトの変更を防ぐ必要があり、Sweep フェーズでは Free List 構造への排他的アクセスが必須です。これらの制約により、複数スレッドでの並列GCは困難です。

Ractor（Ruby 3.1以降）の登場で並列処理の需要が高まっています。しかし共有オブジェクトのレースコンディション問題が生じ、並列GC実装が複雑化しています。

Dynamic Object Sizes の管理

現在のRubyは基本的に40バイトのRVALUEを確保し、超過分をmallocで別途割り当てています。VWAで改善されましたが、ImmixやLXRなどの他のGCアルゴリズムはより柔軟なサイズ管理を持っています。

スロットサイズが動的な場合の最適化は他の言語実装にはありますが、Rubyには専用の実装がありません。これは異なるGCアルゴリズムを実装する際の障壁になります。

05

Modular GC API — アーキテクチャの分離

Ruby Issue #20470で提案されたModular GC APIは、Ruby内核のGC実装を独立したモジュールとして扱うアーキテクチャです。

設計のポイント

従来の統合的なgc.cから機能を分離：

  • gc.c — 統合層（Ruby VMとの接点）
  • gc/default/default.c — デフォルトGC実装
  • gc_impl.h — パブリックAPI定義

GC実装は共有ライブラリ（.soファイル）として独立してビルドされ、実行時にdlopenでロードされます。

ビルドと実行

# Modular GC対応でビルド
./configure --with-modular-gc
make modular-gc MODULAR_GC=mmtk

# 実行時にGCライブラリを指定
RUBY_GC_LIBRARY=mmtk ruby script.rb
RUBY_GC_LIBRARY=/path/to/custom_gc.so ruby script.rb

デフォルトGCもlibgc_default.soとして独立ビルド可能です。GC実装はパブリックAPIのみを使用するため、Ruby内部への直接依存を排除できます。これにより、異なるGCアルゴリズムの試験的統合やカスタムGC開発が容易になります。

06

MMTk — 次世代GCフレームワーク

MMTkとは

Memory Management Toolkit (MMTk)は、VM非依存のGCフレームワークです。Rustで実装されたmmtk-coreとして提供され、V8（JavaScript）、JVM、OpenJDK等の複数のVMで採用されています。

開発はANU（オーストラリア国立大学）の Steve Blackburn 研究室が主導しており、学術的な最新GC研究を実装面で実現しています。Ruby統合はruby/mmtkリポジトリで進行中です。

アーキテクチャ

Ruby ← C Binding ← Rust Binding ← MMTk core という階層構造により、RubyとMMTkの間を疎結合に保っています。

利用可能なGCプラン

  • NoGC — GCしない（テスト用）
  • MarkSweep — Ruby標準GCに類似
  • Immix — ブロック/ライン階層ベースの効率的GC
  • StickyImmix — 世代別Immix（新しい世代を重点的に）

LXRもOpenJDKで実績を持っており、PLDI 2022で発表された研究によると、ShenandoahやZGCより低レイテンシ、G1より高スループットを実現しています。

利点： MMTkを活用することで、研究者はRubyの内部実装を深く理解しなくても、新しいGCアルゴリズムをRuby上で直接実験できるようになります。学術的GC研究と本番実装のギャップが大幅に縮小されるのです。

07

現在の状況と制約

Ruby 3.4での搭載

Ruby 3.4（2024年12月リリース）には、Modular GC APIとMMTk統合がexperimental機能として搭載されました。この実装により、開発者はRubyの選択肢を広げることができます。

現在の制限事項

  • ⚠️ パフォーマンス — デフォルトGCより大幅に劣る。本番環境での使用は非推奨
  • 並列化 — GCは現在シングルスレッド。マルチスレッド環境での最適化は未実装
  • アルゴリズムサポート — MMTkではMarkSweep、Immix程度の基本実装のみ
  • 互換性 — C拡張機能との完全な互換性がまだ確認されていない場合がある

今後の改善課題

  • 並列マーキングと並列スイープの実装
  • Ractor対応の並列GC
  • より高度なアルゴリズム（LXR等）のサポート
  • パフォーマンスのデフォルトGCとのパリティ達成

⚠️ 注意： Modular GCは現在experimentalです。本番環境では必ずデフォルトGCを使用してください。テスト環境での評価と フィードバック提供が、この機能の成熟化に貢献します。

08

今後の展望

Modular GCエコシステムは、まだ初期段階ですが、複数の展望があります：

  • 並列GCの実現 — Ractor対応と連携し、マルチスレッド環境での真の並列GCを実現
  • MMTk GCプランの拡充 — LXR、Z Garbage Collector相当など、最新アルゴリズムのサポート
  • パフォーマンス最適化 — デフォルトGCとのパリティ達成に向けた精度向上
  • カスタムGCエコシステム — 企業やコミュニティが独自のGC実装を提供し、特定ワークロード（高レイテンシ要件、メモリ制限環境等）に最適化
  • 実装ガイドの整備 — ruby/gc/README.mdにGC開発者向けの詳細ガイドを拡充し、参入障壁を低減

これらが実現すれば、Ruby は単なる「デフォルトGC搭載言語」から「GC選択可能な言語」へと進化します。

09

まとめ

Modular GCは、Rubyのメモリ管理アーキテクチャに根本的な転換をもたらします。

  • GCアルゴリズムの選択肢 — ワークロードに応じた最適なGC選択が可能に
  • 研究と本番の融合 — MMTkを通じて、学術的GC研究をRuby実装へ直接適用
  • カスタムGC開発 — 企業や開発者が独自ニーズに応じたGC実装を容易に構築可能
  • 言語としての成熟化 — メモリ管理の自由度向上により、Ruby の適用範囲が拡大

現在 experimental ですが、このModular GCの推進により、Ruby はメモリ管理の分野で大きな前進を遂げるでしょう。
