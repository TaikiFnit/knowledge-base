01

AtCoderのRuby実行環境

AtCoderで現在利用できるRubyはRuby 3.2.2（MRI）で、--jit --yjit-exec-mem-size=128オプション付きで実行される。つまりYJITが有効だ。ただし実行時間制限はC++と完全に同一（多くの問題で2秒）であり、言語ごとの補正は一切ない。

利用可能なライブラリ
競プロで重要なac-library-rb（ACLのRuby移植版）が使える。DSU、セグメント木、遅延セグ木、BIT、優先度付きキュー、最大フロー、SCC、2-SATなどを網羅。加えてrbtree、sorted_set、numo-narrayも利用可能。
2025年秋の言語アップデート
Ruby 3.4への更新に加え、TruffleRuby（GraalVMベースのJIT実装）が新規追加予定。MRIより大幅な高速化が見込まれ、「想定解法なのにTLE」の頻度が減る期待がある。
02

ループ構文の選択が生死を分ける

Rubyにおける最大の高速化ポイントはループ構文の選択だ。times、each、for、loopはブロック呼び出しのオーバーヘッドを伴うが、whileはブロックを使わないため大幅に速い。

構文 実行時間 相対速度
while 0.031s
times 0.050s
each 0.051s
for 0.054s
loop 0.070s

whileはloopの約2倍速、timesの約1.6倍速。二重ループの計測ではN=5000のときforが1002msに対しwhileは615msで、2秒制限でAC/TLEを分ける決定的な差となる。

✗ 遅い
n.times { |i| process(i) }
✓ 速い
i = 0
while i < n
  process(i)
  i += 1
end
目安
計算量が106を超えるループではwhileへの書き換えが必須。多重ループで効果は倍増する。
03

入出力・文字列・配列の落とし穴

入出力の高速化

出力関数の選択で3〜6倍の差がつく。pが1913msに対しputsは499ms、文字列にためて一括出力すると308msまで短縮される。入力は行ごとのgetsより$stdin.readで一括読み込みが高速。

高速入出力テンプレート 
# 高速入力
N, M, *AS = $stdin.read.split.map(&:to_i)

# 高速出力
buf = ""
results.each { |r| buf << "#{r}\n" }
puts buf

文字列 → 数値配列に変換せよ

文字列の添字アクセスは配列の約3倍遅い。重要度★★★★★のテクニックが、文字列をbytes配列に変換して処理する方法だ。

✗ 遅い：文字列のまま
s = gets.chomp
s[i] == "#"
✓ 速い：bytes変換
s = gets.chomp.bytes
s[i] == 35  # '#'

配列操作の罠

✗ 一時配列を毎回生成
ans = [ans, result].max
✓ 直接比較
ans = result if result > ans

BFSのキューはArray#shiftではなく、インデックス管理方式が最も安全かつ高速。二次元配列アクセスでは、内側の配列を変数に取り出すだけで約7%の高速化が見込める。

BFSキューの正しい実装 
queue = [start]
head = 0
while head < queue.size
  v = queue[head]
  head += 1
  # ... queue.push(next_v) で追加
end
04

多倍長整数とGCが引き起こす見えないTLE

Rubyは整数オーバーフローが起きず自動的にBignum（多倍長整数）に拡張される。安全だが、Bignumの演算はFixnumより桁違いに遅い。剰余を取らずに巨大整数を計算し続けるとTLE/MLEの原因になる。

✗ 致命的に遅い
power = (i ** j) % MOD
✓ 組み込み繰り返し二乗法
power = i.pow(j, MOD)

Integer#pow(exp, mod)は内部でC実装の繰り返し二乗法を使うため、自前実装より圧倒的に速い。この差は「計算不能 vs 一瞬」レベル。

frozen_string_literal
ファイル先頭に# frozen_string_literal: trueを記述すると、文字列リテラルが再利用されGC圧力が大幅に減少する。puts "Yes"を100万回実行で289ms → 213msへ改善。
05

C++では余裕だがRubyでTLEになるパターン

ダイクストラ法 / 優先度付きキュー

Rubyの標準ライブラリには優先度付きキューがない。毎回sortして最小値を取り出す実装は壊滅的に遅い。ac-library-rbのPriorityQueueを使うか、辺の重みが少数種類なら01-BFSで優先キューごと回避する。

セグメント木 / BIT

再帰型は非再帰型より遅い（メソッド呼び出しオーバーヘッド）。非再帰型またはac-library-rbのSegtreeを使うべき。BITで済む場面ならBITの方が定数倍が軽い。

2次元DPとO(N²)解法

重要
N≤5000のO(N²)解法はRubyではほぼ通らない。N=2000のN²=4×10⁶がwhileでギリギリ。配列初期化もArray.new(n, 0)はブロック付きの10倍速。2次元配列を1次元にフラット化し、MOD演算でBignumを回避する工夫が必要。

Union-Find / BFS / DFS

再帰DFSはメソッド呼び出しオーバーヘッド＋スタック制限の二重苦。明示的スタックによる非再帰化が鉄則。Union-Findの経路圧縮もwhileループで実装すべき。

O(N√N) 以上は実質不可能

N=10⁵でN√N≈3×10⁷はRubyの実質上限を超える。logを1つ落とせないか（二分探索→しゃくとり法など）検討する価値がある。

06

レーティング帯別のRuby適性

灰〜茶色
全く問題なし。便利メソッド群（gcd, bsearch, pow）はむしろ有利
緑〜水色
一部TLEが出始めるが、定数倍高速化テクニックで対処可能
青
RubyのみではACできない問題が増加。Crystal/C++の併用を推奨
黄色以上
Rubyのみでは極めて厳しい
Crystalという選択肢
「Fast as C, Slick as Ruby」を掲げるCrystalは、Rubyとほぼ同じ文法でC言語並みの速度を実現。kona0001氏（青コーダー）は「約30問解けば慣れる」と報告。ただし多倍長整数がなく、型指定・Nilチェックが必要。
07

チェックリスト

提出前に確認すべき最重要テクニック。これらを徹底すればC++比10倍のハンデを相当程度埋められる。

❶ 計算量10⁶超のループをwhileに書き換えたか
❷ 入力は$stdin.readで一括読み込みしているか
❸ 文字列を.bytesで数値配列に変換しているか
❹ MOD演算をこまめに行いBignumを回避しているか
❺ データ構造はac-library-rbを活用しているか
❻ 再帰DFSを明示的スタックの非再帰に書き換えたか
❼ [x,y].min/maxをif文の直接比較に置き換えたか
❽ BFSキューをインデックス管理方式にしているか
