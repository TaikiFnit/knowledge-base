Chapter 1

Data Mesh とは何か（ナットシェル）

コア定義

Data Mesh = 複雑・大規模環境において、分析用データ（Analytical Data）の共有・アクセス・管理を行うための、分散型ソシオテクニカル（社会技術的）アプローチ。

「分析用データ」= レポーティング・診断・機械学習に使われるデータ。運用系（トランザクション）データとは区別される。

Data Mesh が目指す3つのアウトカム

⚡

Agility at Scale

規模でのアジリティ — 大規模環境でもスピードを維持できる俊敏性

🔬

Data-Driven Innovation

データ駆動のイノベーション — データを活用した継続的な価値創造

🌐

Data Democratization

データの民主化 — 組織全体でデータにアクセス・活用できる状態

3つのパラダイムシフト

従来のパラダイム → Data Mesh のパラダイム
中央集権型データ管理 → ドメインオーナーシップへ分散
データを業務の副産物として扱う → データをプロダクトとして設計・提供
ビスポーク（一点物）パイプライン → セルフサーブ・プラットフォーム
アドホックなガバナンス → 連合型計算ガバナンス（自動化）

4つの相互依存する原則

Data Mesh は以下4原則が相互依存している。どれか一つを欠いても機能しない。

① ドメインオーナーシップの原則

データの責任をそのデータを最もよく知るドメインに分散させる。これが基盤となる原則。

② データをプロダクトとして扱う原則

データをプロダクト思考で管理。コンシューマーが顧客であり、品質・発見可能性を保証する。

③ セルフサーブ・データプラットフォームの原則

ドメインが自律的にデータプロダクトを構築できるインフラ基盤を提供する。

④ 連合型計算ガバナンスの原則

意思決定を連合体が行い、ポリシーをコードとして自動適用する。

Data Product Quantum（DPQ）

DPQ は Data Mesh における最小の独立したデプロイ可能ユニット。3つの要素が三位一体で機能する。

⚙️

Code

データプロダクトのロジック・変換処理・ポリシー実装

🗄️

Data / Metadata

実際のデータと、その意味・文脈・品質を記述したメタデータ

🏗️

Infrastructure

動作に必要なコンピューティング・ストレージ・ネットワーク基盤

Data Mesh の起源：3世代の失敗

Dehghani は、データアーキテクチャ3世代の失敗パターンから Data Mesh を導出した。根本原因は技術的ではなく、組織的なものだった。

Gen 1

データウェアハウス（DWH）時代

中央の DWH チームへのボトルネック。変化への適応が遅く、スケール不可能。ビジネス部門の要望に応えるには中央チームを介す必要があった。

Gen 2

データレイク時代

ガバナンス欠如による「データの沼（Data Swamp）」化。データは蓄積されたが品質・文脈が失われ、利用価値が低下した。

Gen 3

クラウドプラットフォーム時代

テクノロジーは進化したが組織構造は変わらなかった。中央集権的なデータチームへの依存が続き、サイロと遅延の問題は解消されなかった。

ソフトウェアエンジニアリングからの教訓

Data Mesh は、マイクロサービスアーキテクチャと DDD（ドメイン駆動設計）が解決した問題を、分析データ領域に適用する。チームの自律性・責任の局所化・明確なインターフェース — これらをデータに持ち込む。

Chapter 2

ドメインオーナーシップの原則

データの責任をそのデータを最もよく知る人たち（= データを生み出すドメイン）へ分散・移譲する。これは Data Mesh の基盤原則であり、この原則なしに残りの3つは成立しない。

分解の軸：テクノロジーではなくビジネスドメイン

従来の分割軸（技術的境界）

「インジェスチョンチーム」「変換チーム」「BI チーム」など、技術的な機能で組織が縦割りされていた。データの意味・文脈はどこにも残らない。

Data Mesh の分割軸（ドメイン境界）

DDD のバウンデッドコンテキストに沿って責任を分割。「注文ドメイン」「顧客ドメイン」などビジネスの実体に対応した責任体制。

3種類のドメインデータアーキタイプ

TYPE 01

Source-Aligned（ソース整合型）

業務システムの実データに近い形で提供。データの生成元ドメインが責任を持つ。

例：注文ドメインの生注文データ、ユーザーイベントログ

TYPE 02

Aggregate（集約型）

複数のソースドメインから集約・統合したデータ。組織横断的な視点を提供する。

例：全社売上サマリー、顧客360ビュー

TYPE 03

Consumer-Aligned（コンシューマー整合型）

特定のコンシューマー（MLモデル、BI、外部API）のユースケースに特化して設計。

例：MLモデル学習用の特徴量データセット

データオーナーシップの上流化

データオーナーシップを「上流」（= データを生み出す側）に押し上げることが重要。中央のデータチームへの丸投げではなく、各ドメインが自分たちのデータに責任を持つ体制を構築する。

反パターン：データパイプラインの外部公開

各ドメインのデータパイプラインは、そのドメインの内部実装の詳細として隠蔽すべき。外部（他ドメイン）はパイプラインを意識せず、公開されたデータプロダクトのみを利用する。

単一真実源（Single Source of Truth）への執着を捨てる

Data Mesh は「唯一の真実」という概念を否定する。代わりに、複数のコンテキスト的な真実を受け入れる。異なるドメインが異なる視点から同じ現実を表現することを許容し、それぞれのユースケースに最適化された表現を持つ。

なぜ「基盤原則」なのか

ドメインオーナーシップがなければ、「データをプロダクト」にする責任者がいない。セルフサーブプラットフォームを誰も使わない。ガバナンスに参加する主体が存在しない。この原則が他の3つを成立させる土台となる。

Chapter 3

データをプロダクトとして扱う原則

課題：データフリクション問題

Anaconda 2020年レポートによると、データサイエンティストの約50%の時間がデータ前処理に費やされている。アクセスが困難、品質が不明確、文書化が不十分 — これらが「データフリクション」を生む。

ドメインオーナーシップだけでは、データサイロをさらに悪化させる可能性がある。そこで必要なのがこの原則。

プロダクト思考の適用

データコンシューマー = 顧客

データを利用する人・チーム・システムは「顧客」として扱われる。顧客体験（= データへのアクセスと利用体験）を中心に設計する。

データプロダクトオーナーの設置

各ドメインにデータプロダクトオーナーを置く。ライフサイクル全体・品質基準・コンシューマー体験・継続的改善の責任者。

副産物からプロダクトへ

データは「業務システムの副産物」から「オーナーシップ・品質基準・継続改善を伴うプロダクト」へと変容する。

8つのユーザビリティ特性

高品質なデータプロダクトが備えるべき8つの特性。これらすべてを満たすことで「使える」データプロダクトとなる。

01

Discoverable（発見可能）

データカタログや検索機能で見つけられる。存在を知らなければ使えない。

02

Addressable（アドレス可能）

一意のアドレス（URI）でプログラマティックにアクセスできる。

03

Understandable（理解可能）

データの意味・文脈・ビジネス定義が明確に文書化されている。

04

Trustworthy & Truthful（信頼可能・真実）

SLO（品質目標）が定義され、品質が継続的に計測・保証されている。

05

Natively Accessible（ネイティブアクセス可能）

標準的な方法（SQL、API、ファイル）で直接アクセスできる。

06

Interoperable & Composable（相互運用・組み合わせ可能）

他のデータプロダクトと連携・組み合わせて新しい価値を生み出せる。

07

Valuable on Its Own（単独で価値がある）

他のプロダクトに依存せず、単体として意味のある価値を提供する。

08

Secure（セキュア）

アクセス制御が適切に実装され、コンプライアンス要件を満たしている。

Data Product Quantum（DPQ）の再確認

データプロダクトの最小デプロイ可能ユニット。コード + データ/メタデータ + インフラの三位一体が揃って、はじめて独立して機能するデータプロダクトとなる。DPQ の境界が、責任の境界でもある。

変容の核心

この原則によって、データは「誰かが作った副産物をたまたま使う」から「オーナーが責任を持って提供し、コンシューマーが信頼して使う」へと変容する。これがデータ品質改善の根本的なインセンティブ構造を生む。

Chapter 4

セルフサーブ・データプラットフォームの原則

ドメインチームが自律的にデータプロダクトを構築・デプロイ・維持できるよう、インフラの複雑性を抽象化する。プラットフォームなしにドメインオーナーシップを要求することは、ドメインチームに過大な負担を強いる。

設計目標

ドメインの自律性を実現

インフラ専門知識がなくてもデータプロダクトを構築・運用できる環境を提供する。

複雑性の隠蔽

セキュリティ、スケーリング、モニタリングなどのクロスカッティング関心事を自動的に処理する。

専門化の障壁を下げる

分散ストレージやクエリエンジンの専門知識がなくても、宣言的なインターフェースで操作できる。

ドメイン非依存（汎用）

どのドメインも同じプラットフォームを使える。特定ドメイン向けのカスタマイズは不要。

3層のケイパビリティ構成（マルチプレーンアーキテクチャ）

プラットフォームは以下の3層（プレーン）で構成される。上位層ほど抽象度が高く、エンドユーザーに近い。

PLANE 3 / TOP 全組織・ガバナンスチームが利用

Data Mesh Supervision Plane（メッシュ監視プレーン）

グローバルなメッシュレベルのケイパビリティを提供。組織全体にわたる横断的な機能。

データ発見（Discovery） セマンティッククエリ データカタログ グローバルセキュリティ 監視・アラート SLO モニタリング
↕ 機能の依存・呼び出し
PLANE 2 / MIDDLE ドメインチームの開発者が日常的に利用

Data Product Developer Experience Plane（DX プレーン）

ドメインチームがデータプロダクトのライフサイクルを宣言的に管理できるインターフェース。クロスカッティング関心事を自動実装。

宣言的インターフェース スキーマレジストリ パイプラインオーケストレーション データリネージュ テスト自動化 デプロイメント管理
↕ 機能の依存・呼び出し
PLANE 1 / BOTTOM 他プレーンまたは上級開発者が利用

Data Infrastructure Provisioning Plane（インフラプレーン）

最下位の基盤層。分散処理・ストレージ・アクセス制御の実装を担う。他のプレーンの動作を支える土台。

分散ストレージ ACL（アクセス制御） オーケストレーション基盤 クエリエンジン コンピューティング ネットワーク

プラットフォームチームの役割

プラットフォームチームはプロダクトチームとして運営する。ドメインチームが「顧客」であり、プラットフォームの開発優先順位はドメインのニーズに基づいて決定される。

現状の技術的課題

運用系テックスタック（Docker / Kubernetes）とデータ系テックスタック（Spark / Databricks）の乖離が現在の主要な課題。この2つの世界をシームレスに統合することが、真のセルフサーブ実現の鍵となる。

連合型計算ガバナンスの原則

この原則は、データガバナンスの二項対立を解消する。

❌ 中央集権型ガバナンス

1つの中央チームが全ポリシーを決定・適用。スピードが遅く、スケールしない。ボトルネックになる。

❌ 無政府状態（データアナーキー）

各ドメインが好き勝手にやる。互換性なし、コンプライアンス違反、セキュリティ崩壊。

✅ 連合型計算ガバナンス

各ドメインの代表者が連合体を形成し、グローバル基準を合意・コードで実装・自動適用する。

「連合型（Federated）」の意味

意思決定をドメインのデータプロダクトオーナーとプラットフォームプロダクトオーナーの連合体が行う。各ドメインの代表者が集まって決定するギルドモデル。中央チームの独裁でも、各ドメインの独走でもない。

「計算型（Computational）」の意味

ポリシーをコードとして実装し、プラットフォームが自動的に適用・強制する。人手による文書ベースのガバナンスではなく、機械が実行可能なポリシーとして表現する。

// ポリシーの例（概念的表現） policy "data-product-security" { require: encryption_at_rest = true require: access_control = role_based require: pii_fields.masked = true slo: freshness_sla <= 24h slo: availability >= 99.9% } // → プラットフォームが自動的にすべてのDPQに適用・検証

グローバル基準 vs ローカル決定

種別 決定主体 内容例
グローバル基準
全ドメイン共通		 連合体（全ドメイン合意） 相互運用性規格、セキュリティポリシー、コンプライアンス、SLO定義の枠組み、識別子体系
ローカル決定
各ドメイン自律		 各ドメインのDPO ドメインデータモデル、内部ビジネスロジック、スキーマ設計、ドメイン固有のSLO値

成功指標：ネットワーク効果

ガバナンスの成功は「管理されたデータの量」ではなく、データ消費の接続数（ネットワーク効果）で測定する。データプロダクトが広く使われ、組織全体でデータの流れが増えているかが真の指標。

エラー対応の哲学転換

旧来（予防主義）： 事前の手動チェック・承認プロセスでエラーを防ぐ → 遅く、スケールしない

Data Mesh（検出と回復）： 自動化された検知・修正で素早く対応 → 速く、スケール可能

ポリシーの Git 管理

ガバナンスポリシーはコードとして、Gitで管理する。これにより：

  • バージョン管理： ポリシーの変更履歴を追跡可能
  • コードレビュー： Pull Request で変更の審査プロセスを実施
  • テスト： ポリシーの自動検証・CI/CD パイプラインで品質保証
  • ロールバック： 問題があれば即座に以前のバージョンに戻せる

Pre-Data-Mesh vs Data Mesh：ガバナンス比較

観点 従来（Pre-Data-Mesh） Data Mesh
チーム構成 中央集権チーム（専任ガバナンスチーム） 連合型チーム（各ドメイン代表 = ギルド）
適用方法 手動（文書レビュー・承認プロセス） 自動化（コードとしてプラットフォームが強制）
ポリシー形態 静的ドキュメント（PDF・Wiki） 動的コード（Git管理・CI/CD対応）
意思決定 トップダウン（中央が一方的に決定） ボトムアップ＋合意形成（ドメイン参加型）
エラー対応 予防重視（事前チェック・ゲートキーパー） 検出と回復（自動検知・自動修正）
スケーラビリティ スケールしない（中央チームがボトルネック） スケール可能（自動化 + 分散意思決定）
変更速度 遅い（承認待ち、会議依存） 速い（コードレビュー + 自動デプロイ）
成功指標 管理されたデータ量、コンプライアンス率 データ消費接続数（ネットワーク効果）

Summary

4原則の相互関係と学習から得た洞察

Data Mesh の4原則は独立した施策ではなく、相互依存する一つのシステムを形成している。どれか一つを欠いても全体が機能しない。

①

ドメインオーナーシップの原則

基盤原則。データの責任とオーナーシップをドメインへ分散させる。これなしに残りの3原則は成立しない。

→ ②のオーナーを生み出す / ③の利用者を定義する
②

データをプロダクトとして扱う原則

ドメインオーナーシップだけでは生まれるサイロを解消。データに品質・発見可能性・継続改善を組み込む。

→ ④のガバナンス対象となるプロダクトを定義する
③

セルフサーブ・データプラットフォームの原則

ドメインチームに自律性を与える基盤。これなしにドメインへの責任移譲は過大な負担になる。

→ ④のポリシーを「計算的」に実行する手段を提供する
④

連合型計算ガバナンスの原則

中央集権とアナーキーの間の均衡点。グローバル基準を保ちながらドメインの自律性を尊重する。

→ ①②③が揃って初めて意味を持つ統治機構

依存関係の整理

依存関係 なぜ依存するか
① → ② ドメインが責任を持つことで、はじめてデータをプロダクトとして設計・維持するインセンティブが生まれる
① → ③ プラットフォームなしにドメインに責任を持たせるのは無理。自律性を与えるためのインフラが不可欠
② → ④ データプロダクトが存在してはじめてガバナンスの対象が明確になる。ガバナンスはプロダクトの品質を保証する仕組み
③ → ④ プラットフォームがガバナンスポリシーの「計算的」な実行（自動適用）を担保する。プラットフォームなしにポリシーの自動化は不可能
本質的な洞察

Data Mesh の革新性は技術アーキテクチャではなく、組織・カルチャーの設計にある。3世代のデータアーキテクチャが失敗した根本原因（中央集権的な組織構造）に対して、ソフトウェアエンジニアリングで実証済みの知見（DDD・マイクロサービス）を援用して解決しようとしている。

テクノロジーを変えるだけでは問題は解消されない。誰がデータに責任を持つかという組織設計こそが、データアーキテクチャの本質的な問いとなる。
