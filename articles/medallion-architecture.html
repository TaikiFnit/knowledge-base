01

Medallion Architectureとは — 概要と歴史

Medallion Architectureは、2020年にDatabricksが提唱したデータレイクハウスの標準アーキテクチャです。生データから分析可能な形式へと段階的に洗練させる3層構造（Bronze・Silver・Gold）によって、スケーラビリティ、データ品質、ガバナンスの課題を同時に解決します。

従来のデータウェアハウス（DWH）は高価で柔軟性に欠け、データレイク（Data Lake）は構造化されず「沼」となってしまう傾向がありました。Medallion Architectureは両者の長所を取り入れ、クラウドネイティブなスケーラビリティと OLAP スタイルの分析が共存する新しいパラダイムを実現します。

Key Insight: Medallion Architectureの本質は「段階的な変換と品質向上」です。各層で明確な責任を持つことで、データパイプラインは複雑さを管理しやすくなり、デバッグやメンテナンスが格段に簡単になります。

歴史的背景： Databricksが2020年のブログポスト『Medallion Architecture』で発表して以来、クラウドデータベース業界の事実上の標準となりました。Snowflake、BigQuery、Redshift など主流プラットフォームでも同パターンの採用が推奨されるようになり、データエンジニアリングの共通言語として認識されています。

Medallion Architecture の基本原理

  • Bronze層： ソースシステムから のデータを「ありのまま」保存。スキーマはソースに準じ、データ品質チェックはまだ行わない。
  • Silver層： Bronze から取り込んだデータを クレンジング・標準化・統合。ビジネスロジック適用前の「きれいな単一ソース」を目指す。
  • Gold層： Silver からさらに集約・非正規化し、ビジネスユースケースに最適化したビュー・テーブルを提供。ダッシュボードやML、分析に即座に使える形。

この3層モデルにより、各ユースケースに応じた柔軟な設計、透明性のある監査証跡、スケーラブルで保守性の高いパイプラインが実現されます。

02

Bronze層 — 生データの取り込みと保存

Bronze層はすべてのデータの「入口」です。ソースシステム（ERP、CRM、ログシステム、IoT センサ など）から のデータを原則として変更なく保存します。エラーレコードや古いデータも含めて、「捨てない」という原則が重要です。

取り込みパターン

パターン 説明 用途
バッチ 毎日・毎時間など定期的にデータをコピー 売上データ、マスターデータ
ストリーミング イベントが発生した瞬間に取り込み（Kafka、Kinesis等） ログ、トランザクション、リアルタイム分析
CDC 変更捕捉（Change Data Capture）で差分のみ取得 データベースの増分レプリケーション
API REST/GraphQL経由で外部サービスから取得 SaaS統合、第三者データ

Bronze層の特性

  • スキーマ・オン・リード： 取り込み時点では型チェック・変換を最小限にし、読み込み側でスキーマ解釈を行う柔軟性を保つ。
  • アペンドオンリー： 一度書き込んだデータは上書きせず、タイムスタンプ付きで保存。履歴追跡とデバッグが容易。
  • メタデータ保持： 取り込み日時、ソースシステム、バージョンなどのメタデータを付与。
  • 大量保存容易性： ストレージコストが低いため、「もしかして必要かもしれない」データも保存する余裕が生まれる。

Pythonコード例：バッチ取り込み

# PySpark を使ったCSVファイル → Bronze の取り込み
from pyspark.sql import SparkSession
from datetime import datetime

spark = SparkSession.builder.appName("bronze_ingest").getOrCreate()

# ソースCSVを読み込み
source_df = spark.read.csv(
    "/source/sales_data.csv",
    header=True,
    inferSchema=True
)

# メタデータカラムを追加
from pyspark.sql.functions import lit, current_timestamp
bronze_df = source_df.withColumn(
    "ingestion_date", current_timestamp()
).withColumn(
    "source_system", lit("SAP_ERP")
).withColumn(
    "version", lit("1.0")
)

# Bronze に書き込み（append mode）
bronze_df.write.mode("append").format("delta").save(
    "/lakehouse/bronze/sales"
)

print(f"Ingested {source_df.count()} records to Bronze")
Best Practice: Bronze層はストレージ効率よりも「取得の確実性」を優先します。圧縮はしても、フォーマット変換は最小限に。エラーハンドリングは Silver で行うメンタルモデルを持つことが大切です。

CDC（Change Data Capture）の取り込み例

# Debezium/Kafka 経由でオンプレデータベースの変更を取り込み
# CDC フィードから Bronze へ
cdc_stream = spark.readStream.format("kafka").option(
    "kafka.bootstrap.servers", "kafka:9092"
).option(
    "subscribe", "db_changes"
).option(
    "startingOffsets", "earliest"
).load()

# JSON形式のCDCメッセージをパース
from pyspark.sql.functions import from_json, col
parsed = cdc_stream.select(
    from_json(col("value").cast("string"),
              "op STRING, table STRING, before MAP, after MAP").alias("cdc")
).select("cdc.*")

# Bronze に書き込み（streaming write）
query = parsed.writeStream \
    .format("delta") \
    .option("checkpointLocation", "/lakehouse/checkpoints/cdc") \
    .outputMode("append") \
    .start("/lakehouse/bronze/customers_cdc")

query.awaitTermination()
03

Silver層 — データ品質とクレンジング

Silver層は Bronze の粗データを精製し、組織全体で信頼できる「単一の正解」（SSOT: Single Source of Truth）を実現します。ここが最もエンジニアリング負荷が高い層であり、データ品質の大半はこの層で担保されます。

Silver層で実施する主な処理

  • データクレンジング： NULL値の処理、形式の統一、外字・特殊文字の除去。
  • 重複排除： データソース側の重複や、取り込みエラーによる重複を検出・削除。
  • スキーマ標準化： 複数のソースを統一スキーマにマッピング。日付形式、通貨、単位などの統一。
  • データ型の厳密化： スキーマ・オン・リードから スキーマ・オン・ライト へ。型定義を明示。
  • ビジネスキーの定義： 自然キー（顧客ID、注文番号など）を明確にし、SCD（Slowly Changing Dimensions）対応を準備。
  • 品質メトリクスの計算： 記録数、NULL比率、異常値フラグなどを記録。

Silver層の設計パターン

パターン 説明 メリット
テーブル毎 bronze/customers → silver/customers シンプル、変更影響が限定的
ドメイン毎 bronze/sales, bronze/inventory → silver/crm_unified クロス・ドメイン統合が容易
階層化 bronze → silver_raw → silver_cleaned → gold 段階的品質向上、中間ステップでのデバッグ容易

SQLコード例：顧客マスターの標準化

-- Bronze 顧客データから Silver へのETL
CREATE OR REPLACE TABLE silver.customers AS
SELECT
    -- ビジネスキー（重複排除に使用）
    customer_id,

    -- クレンジング済みカラム
    LOWER(TRIM(first_name)) as first_name,
    LOWER(TRIM(last_name)) as last_name,
    CAST(birth_date AS DATE) as birth_date,
    CASE
        WHEN email LIKE '%@%.%' THEN LOWER(TRIM(email))
        ELSE NULL
    END as email_validated,

    -- 正規化：電話番号をE.164形式へ
    REGEXP_REPLACE(phone_number, '[^0-9+]', '') as phone_e164,

    -- ステータスの標準化
    CASE
        WHEN status IN ('ACTIVE', 'A', 'Y', '1') THEN 'ACTIVE'
        WHEN status IN ('INACTIVE', 'I', 'N', '0') THEN 'INACTIVE'
        ELSE 'UNKNOWN'
    END as status_normalized,

    -- 品質指標
    CASE
        WHEN email IS NULL OR phone_number IS NULL THEN TRUE
        ELSE FALSE
    END as has_missing_contact,

    -- 監査カラム
    FROM_UNIXTIME(created_timestamp) as created_at,
    FROM_UNIXTIME(updated_timestamp) as updated_at,
    current_timestamp() as silver_processed_at,

    -- ハッシュ：行全体の変更検知用（SCD実装に必要）
    MD5(CONCAT_WS('|',
        first_name, last_name, email, phone_number, status
    )) as row_hash

FROM bronze.customers
WHERE
    -- エラーレコードをスキップ（フラグ付けは金）
    customer_id IS NOT NULL
    AND created_timestamp > 0
ORDER BY customer_id, updated_timestamp DESC
QUALIFY ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY updated_timestamp DESC) = 1;

PySpark：SCD Type 2（履歴追跡）の実装

from pyspark.sql.functions import *
from pyspark.sql.window import Window

# Silver から新しい顧客データを取得
new_customers = spark.read.table("silver.customers_staging")

# 現在の SCD テーブルを読み込み
try:
    current_scd = spark.read.table("silver.customers_scd_type2")
except:
    # 初回実行時は空のテーブルを作成
    current_scd = spark.createDataFrame([], """
        customer_id STRING,
        first_name STRING,
        email_validated STRING,
        status_normalized STRING,
        row_hash STRING,
        valid_from DATE,
        valid_to DATE,
        is_current BOOLEAN
    """)

# 既存レコードと新規データをマージ
merged = new_customers.join(
    current_scd.where(col("is_current") == True).select(
        "customer_id", "row_hash"
    ),
    "customer_id",
    "left_outer"
)

# ハッシュが異なる = 属性が変わったレコード
changed = merged.filter(
    (col("row_hash") != col("row_hash")) | (col("row_hash").isNull())
)

# 既存の "is_current" レコードを "valid_to" を今日に設定して非活性化
to_close = current_scd.where(
    current_scd.customer_id.isin([row.customer_id for row in changed.select("customer_id").collect()])
    & (col("is_current") == True)
).withColumn(
    "valid_to", current_date()
).withColumn(
    "is_current", lit(False)
)

# 新しいレコードを挿入（valid_from = 今日、valid_to = 9999-12-31）
new_records = changed.select(
    "customer_id", "first_name", "email_validated", "status_normalized", "row_hash"
).withColumn(
    "valid_from", current_date()
).withColumn(
    "valid_to", lit("9999-12-31").cast("date")
).withColumn(
    "is_current", lit(True)
)

# 変更のなかったレコードはそのまま
unchanged = current_scd.join(
    new_customers.select("customer_id", "row_hash"),
    ["customer_id", "row_hash"],
    "inner"
)

# 3つを union して新しい SCD テーブルを作成
final_scd = (
    to_close.union(new_records).union(unchanged)
)

final_scd.write.mode("overwrite").saveAsTable("silver.customers_scd_type2")
Deep Dive: SCD（Slowly Changing Dimensions）はデータウェアハウジングの重要な概念です。Type 1（上書き）は履歴喪失、Type 2（履歴追跡）は行数増加のトレードオフがあります。ビジネス要件に応じて使い分けることが重要です。

Gold層 — ビジネスに最適化されたデータ

Gold層は分析やMLの「直前」のステージです。ここでのデータは ユーザーがすぐに使える形 に最適化されています。集約、非正規化、キャッシング など、パフォーマンスとビジネス要件を両立させます。

Gold層の責務

  • 集約テーブル： 日次・月次の売上集計、顧客ライフタイムバリュー（LTV）など、計算コストが高い指標を事前計算。
  • 非正規化（Denormalization）： Silver の正規化された構造を、分析用に非正規化。JOIN の手数を削減。
  • スター・スキーマ： 事実テーブル（Fact）と次元テーブル（Dimension）を設計。古典的な BI スタイル。
  • ビジネスロジック適用： マージン計算、セグメント分類、異常検知フラグ など、ドメイン知識の具現化。
  • キャッシング・マテリアライズド・ビュー： 頻繁にアクセスされるビューを物理テーブルとして保存。

Gold層のパターン

パターン 例 特徴
集約テーブル daily_sales, monthly_revenue 時間軸で粒度を落とし、計算量削減
スター・スキーマ fact_orders + dim_customers, dim_products BI ツール統合に最適、JOIN最小化
キューブ・テーブル sales_cube（年×月×製品×地域） 多次元分析、OLAP ドリルダウン対応
特徴テーブル customer_features（ML学習用） ML パイプライン向け、時間スライス対応

SQLコード例：売上スター・スキーマ

-- Gold層：売上スター・スキーマ
-- fact_orders：事実テーブル
CREATE OR REPLACE TABLE gold.fact_orders AS
SELECT
    o.order_id,
    o.customer_id,
    o.product_id,
    o.store_id,
    CAST(o.order_date AS DATE) as order_date,
    EXTRACT(YEAR FROM o.order_date) as order_year,
    EXTRACT(MONTH FROM o.order_date) as order_month,
    EXTRACT(QUARTER FROM o.order_date) as order_quarter,

    -- 数値指標（ファクト）
    o.quantity,
    o.unit_price,
    ROUND(o.quantity * o.unit_price, 2) as gross_amount,

    -- ビジネスロジック：マージン率計算
    ROUND(
        (o.quantity * o.unit_price - o.quantity * p.cost_price) /
        NULLIF(o.quantity * o.unit_price, 0),
        4
    ) as margin_rate,

    -- 割引・キャンペーンフラグ
    CASE
        WHEN o.discount_code IS NOT NULL THEN TRUE
        ELSE FALSE
    END as has_discount,

    -- パフォーマンス最適化：粒度判定（集計用の粒度フラグ）
    CASE
        WHEN DATE_DIFF(CURRENT_DATE, DATE(o.order_date)) <= 30 THEN 'RECENT'
        WHEN DATE_DIFF(CURRENT_DATE, DATE(o.order_date)) <= 365 THEN 'HISTORICAL'
        ELSE 'ARCHIVE'
    END as temporal_bucket

FROM silver.orders o
LEFT JOIN silver.products p ON o.product_id = p.product_id
WHERE o.order_id IS NOT NULL;

-- dim_customers：顧客次元テーブル
CREATE OR REPLACE TABLE gold.dim_customers AS
SELECT
    c.customer_id,
    c.first_name,
    c.last_name,
    c.email_validated,
    c.status_normalized,

    -- セグメント分類
    CASE
        WHEN ltv.total_spent > 10000 THEN 'VIP'
        WHEN ltv.total_spent > 1000 THEN 'PREMIUM'
        ELSE 'STANDARD'
    END as customer_segment,

    -- 購買パターン
    COALESCE(ltv.purchase_count, 0) as lifetime_purchases,
    COALESCE(ltv.total_spent, 0) as lifetime_value,
    COALESCE(ltv.avg_order_value, 0) as avg_order_value,

    c.created_at,
    DATEDIFF(CURRENT_DATE, DATE(c.created_at)) as days_since_signup

FROM silver.customers c
LEFT JOIN (
    -- 顧客LTV集計
    SELECT
        customer_id,
        COUNT(DISTINCT order_id) as purchase_count,
        SUM(quantity * unit_price) as total_spent,
        AVG(quantity * unit_price) as avg_order_value
    FROM silver.orders
    WHERE order_date < CURRENT_DATE
    GROUP BY customer_id
) ltv ON c.customer_id = ltv.customer_id;

-- dim_products：製品次元テーブル
CREATE OR REPLACE TABLE gold.dim_products AS
SELECT
    product_id,
    product_name,
    category,
    UPPER(status) as status,
    cost_price,
    ROUND(cost_price * 1.3, 2) as standard_margin_price,

    -- 製品カテゴリの上位分類
    CASE
        WHEN category IN ('Electronics', 'Computers') THEN 'Tech'
        WHEN category IN ('Clothing', 'Accessories') THEN 'Fashion'
        ELSE 'Other'
    END as super_category
FROM silver.products
WHERE status NOT IN ('DISCONTINUED', 'INACTIVE');

Gold層：日次集約テーブル

-- 日次売上集約（高速ダッシュボード用）
CREATE OR REPLACE TABLE gold.daily_sales_summary AS
SELECT
    order_date,
    EXTRACT(YEAR FROM order_date) as year,
    EXTRACT(MONTH FROM order_date) as month,
    EXTRACT(WEEK FROM order_date) as week,
    DAYNAME(order_date) as day_of_week,

    -- ディメンション別集約
    customer_segment,
    super_category,

    -- 指標
    COUNT(DISTINCT order_id) as total_orders,
    COUNT(DISTINCT customer_id) as unique_customers,
    ROUND(SUM(gross_amount), 2) as total_revenue,
    ROUND(SUM(gross_amount * margin_rate), 2) as total_margin,
    ROUND(AVG(gross_amount), 2) as avg_order_value,
    ROUND(AVG(margin_rate), 4) as avg_margin_rate,
    SUM(quantity) as total_quantity,

    -- Year-over-year 計算用のハッシュ
    CURRENT_TIMESTAMP as aggregated_at

FROM gold.fact_orders f
LEFT JOIN gold.dim_customers c ON f.customer_id = c.customer_id
LEFT JOIN gold.dim_products p ON f.product_id = p.product_id
WHERE order_date >= DATE_SUB(CURRENT_DATE, INTERVAL 24 MONTHS)
GROUP BY
    order_date,
    EXTRACT(YEAR FROM order_date),
    EXTRACT(MONTH FROM order_date),
    EXTRACT(WEEK FROM order_date),
    DAYNAME(order_date),
    customer_segment,
    super_category;
Design Principle: Gold層のテーブルは「何度でも読み込まれる前提」で設計します。書き込み頻度は低く（通常1日1回の更新）、読み取りは高速に。SSD キャッシング、インデックス、パーティション戦略の最適化が重要です。

Delta Lakeとの統合 — ACID, Time Travel, Schema Evolution

Delta Lakeは Apache Spark 上のオープンソース・ストレージフォーマットで、Medallion Architecture を実装する際の「最適なパートナー」です。ACID トランザクション、データのバージョン管理、スキーマ進化に対応し、データウェアハウスレベルの信頼性をデータレイク上で実現します。

Delta Lakeの核となる3つの機能

  • ACIDトランザクション： 複数の書き込みが同時実行されても整合性が保証される。部分的な失敗からの復帰も安全。
  • Time Travel（タイムトラベル）： テーブルのスナップショットを過去のバージョンに遡って読み出し。監査やロールバック対応。
  • Schema Evolution： スキーマ定義を柔軟に追加・変更。ダウンストリームの破損なし。

Delta Lake の ACID トランザクション

# Delta Lake による ACID 書き込みの例
from pyspark.sql import SparkSession
from pyspark.sql.functions import *

spark = SparkSession.builder \
    .appName("delta_acid_demo") \
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension") \
    .getOrCreate()

# 同時実行の安全性を確保した書き込み
orders_df = spark.read.csv("/data/orders.csv", header=True)

# Delta フォーマットで書き込み（ACID保証）
orders_df.write \
    .format("delta") \
    .mode("overwrite") \
    .partitionBy("order_date") \
    .option("overwriteSchema", "true") \
    .save("/lakehouse/gold/orders")

# テーブル化（メタストア登録）
spark.sql("""
    CREATE TABLE IF NOT EXISTS gold.orders
    USING DELTA
    LOCATION '/lakehouse/gold/orders'
""")

# 別のプロセスが同時にアップデートしても安全
spark.sql("""
    UPDATE gold.orders
    SET status = 'COMPLETED'
    WHERE order_id IN (123, 456, 789)
        AND status = 'PENDING'
""")

print("ACID transaction completed safely")

-- パフォーマンス影響なし（メタデータのみの変更）

Delta Lake × Medallion：MERGE INTOパターン

-- Silver層の重複排除・更新処理（MERGE INTO パターン）
MERGE INTO silver.customers target
USING (
    -- Bronze から新しいデータを持ってくる
    SELECT
        customer_id,
        first_name,
        email,
        status,
        MD5(CONCAT_WS('|', first_name, email, status)) as row_hash,
        CURRENT_TIMESTAMP as updated_at
    FROM bronze.customers_new
) source
ON target.customer_id = source.customer_id
WHEN MATCHED AND target.row_hash != source.row_hash THEN
    -- 属性変更時は更新
    UPDATE SET
        first_name = source.first_name,
        email = source.email,
        status = source.status,
        row_hash = source.row_hash,
        updated_at = source.updated_at
WHEN NOT MATCHED THEN
    -- 新規顧客は挿入
    INSERT (
        customer_id, first_name, email, status, row_hash, updated_at
    ) VALUES (
        source.customer_id, source.first_name, source.email,
        source.status, source.row_hash, source.updated_at
    );
Caution: Delta Lake の MERGE INTO は便利ですが、大規模テーブル（10億行以上）では シャッフル コストが高くなります。ファイル削除・重書き戦略の最適化が必須です。

Delta Lake のメタデータ最適化

-- テーブルの小ファイル統合（OPTIMIZE）
OPTIMIZE gold.daily_sales_summary
ZORDER BY order_date, customer_segment;

-- テーブルの統計情報を更新
ANALYZE TABLE gold.fact_orders COMPUTE STATISTICS;

-- 古いバージョン（snapshot）を削除してストレージ最適化
VACUUM gold.orders RETAIN 30 DAYS;  -- 30日以上前のバージョンを削除

-- テーブル情報の詳細確認
DESCRIBE DETAIL gold.orders;
-- numFiles: ファイル数
-- sizeInBytes: 合計サイズ
-- numRecords: 行数
-- minFileModificationTime: 最古ファイルのタイムスタンプ

パターン2：ハイブリッドバッチ＆ストリーミング

マスターデータはバッチ、イベントはストリーミングで並行処理。大規模リアルタイム企業向け。

# Spark Structured Streaming での実装
from pyspark.sql import SparkSession
from pyspark.sql.functions import *

spark = SparkSession.builder \
    .appName("hybrid_medallion") \
    .config("spark.sql.streaming.checkpointLocation", "/checkpoints") \
    .getOrCreate()

# ========== Streaming: Bronze → Silver →Gold ==========
# Kafka から イベント取り込み（Bronze）
event_stream = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "kafka:9092") \
    .option("subscribe", "order_events") \
    .option("startingOffsets", "latest") \
    .load() \
    .select(from_json(col("value").cast("string"),
                     "order_id STRING, customer_id STRING, amount DOUBLE").alias("data")) \
    .select("data.*")

# Bronze へ書き込み（ストリーミング）
query_bronze = event_stream \
    .withColumn("ingestion_ts", current_timestamp()) \
    .writeStream \
    .format("delta") \
    .option("checkpointLocation", "/checkpoints/bronze_events") \
    .outputMode("append") \
    .start("/lakehouse/bronze/order_events")

# Silver への クレンジング（ストリーミング）
silver_stream = spark.readStream.format("delta").load("/lakehouse/bronze/order_events") \
    .withColumn("order_amount_validated",
                when(col("amount") > 0, col("amount")).otherwise(null())) \
    .select("order_id", "customer_id", "order_amount_validated", "ingestion_ts")

query_silver = silver_stream \
    .writeStream \
    .format("delta") \
    .option("checkpointLocation", "/checkpoints/silver_events") \
    .outputMode("append") \
    .start("/lakehouse/silver/order_events")

# Gold への集約（ウィンドウド集計）
gold_stream = spark.readStream.format("delta").load("/lakehouse/silver/order_events") \
    .withWatermark("ingestion_ts", "10 minutes") \
    .groupBy(
        window(col("ingestion_ts"), "5 minutes"),
        col("customer_id")
    ).agg(
        count("order_id").alias("order_count"),
        sum("order_amount_validated").alias("total_amount"),
        avg("order_amount_validated").alias("avg_amount")
    )

query_gold = gold_stream \
    .select(
        col("window.start").alias("window_start"),
        col("window.end").alias("window_end"),
        col("customer_id"),
        col("order_count"),
        col("total_amount"),
        col("avg_amount")
    ) \
    .writeStream \
    .format("delta") \
    .option("checkpointLocation", "/checkpoints/gold_aggregates") \
    .outputMode("update") \
    .start("/lakehouse/gold/order_aggregates")

# ========== Batch: マスターデータの並行処理 ==========
# 毎朝、顧客マスターをバッチで更新（Silver）
customers_batch = spark.read.parquet("/source/customers_master.parquet") \
    .withColumn("updated_at", current_timestamp())

customers_batch.write.mode("overwrite").format("delta") \
    .option("overwriteSchema", "true") \
    .save("/lakehouse/silver/customers")

# クエリ実行
query_bronze.awaitTermination()
query_silver.awaitTermination()
query_gold.awaitTermination()

-- Databricks Unity Catalog による Medallion 層の管理
-- 3つのカタログ = 3つの層

CREATE CATALOG IF NOT EXISTS bronze_prod;
CREATE CATALOG IF NOT EXISTS silver_prod;
CREATE CATALOG IF NOT EXISTS gold_prod;

-- Bronze スキーマ（全員読み取り専用）
CREATE SCHEMA IF NOT EXISTS bronze_prod.raw_data;
GRANT SELECT ON SCHEMA bronze_prod.raw_data TO `analysts@company.com`;

-- Silver スキーマ（データチームのみ読み書き）
CREATE SCHEMA IF NOT EXISTS silver_prod.cleaned_data;
GRANT SELECT ON SCHEMA silver_prod.cleaned_data TO `analysts@company.com`;
GRANT SELECT, MODIFY ON SCHEMA silver_prod.cleaned_data TO `data_engineers@company.com`;

-- Gold スキーマ（全員読み取り）
CREATE SCHEMA IF NOT EXISTS gold_prod.analytics;
GRANT SELECT ON SCHEMA gold_prod.analytics TO `*`;  -- 全員

-- テーブルレベルのデータマスキング（PII保護）
ALTER TABLE silver_prod.cleaned_data.customers
SET TBLPROPERTIES ('delta.columnMapping.mode' = 'name');

-- 列レベルのアクセス制御
GRANT SELECT(customer_id, created_at) ON TABLE silver_prod.cleaned_data.customers
TO `analysts@company.com`;

-- PII列はマスク
GRANT SELECT(
    customer_id,
    created_at,
    MASK(first_name) as first_name,
    MASK(email) as email
) ON TABLE silver_prod.cleaned_data.customers
TO `external_analysts@company.com`;

他のアーキテクチャとの比較 — Data Vault, Kimball, Lambda

Medallion vs Data Vault

特性 Medallion Data Vault
層数 3層（Bronze/Silver/Gold） 3層以上（RAW/VAULT/Mart）
スキーマ設計 シンプル、段階的に正規化 複雑（Hub-Link-Satellite）、厳密な正規化
学習曲線 低い、実装が早い 高い、専門知識が必須
スケーラビリティ 優秀（クラウドデータレイク向け） 優秀（大規模企業向け）
監査・追跡 あり（各層でタイムスタンプ） 強力（元々監査・コンプライアンス向け）
推奨される規模 中堅～大規模、スタートアップ向け 大規模企業、金融・保険向け

Medallion vs Kimball Dimensional Modeling

特性 Medallion Kimball
焦点 データの段階的品質向上 BI/分析を最優先
スキーマ 各層で柔軟（Star→Snowflakeも可） Star Schema（Fact + Dimensions）に限定
導入コスト 中程度（Bronze→Gold の構成） 高い（スターモデルの明確な定義が必須）
柔軟性 高い、未見のユースケースに適応可 中程度、Mart が増えると複雑化
実装ツール Databricks, Spark, Duckなど Snowflake, BigQuery, RedshiftなどDWH
Pro Tip: Medallion と Kimball は相互補完的です。Bronze→Silver でデータを清潔にしておき、Gold 層で Kimball の Star Schema を実装する「Medallion + Kimball ハイブリッド」が多くの組織で最適です。

Medallion vs Lambda/Kappa アーキテクチャ

特性 Medallion Lambda Kappa
パイプライン 段階的変換（3層） バッチ + ストリーム並行 ストリームのみ
レイテンシ 中程度（層ごとのクロンジョブ） 低い（ストリーム側） 低い（全てストリーム）
複雑性 中程度（3層の管理） 高い（2つの経路を同期） 中程度（ストリーム処理のみ）
使用例 一般的なDEプロジェクト 高精度 + リアルタイム併用（株価など） IoT、ログストリーム

推奨事項： 大多数の組織は Medallion + ハイブリッドバッチ&ストリーミング で充分です。Lambda はメンテナンス負荷が高く、Kappa は障害時の復旧が難しいため、よほどのユースケースでなければ避けるべきです。

アンチパターンと注意点

アンチパターン1：層を多すぎる/少なすぎる設計

❌ 反例： Bronze → Silver1 → Silver2 → Silver3 → Gold1 → Gold2 → Gold3 （7層！）

各層に「なぜ必要か」の明確な責務がないと、デバッグが地獄になります。

✅ 推奨： 原則 3層。どうしても必要なら中間層を加えるが、MAX 5層。各層は「読み手」「責務」「責任者」を明確に定義。

アンチパターン2：Bronze で既にクレンジングしてしまう

❌ 反例： Bronze 取り込み時点で NULL 削除、型変換、ビジネスロジック適用

そうすると、元データへ遡ることができず、デバッグが困難。

✅ 推奨： Bronze はメタデータ（タイムスタンプ、ソース）のみ追加。全ての変換は Silver で。

アンチパターン3：Delta Lake なしでこれを実装

❌ 反例： CSV / Parquet でメダリオンを実装。ACID 保証なし。

並行書き込みでデータ破損、重複排除失敗のリスク高い。

✅ 推奨： Databricks、Snowflake、BigQuery など Delta Lake / Iceberg 対応の環境を使う。ACID が基本。

アンチパターン4：Gold 層で無限に JOIN

❌ 反例： fact_orders ← customers ← addresses ← countries ← regions （5段階 JOIN）

クエリが遅い。デバッグ困難。

✅ 推奨： 必要な次元テーブルは Gold で予め非正規化。（customer 情報を orders に直接保持）

アンチパターン5：スキーマ版管理がない

❌ 反例： テーブルの構造が勝手に変わり、ダウンストリームが突然エラー

破断的な変更に気づかない。

✅ 推奨： スキーマを git で管理。dbt contracts で型チェック。Delta の DESCRIBE を定期確認。

アンチパターン6：ドキュメント/メタデータがない

❌ 反例： テーブル名「t1, t2, t3」。カラム「col_a, col_b」。誰も使い方が分からない。

分析者がレイアウト作成費やされる。

✅ 推奨： データカタログ（Databricks Data Lineage、Collibra など）を使用。テーブル/カラムに説明を付与。

アンチパターン7：小さなデータセットでオーバーエンジニア

❌ 反例： 日々 100 行しかないデータを、Medallion + Delta + Iceberg + Unity Catalog で管理

過度な複雑さ。メンテナンスコストが見合わない。

✅ 推奨： 小さいうちは SQLite や Postgres で OK。スケールしてから Medallion へ移行。段階的な成長を許容。

アンチパターン8：監視・アラートがない

❌ 反例： パイプラインが 1 週間サイレント失敗。誰も気づかない。古いデータが使われ続ける。

ビジネスインパクト甚大。

✅ 推奨： 各層の "freshness"（更新日時）、 "completeness"（欠落率）、 "accuracy"（整合性）を自動監視。Soda、Monte Carlo Data など。
09

まとめ — 10年腐らない設計原則

Medallion Architecture は単なる「層構造」ではなく、「データの段階的品質向上とビジネス価値化のメンタルモデル」です。

Medallion を選ぶべき理由

  • シンプルで理解しやすい： DWH のように複雑でなく、Data Lake のように無統制でもない。中庸が美徳。
  • スケーラビリティ： クラウドストレージの安さを活かし、大規模化に強い。
  • 段階的な品質管理： 各層で責務が明確。デバッグ・保守が容易。
  • 監査・ガバナンス： Delta Lake の Time Travel と Unity Catalog で、コンプライアンス要件に対応。
  • 柔軟な進化： データ要件の変化に対して、各層を個別に改善できる。

実装への第一歩

  1. Bronze の構築： 全てのソースを「ありのまま」取り込む パイプラインを作る。1～2 ヶ月。
  2. Silver の整備： ビジネスキー・品質ルール定義。重複排除・クレンジング。2～3 ヶ月。
  3. Gold の構築： 分析チームと共に、集約テーブル・スター・スキーマを定義。1～2 ヶ月。
  4. ガバナンス体制： Data Catalog、SLA 定義、監視・アラート。1 ヶ月。

総投資： 最初の 6～9 ヶ月で基盤完成。その後は改善・拡張。

10年腐らない設計の要諦

1. 各層に「なぜ存在するか」の責務を持たせる
Bronze = ソースのコピー。Silver = 品質向上。Gold = ビジネス最適化。
2. メタデータ・ドキュメントを第一級市民にする
テーブル・カラムの説明なしに、数ヶ月後には存在意義を忘れている。
3. 段階的な改善を前提に設計する
最初は 100% 完璧でなくても OK。反復的に高度化していく柔軟性を持たせる。
4. ACID と Time Travel を当たり前のものにする
Delta Lake / Iceberg などで、「障害復旧は簡単」という安心感が生まれる。

Medallion Architecture は、データエンジニアが「データ資産」を確実に構築・管理するための、20年先も通用する基本原則です。
