基調講演

Conversation is the New Dashboard
属人性を排除する第4世代BIの勢力図

W
ウィル (Sho Maekawa)
コミューン株式会社 — R&D データチームマネージャー
Speaker Deck →

ウィル氏がタイトルで最も注目してほしいのは「属人性を排除する」という部分。データアナリスト歴10年以上の中で、「なんでこんな簡単な作業を毎回コツコツやらなきゃいけないんだ」と悔し泣きした日が何日もあったという。2026年以降、その属人性をようやく排除できるフェーズに入ったのではないかという強い思いがタイトルに込められている。

BIツール世代の整理
世代 別名 代表ツール 本質
1st (1990s) トラディショナルBI MicroStrategy, Cognos キューブ構築・IT専任
2nd (2010s) セルフサービスBI Tableau, Power BI GUI操作・ビジネス普及
3rd (2015〜) DWHネイティブBI Looker, ThoughtSpot コード管理・セマンティック層
3.5 (2020〜) セマンティック民主化BI Steep, Lightdash dbt直結・非技術者に開放
4th (2024〜) AIネイティブBI Lightdash? Codatum? AIからダッシュボードを作成

質疑応答では、QuickSightが表に入っていない点について「BIツール大好きなので本当は1世代につき5ツールくらい入れたいが、見づらくなるので恣意的に2個くらいに絞っている」とコメント。

セルフサービスBIの限界と4つの進化シナリオ

2026年はBIツールの変革期。セルフサービスBIが10〜15年間市場を席巻してきたが、「突き詰めるとデータ人材を介さないと運用できない」という限界がある。ウィル氏は、AIネイティブBIへの移行に関して4つの可能性を提示した:

01 第3〜3.5世代のコード管理BIツールが進化していく道
02 完全内製で構築する道
03 セルフサービスBIがアーキテクチャを刷新して覇権を取る道
04 バイブコーディングBI（squadbaseなど）が新たに覇権を取る道
squadbaseへの注目

squadbaseについて、ウィル氏は4ヶ月前から検証を続けており「4ヶ月でものすごく進化した」と強調。

▸ 4ヶ月前は大きなデータや具体的な指示で「サクサクしない」部分があったが、直近では大幅に改善
▸ UIパーツが月を追うごとに増え、指示のフィードバックを学習して最適化している様子が肌で感じられる
▸ データソースをつなぐとsquadbase側から自発的に分析を提案してくれるコンセプトが次世代的
▸ コードが全面公開されていないため、体裁にこだわる企業では合わないケースもある
社内コンテキストをBIにどう組み込むか

「分析エージェントの会話履歴をデータとして残している。1ヶ月で約300会話溜まる。その会話を抽出してビジネスメトリクスを出すと、自分の知らなかったメトリクスが抽出できたりする。ユーザーが発しているビジネスロジックをAIで取得し、それを定義していくことで、セマンティックレイヤーに落とし込める可能性がある。」

相良氏も「Snowflakeがユーザーのクエリ履歴をもとにセマンティックレイヤーを作る機能を出していた。Slackの会話履歴やGoogleドライブのドキュメントからコンテキストを拾ってAIに渡す方向性もありうる」と補足。

dbt未導入の場合のBI選定

「銀の弾丸はないが、squadbaseのようなバイブコーディング型は4ヶ月で驚異的に進化しており、セマンティックレイヤーを持たなくても一定品質のグラフ生成ができるようになるかもしれない。」

コミューンのアプローチ

ThoughtSpot（DWHネイティブ第3世代BI）と内製の分析エージェントを組み合わせ、ThoughtSpotのTML（ThoughtSpot Modeling Language）を介してビジネスユーザーもダッシュボードを作れるか実験中。BIのCI/CDも実現しており、TMLへの変更時のガードレールやテストが機能している。

次世代BIの核心
2026年はAIネイティブBIへの「準備期間」。爆発的に使えるプロダクトはまだ完成していないが、先進企業から事例が出てくる年になるとウィル氏は見立てる。
02 — TALK 01
講演①

開発者/利用者の両視点で
"Lightdash愛" を語りたい

K
K (Takayuki Komura)
株式会社Leaner Technologies — 技術基盤チーム データエンジニア
Speaker Deck →

K氏の冒頭メッセージは「BIツール、多くないですか？」。Looker → Metabase → Lightdashと複数のBIツールを渡り歩いた経験から、選定基準として意外な軸を提示した。

「愛せるかどうか」というBI選定基準

「どれだけそのBIツールを愛せるかで決めている。半分冗談で半分本気。BIツールの選定から任される人は、ほとんどの場合その後も管理者として保守運用を担う。愛せなかったらモチベーションが上がらないし、管理も煩雑になってカオスに向かう。なんとなく好き、使ってて楽しいくらいの抽象的な感じでも全然いい。」

最も愛せるBIは？と質疑で問われ、「やっぱりLightdash。手触り感、触ってて直感的に使える。個人的にLightdashという名前がすごい好き」と即答。

BIツール「あるある」3選
01 銀の弾丸は存在しない — 導入したからデータの民主化が叶うわけではない。ただし「ツールを導入することで文化が生まれることもあるので、いいやで入れてしまうのもあり」
02 コモディティ化 — 導入当時はそのツール固有だった機能が、他社のアップデートや新サービスで追いつかれる。特にAIエージェント機能搭載BIが増えている
03 管理者不明による治安悪化 — 運用ルールを定めないまま使い始めてカオスに。導入担当者の退職で野良ダッシュボードが量産される
Lightdashの開発者視点での推しポイント
セマンティックレイヤー
dbtプロジェクトのYAML上に直接定義可能。別リポジトリや別サービスを立ち上げる必要がなく「行動範囲を広げずに済む」のが大きなメリット。Cloud Codeを使うと、YAMLを生成する際にセマンティックレイヤーのディメンションやメトリクスも一緒に定義してくれる。
プレビュープロジェクト
新規のダッシュボードやチャートは「直接本番ではなくプレビュー環境で作成し、コードレビューを経てマージ」というフローを厳格化。面倒だがコード管理への強い意志が感じられる運用。
BI as Code
ダッシュボードやチャートの情報をYAMLとして管理可能。ただし、ユーザー権限管理・スケジュール配信（メール/Slack）などの制御系はコード管理の対象外。Terraformプロバイダーは存在するが、欲しい機能がまだ足りず未使用。APIで情報を抜くことは可能なので「頑張れば管理できる」が、そこまでやるかという判断。
OSS版 vs Cloud版

Leaner TechnologiesではOSS版をCloud Runでホスティング。OSS版ならではの辛みとして:

▸ AI機能（LLMへの問い合わせ）がCloud版限定 — 「2025年に一番注文された商品は？」と聞くとSQLを生成し、結果とグラフを一緒に返してくれる機能がOSSでは使えない
▸ 埋め込み機能もCloud版限定 — プロダクト上へのダッシュボード埋め込みが不可
▸ ホスティング自体は「今の時代AIに聞けばやり方を教えてくれるのでハードルは高くない」が、セキュリティ面での不安は残る
AI × Lightdashの未来

Lightdash公式Xが「ラフ画（手描きのダッシュボードの絵）をもとにダッシュボードを作らせる」デモを投稿しており、K氏は「そういった世界線も今後増えていくのではないか」と期待。Cloud Codeでdbt YAMLのセマンティックレイヤーを生成する際も、Issueに追加したいディメンションやメトリクスを書いておけば作ってくれるが、ゼロからのダッシュボード作成はまだ課題あり。

K氏の結論
BIツールは「愛せるかどうか」で選ぶ。Lightdashの手触り感と直感性が決め手。OSS版の制約はあるが、dbtと同じYAML空間でセマンティックレイヤーを管理できる体験は他にない。
03 — TALK 02
講演②

dbt Semantic Layer × Steep
2年間運用してみて感じたこと

T
Taiki Hoshino
株式会社Gaudiy — アナリティクスエンジニア
Speaker Deck →
導入の背景 — 2人チームの切実な課題

Gaudiyに一人目のデータ人材として入社（もう一人は業務委託）。「AIの発展を考えた時にセマンティックレイヤーに注力すれば、課題解決と次世代のデータ基盤構築が同時にできる」と判断した。

01 SQLの手間と負担 — SQLスキルがあってもデータのドメイン知識が社内に定着せず、SQL作成依頼やレビュー依頼が星野氏に集中
02 新メンバー増加への対応 — 入社した人がダッシュボードの数値を深掘りしたいがドメイン知識がなく、SQL書ける人でも結局データチームの支援が必要
dbt Semantic Layerを選んだ理由
▸ すでにdbt Cloudを利用しており、ほぼ環境構築不要で使えた
▸ 接続可能なBIツールが他のセマンティックレイヤーより多く、将来のBIツール乗り換えを見越せた
▸ 当時社内ではOpenTelemetryやOpenFeatureなど「ベンダーロックインされないツール」が話題だった。もしOpen Semantic Layerがこの当時あったらすぐ飛びついていたとのこと
dbt Semantic Layer運用の実態（辛み編）

「measureという概念とmetricsという概念の違いが最初まったくわからなかった。調べても意外と情報が出てこなくて、公式ドキュメントをひたすら読み続けた。最初の1年は苦戦したが、パターンが決まっていくと2年目はサクサク開発できるようになった。」

⚠ JOIN周りの特殊さ — dbt Semantic LayerはPrimary Key / Foreign Keyを定義すると自動的にJOINされる仕様。Lookerなどと比べて特殊で、最初は「何にJOINされるんだろう」と混乱する
⚠ マルチホップJOINの制約 — Steepの制約でマルチホップJOINが効かず、ブリッジテーブルが使えない。結果として「愚直にスタースキーマをやり続けた」結果、SQLのテキストが非常に長くなった
⚠ 開発検証のつらさ — Steep上で開発環境のdbt Semantic Layerコードを検証することができず、「夜のみんなが退勤する頃にやって、ダメだったら戻す」という一か八かの運用
Steepの運用形態
▸ 経営層が見るトップレベルのKPIとそのドリルダウンをSteepに集約。事業上の会話に出てくる指標は基本Steepで整備
▸ アドホック分析はAI（分析エージェント）で対応。アドホック → 定点観測への「昇格フロー」は現状なし
▸ Redashも並行利用。開発チームやBI系のメンバーが使い、3ヶ月間保持される一時的なクエリ保存先として機能
BIツールへの向き合い方

「お二人がすごいBIツールへの愛が強くて、僕は発表でも "BIツールにこだわりありません" と言ってるぐらい。もうちょっとちゃんと愛を持って接してあげればよかったなと思ったので、ちゃんとBIツールに向き合いたい。」

Gaudiyのデータサイエンスの未来（余談）

ファン一人ひとりの貢献を計測・可視化するために「エージェントベースドモデリング」に注力予定。ユーザー一人一人をエージェントで模倣し、リアルのファンコミュニティではABテストが難しいため仮想空間上でシミュレーションを行うアプローチ。

2年間の結論
学習コストは高いが、パターンが固まれば開発速度は劇的に上がる。セマンティックレイヤーへの投資はAI時代の基盤となる。ただしツールへの「愛」ではなく「実用」で選んだ結果、壁にぶつかることも。
04 — LT
共催LT

primeNumber — BIの手前のデータ品質

Y
山本
株式会社primeNumber

AIでのデータ分析やBIが普及するほど、上流のデータ品質の重要性が増すという警鐘。

▸ Text to SQLの普及やバイブコーディングBIにより、データエンジニアを介さずにデータを直接利用する人が増加
▸ かつてはデータエンジニアが現場で気づいて防いでいた品質の不備が、利用者の拡大により検知されないままBIやAIに流れるリスク
⚠ AIは低品質なデータでも「それっぽい言葉でそれっぽく回答してしまう」 — Garbage In / Garbage Out問題の増幅
TROCCOの新機能
GUIベースで品質定義できるデータチェック機能（2月リリース）を紹介。SQLを書かずにユニークネスやNULLチェックが画面上から設定可能。AIネイティブBI時代ほど、データパイプラインの品質管理が重要になる。
05 — INSIGHTS
総括

イベント全体を通じたインサイト

BIツールの変革は「準備期間」

ウィル氏はクロージングで「Xのコメントやお二人の発表を見ていても、まさにAIネイティブBIへの準備期間だと感じた」と総括。爆発的に使えるAIネイティブBIはまだ完成していないが、2026年中に先進企業から事例が出てくるのではないかという見通し。

運営コメント
運営チーム（北川氏）によれば「データエンジニアリングスタディとしてBIツールをテーマにした回は10回以上やっていなかった」。今回はLightdashとSteepという比較的新しいツールに焦点を当てた企画。
3者3様のBIとの向き合い方
ウィル氏
BIツール研究が趣味。書籍・YouTube・Udemy・noteで発信。AIネイティブBIの未来を俯瞰する研究者的スタンス。
K氏
「愛せるかどうか」で選ぶ。Lightdashの手触り感と直感性に惹かれる実践者。BI選定の哲学を持つ。
星野氏
「BIツールにこだわりはない」と公言するが、セマンティックレイヤーという基盤技術に2年間真摯に向き合った運用者。
3つのBIツールの立ち位置まとめ
Lightdash
第3.5〜第4世代
dbt直結のセマンティックレイヤー管理。BI as Codeを標榜。OSS版とCloud版で機能差あり。愛着を持ちやすい直感的UX。
Steep
第3.5世代
dbt Semantic Layerとの深い連携。経営層向けKPIダッシュボードに強み。マルチホップJOINの制約など学習コストは高め。
squadbase
第4世代（新興）
バイブコーディングBI。自発的な分析提案がコンセプト。4ヶ月で大幅進化中。セマンティックレイヤー不要で使えるポテンシャル。
