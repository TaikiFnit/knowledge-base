## Part III 概要

Part IIIは、Data Meshの「**どう設計するか**」を扱う。Part I（4原則）・Part II（なぜ今？）を受けて、本パートでは原則を具体的なアーキテクチャ構造に落とし込む方法を提示する。

**第9章「The Logical Architecture」** では、4原則それぞれをアーキテクチャコンポーネントに対応させた技術中立な論理設計を解説。**第10章「The Multiplane Data Platform Architecture」** では、ドメインが自律的にデータプロダクトをビルド・デプロイ・発見・消費できるようにするプラットフォームの3プレーン構造を詳述する。

| 章 | テーマ | 主なコンセプト |
|---|---|---|
| Chapter 9 | 論理アーキテクチャ | Data Product Quantum、ポート設計、埋め込み型ガバナンス |
| Chapter 10 | マルチプレーンデータプラットフォーム | 3層プラットフォーム、開発者ジャーニー、コンシューマー体験 |

Dehghaniが推奨する設計の順序：**①データプロダクトを設計する → ②プラットフォームAPIを設計する → ③実装を決める**。

## 第9章：論理アーキテクチャ

第9章は、Data Meshの技術非依存な論理アーキテクチャのブループリントを提示する。Dehghaniはあえて特定ツール・実装を規定せず、設計の自由度を保つ。各アーキテクチャコンポーネントはData Meshの4原則のいずれかに対応している。

> **設計哲学：** Dehghaniが繰り返し強調するのは、「このアーキテクチャは特定の技術スタックを強制しない」という点。論理設計として読み、実装時にチームが創造性を発揮できる余地を意図的に残している。

## データの大分裂（The Great Divide）

現代の組織データは大きく2つの面に分裂している：

- **オペレーショナルプレーン** — トランザクション処理（ドメインアプリ群）
- **アナリティカルプレーン** — ウェアハウジング・BI・MLの世界

この2プレーンはテクノロジー・組織・トポロジーすべての面で乖離してきた。

アナリティカルプレーン自体もさらに内部で分裂している：

- **データウェアハウス** — BI/レポーティング向け
- **データレイク** — データサイエンス向け

脆弱なETLパイプラインが両者と運用システムをつなぎ続けているが、組織が成長するにつれてパイプラインは「継続的に壊れ、複雑さが際限なく増大する」構造となる。

**Data Meshの解決策：** ETLを技術的パッチで修正するのではなく、**ドメインベースの逆転トポロジー**という根本的なアーキテクチャ変更で対応する。

### ドメイン指向の分析データ共有

Data Meshの論理アーキテクチャは、データをテクノロジースタックではなく**ビジネスドメイン**を基準に配置する。

- 各ドメインは**オペレーショナルAPI**（「新しいエピソードを作成する」）と**アナリティカルデータエンドポイント**（「過去N月のエピソードデータ」）の両方を公開する
- 一つのドメインが別ドメインの分析データを消費できる
- 運用系・分析系を独立してリリース・デプロイ可能（自律性）

## Data Product Quantum — アーキテクチャの量子

Data Meshの中核アーキテクチャ概念が**Data Product Quantum（DPQ）**だ。「Quantum」という言葉は進化的アーキテクチャから借用したもので、**高い機能凝集性を持ちながら独立してデプロイ可能な最小アーキテクチャ単位**を意味する。

> *"The data product is the architectural quantum of the data mesh — the smallest unit of architecture that can be independently deployed with high functional cohesion."*
> — Zhamak Dehghani, Data Mesh (O'Reilly, 2022)

データプロダクトはメッシュ上の1つのノードであり、その機能に必要なすべての構造要素を内包する。

### 3つの構造コンポーネント

#### 1. コード（Code）

- **(a) データパイプラインのコード** — 上流データ（ドメインの運用システムまたは上流データプロダクト）を消費・変換・サービングする
- **(b) APIのコード** — データへのアクセス、セマンティック/構文スキーマ、可観測性メトリクス、その他メタデータを提供する
- **(c) トレイト強制のコード** — アクセス制御ポリシー・コンプライアンス・プロベナンスを強制する

#### 2. データ・メタデータ（Data and Metadata）

- **分析データ：** ポリグロット形式（イベント・バッチファイル・リレーショナルテーブル・グラフ — 同一セマンティクス）で格納
- **メタデータ：** 計算ドキュメント、セマンティック/構文宣言、品質メトリクス
- **固有メタデータ（Intrinsic）：** セマンティック定義など
- **ガバナンストレイト：** アクセス制御ポリシーなど

#### 3. インフラストラクチャ（Infrastructure）

- コードのビルド・デプロイ・実行を可能にする基盤
- ビッグデータとメタデータへのストレージとアクセスを提供
- 他のデータプロダクトへの影響なしに独立セグメント（論理的分離）で動作

> **過去のパラダイムとの違い：**
> - **過去：** パイプラインはデータから独立して管理。インフラ（ウェアハウス・レイク）は共有。
> - **Data Mesh：** データプロダクト = コード ＋ データ ＋ インフラ の合成物。ドメインの境界コンテキストの粒度で一体化される。

## データプロダクトのポート

データプロダクトは外部と通信するための標準化されたインターフェース「**ポート**」を持つ。4種類のポートが存在する。

### 1. 入力データポート（Input Data Port）

上流データ（他のデータプロダクトまたはドメインの運用システム）を受信・プル取得するためのAPI/メカニズムのセット。

### 2. 出力データポート（Output Data Port）

分析データとメタデータを提供するインターフェース。主な特性：

- **読み取り専用（Read-only）** — イミュータブル、時間軸変異型
- **ポリグロット対応：** SQL・Blob Storage・イベント・グラフ・セミ構造化ファイル・Pub/Sub・カラムナーファイル
- **読み取り関数は時間の関数** — from-toウィンドウとして定義される
- **マルチモーダル：** BIアナリスト用SQL / データサイエンティスト用カラムナー / アプリ開発者用Pub/Sub

### 3. コントロールポート（Control Port）

ガバナンスポリシーを設定するための新しいAPIセット。

- 集中的にプッシュするか、ローカルで作成される
- ポリシーの受信・変更・実行はすべてローカルで行われる
- 連合ガバナンスモデルの実行点

### 4. ディスカバリーポート（Discovery Port）

出力ポート情報・SLO・ドキュメント・データアクセス・認可情報を提供するルートAPI（アグリゲート）。コンシューマーがデータプロダクトを発見・理解するための入口。

### データプロダクトコンテナ

入力/出力ポート・コントロールポート・ポリシーエンジン・コード・ストレージ・APIをすべてパックした単位がデータプロダクトコンテナとなる。

### メッシュの3層相互接続

データプロダクト間の接続は3つの層から成る：

1. **データフロー層（Data Flow Layer）** — 入力ポート⇔出力ポート間のデータの流れ。これがリネージになる。
2. **スキーマセマンティックリンク（Schema Semantic Links）** — 標準スキーマ言語によりプロダクト横断でのJOINを可能にする。
3. **データリンク（Data Links）** — 実際のデータ関係（例：「このリスナーはこのユーザーと同一人物である」）。

## 埋め込み型コンピュテーショナルポリシー

第9章はガバナンスのアーキテクチャも扱う。Data Meshのガバナンスモデルは**連合型コンピュテーショナルガバナンス**と呼ばれる。

### 連合ガバナンスモデル

意思決定は**ドメインデータプロダクトオーナー ＋ データプラットフォームプロダクトオーナーの連合体**が行う。

- 自律性・ドメインローカルの意思決定と、グローバルルールのバランスをとる
- グローバル決定の目的：**相互運用性**の実現 ＋ **複合的ネットワーク効果**の創出

### サイドカーパターンによるポリシー実行

ポリシーはサービスメッシュのサイドカーと類比して、各データプロダクトに**サイドカーまたはライブラリとして埋め込まれる**。

- アクセス制御・匿名化・暗号化・差分プライバシー・合成データなどの機能
- プログラマティックに構築し、プラットフォームに自動化する
- コントロールポートを通じてポリシーを設定する

### ガバナンス対比表：従来 vs Data Mesh

| 観点 | 従来のデータガバナンス | Data Mesh ガバナンス |
|---|---|---|
| チーム構成 | 集中型チーム | 連合型チーム（ドメイン代表で構成） |
| データ品質 | データ品質に責任を持つ | 「品質とは何か」のモデリング方法を定義 |
| データセキュリティ | データセキュリティに責任を持つ | 機密レベルを定義し、プラットフォームが構築・監視 |
| 規制遵守 | 規制遵守に責任を持つ | 規制要件を定義し、プラットフォームが自動化 |
| データ管理 | 集中型データ管理人 | ドメインによる連合型管理人制度 |
| データモデリング | グローバル標準データモデル | ポリセム（クロスドメイン要素）のモデリング |
| ドメイン関係 | ドメインから独立したチーム | ドメイン代表で構成されるチーム |
| データ構造 | 静的データ構造 | 変化を受け入れるダイナミックなトポロジー |
| テクノロジー | 集中型テクノロジー（モノリシック） | ドメインごとのセルフサーブプラットフォーム技術 |
| 成功指標 | 管理データ量の最大化 | ネットワーク効果（消費接続数）の最大化 |
| プロセス | 手動プロセス、人的介入 | プラットフォームによる自動化プロセス |
| エラー対応 | エラーを防止する | エラーを検知 ＋ 自動処理で回復する |

## 第10章：マルチプレーンデータプラットフォーム

第10章はData Meshの**セルフサーブデータプラットフォームアーキテクチャ**を詳細に解説する。ドメインが自律的にデータプロダクトをビルド・デプロイ・発見・消費できるインフラが本章のテーマだ。

### なぜマルチプレーン？

「プレーン」とはネットワーキングのコントロールプレーン・データプレーンのように、**統合されているが独立した存在の層**を意味する。

- 階層的なアクセスモデルではない
- **異なるユーザープロファイル/ペルソナにそれぞれ対応する**別個の存在として設計する
- 目的：高レベルの抽象化によって複雑さとフリクションを除去する

---

## 3つのプレーン

### Plane 1: インフラプロビジョニングプレーン（ユーティリティプレーン）

最も低レベルの層。生インフラのライフサイクル管理を担う。

**提供機能：**
- 分散ファイルストレージ
- ストレージアカウント
- アクセス制御管理
- データプロダクトの内部コードのオーケストレーション
- データプロダクトのグラフ上の分散クエリエンジン

**特性：**
- インフラアズコード（IaC）のAPIを提供
- スペック/API/コマンドでプロビジョニング
- 他のプレーンや高度な開発者が直接使用する

**例：** Spark Jobs、Kafka Topics、サービス/ストレージアカウント

### Plane 2: データプロダクト開発者体験プレーン

典型的なデータプロダクト開発者のための**メインインターフェース**。

**特性：**
- データプロダクトワークフローの複雑さを抽象化
- プロビジョニングプレーンより高い抽象レベル
- ライフサイクル管理のためのシンプルな**宣言的インターフェース**を提供
- **クロスカッティング関心事**（標準・グローバル規約）を自動実装
- データプロダクトの全ライフサイクルを管理

**ワークフロー：**
1. 開発者が**スペックファイル**（入力/出力・ポリシー・モデルを宣言的に記述）を提供
2. リポジトリにチェックイン
3. API/コマンドに渡す
4. プラットフォームがテクノロジーを自動的に連結する

**アプローチ：** 原始的なスクリプト/テンプレートから始め、洗練させていく反復的進化。

### Plane 3: メッシュ監督プレーン（メッシュ体験プレーン）

メッシュレベルで提供されるグローバルな能力。接続されたデータプロダクトのグラフを扱う。

**提供機能：**
- **検索・ブラウズ：** ユースケースに合ったデータプロダクトを発見する
- **セマンティッククエリ：** メッシュを横断した複数データプロダクトの相関分析
- **ナレッジグラフ：** セマンティックリンク付きのスキーマシステム。ブラウズと検索が可能

**特性：**
- 集中化を避けながら、ローカル定義のスキーマからグローバル標準モデルとして出現する
- 主にデータコンシューマー向けのインターフェース

## 開発者ジャーニーフェーズ

第10章はユーザージャーニーを設計ドライバーとすることを強調する。データプロダクト開発者体験プレーンは全ライフサイクルフェーズを支援するよう設計される。

| フェーズ | 名称 | 内容 |
|---|---|---|
| 01 | **Incept** | どんなデータプロダクトが必要かを考える |
| 02 | **Explore** | 既存データプロダクトを発見・理解する |
| 03 | **Bootstrap** | プラットフォームがインフラを自動プロビジョニング |
| 04 | **Source / Connect** | 上流データソース・入力ポートへ接続 |
| 05 | **Build** | 変換ロジック・データモデルを開発 |
| 06 | **Test** | データ品質・スキーマ・SLOを検証 |
| 07 | **Deploy** | 独立したユニットとして宣言的にデプロイ |
| 08 | **Run** | パイプライン実行・データサービング |
| 09 | **Maintain** | 監視・インシデント対応・更新 |
| 10 | **Evolve** | バージョニング・出力ポート追加・スキーマ変更 |
| 11 | **Retire** | 不要になったデータプロダクトを削除 |

### ThoughtWorksの実装例（OAM）

- **スペックファイル：** `data_product_spec.yml` + `data_product_deploy.yml` としてリポジトリにスキャフォールド
- **Bootstrap：** ドメインごとにSnowflakeウェアハウスをプロビジョニング、データプロダクトごとにスキーマ作成、モニタリング名前空間・カタログコミュニティ・RBACロール（Consumer/Service/Owner）を設定
- **Deploy：** CI/CDパイプラインがスペックを検証し、パイプラインをデプロイ、SLOモニターを作成し、カタログに公開、ポリシーを適用する
- **Retire：** 利用状況メトリクスを追跡し、非アクティブなプロダクトを削除する

### コンシューマー vs 開発者の視点

2つの主要ペルソナが存在する：

**データコンシューマー（Consumer Persona）**
- データを発見・利用する人
- メッシュ体験プレーンが主たるインターフェース
- ジャーニー：発見 → 評価 → アクセス申請 → 消費 → フィードバック提供

**データプロダクト開発者（Developer Persona）**
- データプロダクトをビルド・維持する人
- 開発者体験プレーンが主たるインターフェース
- ジャーニー：Incept → Explore → Bootstrap → ... → Retire（全11フェーズ）

> ブッククラブ参加者の議論では「**コンシューマーから設計を始めるべき**」という意見が強調された。コンシューマーが何を必要とするかが、プラットフォームの要件を逆算的に決定する。

## データプロダクトのアフォーダンス

第9章「Data Product Quantum Blueprint」より、単一のデータプロダクトが必ず提供しなければならない能力（アフォーダンス）を整理する。

| アフォーダンス | 説明 |
|---|---|
| **データサービング** | 読み取り専用・時間軸変異型・イミュータブル。時間ウィンドウ関数として定義。 |
| **セキュリティ** | ローカルに強制され、分散管理される。サイドカーパターンによるポリシー埋め込み。 |
| **ライフサイクル管理** | 他のデータプロダクトを壊さずに独立してデプロイ可能。モジュール性の保証。 |
| **ポリグロットアクセス** | マルチモーダル対応。アナリスト用SQL・データサイエンティスト用カラムナー・アプリ用イベント。 |
| **発見可能性** | 独立して発見可能で、良質なドキュメントを持つ。ディスカバリーポートを通じて実現。 |
| **相互運用性** | 標準インターフェースを介して他のデータプロダクトと連携できる。 |
| **信頼性** | SLO（データ生産間隔・キャプチャから処理までの誤差・統計的形状）で定義・保証。 |

## まとめ — Part IIIの要点

### 第9章の要点

**A. Data Product Quantum（DPQ）**
データプロダクトは「コード + データ/メタデータ + インフラ」の3要素をドメイン境界で一体化したアーキテクチャ量子。過去のパラダイムとの最大の違いはこの一体化にある。

**B. 4種類のポート設計**
Input（受信）・Output（サービング）・Control（ガバナンス設定）・Discovery（発見）の4ポートが、データプロダクトの外部インターフェース全体を定義する。

**C. サイドカーガバナンス**
ポリシーはサイドカーとしてデータプロダクトに埋め込まれ、ローカルに実行される。「防止」ではなく「自動検知・回復」へのパラダイムシフト。

### 第10章の要点

**D. 3プレーン構造**
インフラプロビジョニング（Plane 1）・開発者体験（Plane 2）・メッシュ監督（Plane 3）の3プレーンが、それぞれのペルソナに適した抽象レベルを提供する。

**E. 宣言的スペックファイル**
開発者はYAMLスペックファイルを書くだけ。プラットフォームが残りを自動化する。「原始的なスクリプトから始め、洗練させていく」反復的アプローチ。

**F. 設計の優先順位**
①データプロダクトを設計する → ②プラットフォームAPIを設計する → ③実装を決める。この順序がDehghaniの推奨する設計アプローチ。

### Part III 全体の核心メッセージ

Data Meshのアーキテクチャは特定ツールを強制しない。**論理設計として原則を理解し、チームの文脈で創造的に実装する**ことが求められる。DPQというアーキテクチャ量子の概念と、3プレーンのプラットフォーム設計が、4原則を現実のシステムに翻訳するための中心的な道具立てとなっている。
