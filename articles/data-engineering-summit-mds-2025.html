01 — DEFINITION & SCOPE
MDS定義と調査スコープ

Modern Data Stackとは
何を調査したのか

定義
Modern Data Stack（MDS）とは「最新のクラウドネイティブな技術やサービスを組み合わせたデータ基盤、あるいはその考え方や該当するサービス群」のこと。単一製品ではなく、目的別に最適なSaaSを組み合わせるというアーキテクチャ思想そのものを指す。

さがら氏の調査スタイルは独特だ。2週間ごとにブログ・リリースノート・公式発表を横断的にチェックし、9つのカテゴリにまたがる50以上の製品の動向を継続追跡している。今回の講演はその蓄積を2025年版として総括したものだ。

02 — M&A
企業買収・合併

2025年、買収・合併の年表

2025年はMDS業界にとって「買収の嵐」が吹き荒れた年だった。さがら氏が整理した主要4件の買収は、いずれも業界構造を変えうるインパクトを持つ。特にFivetranの2連続買収は、単なるデータ取り込みツールからデータパイプライン全体の垂直統合プラットフォームへの転換を示している。

2025/1
dbt Labs → SDF
dbt Labs が SDF を買収 → dbt Fusion エンジンとしてリリース
Rust製の高速データ変換エンジン「SDF」をdbt Labsが買収。かつてはdbtの競合製品として名乗りを上げていたSDFが、dbt Labsの傘下に入り「dbt Fusion」としてベータリリース後パブリックプレビューへ移行した。カラムレベルリネージや高速実行が特徴で、dbtのコアエンジン刷新の核となっている。
2025/5
Fivetran → Census
Fivetran が Census（Reverse ETL）を買収
Reverse ETL大手のCensusをFivetranが買収。データウェアハウスからSFAやCRMへデータを同期する逆方向の流れを取り込み、ELだけでなくデータ活用・オペレーショナルアナリティクスまでをカバーする体制を整えた。CensusはHightouchと並ぶReverse ETL市場のリーダーだっただけに、業界へのインパクトは大きい。
2025/5
Salesforce → Informatica
Salesforce が Informatica を買収（包括的データ統合管理）
エンタープライズデータ統合の老舗Informaticaを、SalesforceがCRM覇者の立場から買収。Data Cloudの強化と、企業内分散データの一元管理を実現するための布石とみられる。MDSの「外部」からメガベンダーが市場に参入してきた象徴的な出来事でもある。
2025/9
Fivetran → Tobiko Data (SQL Mesh)
Fivetran が Tobiko Data（SQL Mesh開発元）を買収
dbtの有力な対抗製品「SQL Mesh」を開発するTobiko Dataを、わずか数ヶ月前にCensusを買収したFivetranがさらに続けて買収。EL（Extract/Load）→ データ変換（Transform）→ Reverse ETLというデータパイプライン全体を一手に担う垂直統合が完成した。さがら氏はこの2連続買収を「業界の常識を変える動き」と評した。
背景コンテキスト
2025年10月のdbt Coalesce（dbtの年次カンファレンス）では新ビジョン「Open data infrastructure」が発表された。MDSエコシステム全体がより統合・集約される方向に動いており、「Modern Data Stack」というワード自体もトレンドの節目を迎えているとさがら氏は述べる。

「SnowflakeとDatabricksは強い。LLMの台頭に伴う各機能の発表が多かった。各企業の買収や統合もあり、Modern Data Stackというワードもトレンドの節目を迎えている印象（ちょっと寂しい）」 — さがら氏、クロージングコメントより

03 — DATA EXTRACT / LOAD
Data Extract / Load

データ取り込みの最新動向

ELツールの2025年トレンドは大きく3つに集約される。セキュアなネットワーク接続への対応、生成AIとの組み合わせによる開発効率化、そしてSnowflake/Databricksが自社製EL機能を強化することによる市場変化だ。

セキュアネットワーク接続 — Dockerコンテナエージェントが主流に

オンプレミスや閉域ネットワーク内のデータソースへのアクセスに対応するため、Dockerコンテナエージェントを使った構成が業界標準となりつつある。Fivetranは「Hybrid Deployment」、AirbyteはEnterpriseで「Enterprise Flex」として提供。ファイアウォールの内側にあるデータベースへ、クラウド側のサービスが安全にアクセスできる仕組みを整備している。

Fivetran — Hybrid Deployment
Dockerコンテナエージェントを顧客環境にデプロイし、Fivetranクラウドと連携。閉域NW内データソースへのアクセスを実現。セキュリティ要件の厳しい金融・製造業での採用が進む。
Airbyte — Enterprise Flex
エンタープライズ向け柔軟なデプロイオプション。顧客環境内にエージェントを配置することで、マネージドクラウドサービスの利便性を保ちつつ、オンプレミスデータへのアクセスを可能にする。
GenAI × データロード — Claude Codeによるコネクタ開発

生成AIツールとELパイプラインの組み合わせが実用段階に入りつつある。特に注目されるのがClaude Codeを使った高速開発の事例だ。

▸ Fivetran × Claude Code — 従来は数日〜数週間かかっていたカスタムコネクタ開発が、1時間未満で完了する事例が登場。コネクタのスキャフォールディングからテストまでを生成AI主導で行う。
▸ dlt × Claude Code — Python製の軽量ELフレームワーク「dlt」でClaude Codeを使ったパイプライン構築デモが登場。DuckDBへの格納からVisivoダッシュボード表示まで、コードをほぼ書かずに完成させた事例が示された。
Snowflake / Databricks の独自EL機能強化
Snowflake — Openflow
非構造化データ・構造化データの双方に対応した新しいデータ取り込みパイプライン。Snowflakeプラットフォームへの集約をさらに強化。外部ETLツール不要でパイプラインを完結させるための機能群を整備。
GA / PREVIEW
SAP連携 — ゼロコピー双方向
SAP Business Data Cloudとの連携により、ゼロコピーの双方向データ共有が実現。SAP側のデータをコピーせずSnowflakeから直接クエリ可能。エンタープライズ向けの重要なユースケース。
エンタープライズ向け
04 — DATA WAREHOUSE / LAKEHOUSE
Data Warehouse / Data Lakehouse

最大激戦区
Snowflake vs Databricks と周辺勢力

データウェアハウス・レイクハウス領域は今回の講演で最もスライドが割かれた最大セクションだ。Snowflake・Databricks・BigQuery・Redshiftの主要4プレーヤーに加え、DuckDB/MotherDuck・Onehouseなどの新勢力、そしてIcebergを軸とするオープンテーブルフォーマットの動向まで、多岐にわたる更新があった。

AI導線強化 — 自然言語でデータを問い合わせる

2025年最大のトレンドは3大プレーヤーが揃ってLLMを活用したAI導線を本格強化したことだ。単なるデータウェアハウスを超え、ビジネスユーザーが自然言語でデータに触れる「AIデータプラットフォーム」へと進化しつつある。

Snowflake Intelligence
Cortex Search + Cortex Analystを組み合わせた自然言語エージェント機能。データに関する質問を日本語・英語で問い合わせると、SQL生成〜結果返却まで自動で行う。セマンティックモデルとの連携でビジネス定義を守りながら回答する。
Cortex Agentベース
Databricks One / AI/BI
単一UIから自然言語でダッシュボード表示まで一気通貫。「先月の売上を製品別に見せて」といった自然言語入力からAI/BIダッシュボードを自動生成。Unity Catalogのメタデータを活用してコンテキストを理解する。
Unity Catalog連携
BigQuery — Gemini Enterprise
Data Insights agentがプレビュー提供。Geminiを活用しBigQueryのデータを分析・要約・インサイト提示するエージェント機能。Looker Studioとの統合でビジュアライゼーションも含めた自動分析フローを構築できる。
プレビュー
Redshift — MDDL 一般提供
Machine-learned Dynamic Distribution Layout（MDDL）が一般提供開始。クエリパターンを機械学習で分析し、ソートキーやディストリビューションキーを自動的に動的最適化。DBAによる手動チューニングの必要性が大幅に低下。
GA
開発体験の強化 — GitベースIDEとパイプラインエディタ
Snowflake — Git連携 Workspace IDE
SnowsightにGit連携IDEが搭載。ブランチ管理・コミット・プルリクエストをSnowflake UI内で完結できる。さらにdbt Projects on Snowflakeとして、dbtプロジェクトをSnowflake環境内に取り込んでGUI管理する機能も提供開始。
Databricks — Lakeflow Pipelines Editor
データパイプラインをGUIで設計・管理できるLakeflow Pipelines Editorが登場。Delta Live TablesベースのETLパイプラインを視覚的に構築・デバッグ。DLTの複雑さをUIで隠蔽し、データエンジニア以外でも扱いやすくした。
OLTPの参入 — SnowflakeとDatabricksがトランザクションDB領域へ

2025年の大きなサプライズの一つが、分析系プラットフォームのSnowflakeとDatabricksがOLTPデータベース領域に踏み込んできたことだ。これはデータウェアハウスとオペレーショナルデータベースの境界が消えつつある、業界の根本的な変化を示している。

Snowflake Postgres（未リリース）
100% PostgreSQL互換のSnowflakeネイティブOLTPデータベース。現時点では未リリースだが、将来的にSnowflakeプラットフォーム内でトランザクショナルワークロードも完結させる狙いがある。アプリケーションのデータをSnowflakeに直接書き込める未来が見えてくる。
未リリース / 発表済み
Databricks Lakebase（プレビュー）
Neon技術を採用した低レイテンシOLTPデータベース。10ms未満のレイテンシ、10k qps超の高同時実行性能を実現。ブランチ機能が特徴で、本番データのコピーを瞬時に作成してテスト・開発環境に利用できる。パブリックプレビュー中。
パブリックプレビュー
非構造化データの対応 — FILE型とAI_EXTRACT関数

PDFや画像などの非構造化データをデータウェアハウスで直接扱う動きが本格化した。SnowflakeではFILE型が一般提供（GA）となり、AI_EXTRACT関数を使ってPDFから情報を抽出・構造化できるようになった。従来は外部ETLやPythonスクリプトが必要だった処理がSQL一本で完結する。

AI_EXTRACT の用途例
請求書PDFから金額・日付・発行者を自動抽出してテーブルに挿入、契約書から条項を構造化データに変換、など。RAGパイプラインの前処理をSnowflake内で完結させる実装パターンが登場している。
Iceberg — オープンテーブルフォーマットの標準化

Apache Icebergは2025年、ほぼすべての主要プレーヤーが本格対応を表明し、オープンテーブルフォーマットの事実上の標準となりつつある。

Snowflake — 外部Iceberg REST カタログ
外部IcebergカタログへのRESTカタログ書き込みサポートが追加。SnowflakeのデータをIceberg形式でユーザー管理のS3等に保存し、他のクエリエンジンからもアクセスできる。Polaris CatalogのOSS化も合わせて、ベンダーロックインを軽減する動きを強化。
Polaris Catalog OSS化
Databricks — Unity Catalog Managed Iceberg
Unity Catalog上で管理されるIceberg tablesがパブリックプレビュー。Delta形式との併存を可能にしながら、Iceberg形式でデータを保存・外部エンジンから参照する構成を実現。Apache Polaris（Iceberg REST Catalog実装）への対応も進む。
パブリックプレビュー
DuckDB / MotherDuck — DuckLakeの登場

軽量・高速のOLAP分析エンジンDuckDBとそのマネージドサービスMotherDuckから、DuckLakeという新しいレイクハウスフォーマットが登場した。既存のIceberg/Delta/Hudiとは異なるアプローチで、データベースファイルにメタデータ管理を内包する設計が特徴だ。

DuckLake の特徴
DBメタデータ管理レイクハウスフォーマットとして、IcebergやDeltaが外部メタデータストアを必要とするのに対し、DuckLakeはDuckDB/MotherDuck上でメタデータを管理することでシンプルな構成を実現する。まだ新しい技術だが、DuckDB陣営の影響力を考えると無視できない動きだ。
Onehouse — OneFlow と OneSync

ApacheHudi（Hudi）をベースにしたレイクハウス管理プラットフォームのOnehouseは、OneFlow（ユーザー管理ストレージへの直接保存機能）とOneSync（複数レイクハウスフォーマット間の同期機能）を提供。マルチフォーマット環境での相互運用性を高めている。

BigQuery — CMETA 自動列メタデータインデックス

BigQueryはCMETA（Column Metadata Auto-indexing）として、列のメタデータを自動的にインデックス化する機能を追加。クエリの最適化をさらに自動化し、DBAによるインデックス設計の負担を軽減。データウェアハウスの「自動運転化」が着実に進んでいる。

05 — DATA TRANSFORM / SEMANTIC LAYER
Data Transform / Semantic Layer

dbt FusionとSemantic Layer標準化

データ変換とセマンティックレイヤーの領域では、dbtの大型アップデートとSnowflake/Salesforce/dbt Labs連合によるSemantic Layer標準化の動きが注目を集めた。

dbt Fusion — Rustベースの新エンジンがパブリックプレビューへ

2025年1月にdbt Labsが買収したSDF由来のRustベースエンジンが、dbt Fusionとしてベータ→パブリックプレビューへと進化した。従来のdbtコア（Python実装）と比べた主な強みは以下の通りだ：

① 実行速度の大幅向上 — Rustによるコンパイル・実行で、大規模プロジェクトのビルド時間を大幅短縮。数百モデルのコンパイルが秒単位で完了する。
② カラムレベルリネージ — テーブル単位だけでなく列単位でデータの流れを追跡できるようになった。データ品質・ガバナンス・デバッグに大きく貢献する機能。
③ 型システムの強化 — SDF由来の型チェック機能を統合。SQLを実行前に静的解析で型エラーを検出でき、CI/CDパイプラインの信頼性が向上する。
LLM × dbt の盛り上がり

東京で開催されたTokyo dbt Meetupでは2025年に連続してLLM×dbtのセッションが組まれ、生成AIをdbtワークフローに組み込む取り組みが急上昇している。モデル定義の自動生成、ドキュメント自動生成、テスト生成など、dbtのYAML定義を生成AIで補完するユースケースが実用段階に入っている。

OSI — Semantic Layerの業界標準化へ

Open Semantic Interchange（OSI）という業界横断の標準化イニシアティブが発表された。主導するのはSnowflake・Salesforce・dbt Labsの3社。セマンティックレイヤーのメタデータ定義（指標・ディメンション・エンティティ等）を標準フォーマットで記述し、異なるBI・ETLツール間で共有可能にすることを目指す。

OSI の意義
現状は各BI/セマンティックレイヤーツールが独自フォーマットでメタデータを保持しているため、ツール移行時の定義の再実装が必要だった。OSIが普及すれば定義の可搬性が高まり、ベンダーロックインが軽減される。SnowflakeのSemantic ViewsやDatabricksのUnity Catalog Metric Viewなど、各プラットフォームの標準実装との整合性が今後の焦点となる。
06 — BUSINESS INTELLIGENCE
Business Intelligence

BI市場の地殻変動 — Omni・Hexの台頭

BI市場では2025年、既存の大手（Looker・Tableau・Power BI）の機能強化に加え、OmniとHexというスタートアップ勢の大型資金調達が業界の注目を集めた。セルフサービス型とセマンティックレイヤー型の融合という新しいアーキテクチャが現実のものとなりつつある。

なぜOmniがここまで成長しているのか

Omniの急成長の背景には、従来のBI製品が抱えるジレンマを解決するアーキテクチャ設計がある。さがら氏はQ&Aでこの点を詳しく解説している。

従来型セルフサービスBI の課題
即時分析はできるが、各ユーザーが独自に計算フィールドを定義するため、同じ指標が人によって違う値になるガバナンス問題が発生。スピードとガバナンスがトレードオフになる。
従来型Semantic Layer BI の課題
定義の一元管理はできるが、指標を事前にモデリングする必要があるため、アドホック分析のスピードが落ちる。データエンジニアのボトルネックになりやすい。
Omni の解決策
「GUIベースでデータに繋いで計算フィールドや結合定義を素早く行い、裏側でセマンティックレイヤーのコードに自動変換される」アプローチ。ユーザーはExcelライクな操作で分析しながら、そのロジックが自動的にバージョン管理可能なコードとして保存される。セルフサービスのスピードとコードとしてのガバナンスを両立した点がユーザーから高評価を受けている。
Looker の新機能 — Conversational Analytics

Salesforce傘下のLookerはConversational Analytics（自然言語でデータを問い合わせる機能）を追加し、LookMLを活用したperiod_over_period（前期比分析）の構文を正式サポート。従来は複雑なSQLやカスタムLookML定義が必要だった時系列比較を標準化した。

Tableau Next — Salesforce内BIの変革

Salesforceによる再統合が進むTableauは「Tableau Next」として新世代のアーキテクチャへの移行を進めている。Salesforce Data CloudとのネイティブSemantic Layer統合や、Agentforce（Salesforce AI）との連携強化が焦点だ。旧来のデスクトップアプリからクラウドファーストへの移行期にある。

07 — CATALOG & ORCHESTRATION
Data Catalog / Data Orchestration

カタログのMCP化とKestraの台頭

データカタログとオーケストレーションの両領域で2025年、それぞれ象徴的な動きがあった。カタログはMCP（Model Context Protocol）対応で生成AIとの統合が進み、オーケストレーションではKestraがGitHubスター数でDagster・Prefectを逆転するという注目のトピックが生まれた。

全カタログ製品がMCP Serverをリリース

AnthropicのModel Context Protocol（MCP）は、LLMエージェントが外部ツールやデータソースに接続するための標準プロトコルだ。2025年、データカタログ製品が一斉にMCP Serverをリリースした。

MCP Atlan — エンタープライズデータカタログとして先行してMCP Serverを提供。Claude等のLLMから直接データアセットのメタデータを参照できる。
MCP Select Star — 自動ドキュメント生成に強みを持つカタログ。MCP Serverによりチャットベースのデータ探索が可能に。
MCP Secoda — AIネイティブを謳うカタログ。MCP対応でAIエージェントとのシームレスな統合を実現。
MCP OpenMetadata — OSSカタログ。MCP Serverをコミュニティで開発・提供。セルフホスト環境でも生成AI統合が可能に。
MCP 統合の意義
カタログのMCP化により、Claude等のLLMエージェントがデータカタログの情報（テーブル定義・カラム説明・オーナー情報・データリネージ）を直接参照しながら分析コードを生成できるようになる。「AIに適切な文脈を与える」インフラとしてカタログの重要性が再評価されている。
Kestra — GitHubスターでDagster・Prefectを超えた

データオーケストレーション市場で2025年最大のサプライズの一つが、KestraのGitHubスター数でDagster・Prefectを超えるという逆転劇だ。

Kestra v1.0 — 2025年9月リリース
YAML宣言型でワークフローを定義するアプローチが特徴。PythonコードをそのままDAGとして管理するAirflow/Dagster/Prefectと異なり、タスク定義をYAMLで記述しシステム間連携をより直感的に扱える設計が支持を集めている。v1.0到達とともにGitHubスターが急増し、オーケストレーション市場の勢力図に変化をもたらした。
K YAML宣言型 — コードではなく設定ファイルとしてワークフローを管理。バージョン管理・レビューが容易。
K 豊富なプラグイン — 500以上のプラグインで多様なデータソース・サービスと統合可能。
K UI充実 — フロー管理・実行履歴・依存関係の可視化が洗練されたUIで提供される。
K エンタープライズ対応 — v1.0でSLA・RBAC・マルチテナントが強化され、本番利用に必要な要件が揃った。
08 — FUTURE OUTLOOK
2026年以降の動向予想

さがら氏の2026年予測

2025年の動向を踏まえ、さがら氏は2026年以降のトレンドをいくつかのポイントで予測した。買収の継続、2強体制の固定化、そしてAI-Ready基盤の意識が業界を動かすキーワードとなる。

Data Catalog / Observability 分野で買収が継続

2025年はデータカタログ・データオブザーバビリティ領域での再編が目立った。さがら氏は2026年もこの傾向が継続すると予測する。プレーヤーが多く市場が細分化されているため、大手プラットフォームへの吸収・統合が進みやすい構造を持つためだ。

予測① Data Catalog領域 — Atlan・Select Star・Secoda等の独立系カタログへの買収アプローチが続く。主要DWHプラットフォームやハイパースケーラーが候補買い手。
予測② Observability領域 — Monte Carlo・Soda等、複数プレーヤーが並存する状況から統合が進む。DWHプラットフォームへの組み込みも考えられる。
Snowflake / Databricks 2強の構図が継続

2025年を通じて最も強化されたのがSnowflakeとDatabricksだ。機能面・資金面・エコシステムのいずれにおいても他を圧倒しており、さがら氏は「2強の構図は2026年以降も継続する」と予測する。Q&Aでも「どちらを選んでも外れない」という発言がある通り、この2製品を軸にテック選定を考えることが合理的な状況となっている。

AI-Ready 基盤の意識が高まる

2026年に向けてさがら氏が最も強調したのは「AI-Ready基盤」という意識の重要性だ。具体的には以下の5領域への投資が求められると語った：

AI データモデリング — 整合性のとれたモデルが生成AIに正確な文脈を提供する基盤。ディメンショナルモデリングの基礎が再評価されている。
AI Semantic Layer — 指標・ディメンション定義の一元管理がAIへの文脈提供として直接機能する。OSIによる標準化がこの重要性をさらに高める。
AI データ品質 — Garbage In, Garbage Out。AIへの入力データの品質が直接出力の品質を決める。オブザーバビリティ・テスト・モニタリングへの投資が重要。
AI 非構造化データ対応 — PDFや画像など非構造化データのデータパイプライン組み込みが急務。FILE型・AI_EXTRACTなどのネイティブ機能が整備されつつある。
AI ドメイン知識のドキュメント化 — ビジネスロジック・用語定義・業務文脈をドキュメント化することが、AIへの適切な文脈提供の前提となる。
技術は手段 — 事業貢献意識が最重要
さがら氏 — クロージングメッセージ
「技術は手段です。データ基盤の普及や事業への貢献を意識することが最も重要。SnowflakeとDatabricksが強力なプラットフォームになっていることは確かで、それ自体は喜ばしいことです。ただ、Modern Data Stackというワードはトレンドの節目を迎えているかもしれない——その分だけ、基盤に乗って何を実現するかという問いが重要になってきます」
09 — Q&A SESSION
Q&Aセッション

参加者からの質問
さがら氏の回答

講演後のQ&Aセッションでは、参加者からIceberg・ベンダーロックイン・Semantic LayerおよびOmniに関する実践的な質問が寄せられた。以下はClassmethod DevelopersIOのレポートに基づく全4問のQ&Aだ。

Q1
SnowflakeとDatabricksにベンダーロックインしないためにはどういうことができますか？
A
Icebergなどのオープンテーブルフォーマットを取り入れることが最も効果的な手段です。データ自体をユーザー管理のS3に保持する構成にすれば、特定プラットフォームへの依存度が下がり、別のクエリエンジンからも柔軟にクエリできます。プラットフォーム移行の際もデータの再ロードが不要になるため、移行コストが大幅に低減されます。ただし「多少避けることはできる」というレベルで、完全なロックインフリーは現実的には難しい側面もあります。各プラットフォームの独自機能（Cortex、Unity Catalog等）を使えば使うほど依存度が上がるため、どこまで独自機能を活用するかのトレードオフを意識することが重要です。
Q2
Icebergがレイクハウスフォーマットの筆頭となるのでしょうか？
A
ここ1〜2年の動きを見ていると、AWS・Google Cloud・Snowflake・DatabricksのいずれもがIcebergに最も力を入れている印象があるため、今はIcebergが筆頭と言えます。主要4プレーヤーが同時に注力していること、OSSとして独立したガバナンスを持っていること、RESTカタログ仕様の整備が進んでいることが根拠です。一方で、今回紹介したDuckLakeのように新しいレイクハウスフォーマットも登場しています。Icebergを中心に見つつ、類似技術の動向もウォッチし続けることをお勧めします。技術の地殻変動は想定外のところから来ることも多いので。
Q3
Omniはなぜそこまで伸びているのでしょうか？
A
これまでのセルフサービス型BIとSemantic Layer型BIの良いところ取りをしているからです。セルフサービス型BIはすぐにデータに繋いで分析できますが、ガバナンスが難しく同じ指標が人によって違う値になる問題がある。Semantic Layer型BIは定義を統一できますが、事前のモデリングが必要でスピードが落ちる。Omniは「GUIベースですぐにデータに繋いで計算フィールドや結合定義を行い、裏側でSemantic Layerのコードに変換される」というアプローチで、この両者の課題を同時に解決しています。ユーザーはExcelライクな直感的操作で分析でき、そのロジックがコードとして保存されるのでガバナンスも保たれる。この設計がユーザーから高い評価を受けています。
Q4
Semantic Layerでは何がおすすめでしょうか？
A
最近触った中ではOmniをお勧めします。豊富なAI用コンテキスト設定機能があり、Omni内でも多くの生成AIを使った機能が提供されています。ただし有償のため、気軽に試すならSnowflakeのSemantic ViewやDatabricksのUnity Catalog Metric Viewを第一歩として試してみるのが良いでしょう。これらはそれぞれのプラットフォームを利用していれば追加コストなく試せる入門としてお勧めです。Semantic Layerは最初のセットアップに一定のコストがかかりますが、一度整備すると全BIツールやAIエージェントへの文脈提供基盤として機能するため、長期的なROIは高い領域です。
