# ZJIT — 次世代Ruby JITコンパイラの設計と展望

RubyKaigi 2025 セッションノート — SSA-based IRで切り拓くRubyの高速化の未来

**発表者:** Maxime Chevalier-Boisvert — Shopify Ruby Infrastructure team

## 概要

- **中間表現:** SSA IR
- **パフォーマンス:** Fib比で60%高速化（YJIT比）
- **搭載予定:** Ruby 4.0 experimental搭載

---

## 01 前提知識 — YJITの成功と残された課題

YJITはRuby高速化の歴史において、最も大きな成果をもたらしたJITコンパイラです。Ruby 2.7との比較で3倍の実行速度向上を実現し、Rails のベンチマークでも15～20%の向上を記録しています。Shopify、GitHub、その他多数の企業が本番環境でYJITを有効化し、安定した成果を上げています。

### YJIT の限界

しかし、全てのワークロードが同じ度合いで最適化されたわけではありません。20%の改善が見られるメトリクスもあれば、変わらないメトリクスも存在します。特に以下の場面で課題が浮き彫りになりました：

- **データベースアクセス:** JITコンパイルでは最適化不可能な外部I/O処理
- **純粋計算性ワークロード:** 1億回の入れ子ループ反復でNode.jsに10倍の差で負ける現象
- **メモリ局所性:** キャッシュ効率やメモリレイアウトの改善余地

> **注目:** YJITのLBBV（Lazy Basic Block Versioning）は基本ブロック単位で最適化されるため、メソッド全体を見渡した最適化が不可能でした。この限界を突破するため、ZJITが開発されたのです。

実は、簡単な2重ループであっても、なぜ遅いのか。その答えはYJITのアーキテクチャの制約にありました。ブロック単位の最適化では、メソッド全体の最適化チャンスを逃していたのです。

## 02 ZJITのアーキテクチャ — 教科書的JITを目指して

ZJIT（ジェイアイティー）の名前は、YJITの次のジェネレーションを意味します。ZはYの次のアルファベットであり、次世代の意志を込めた命名です。ZJITの最大の特徴は、**Method-based JIT** です。

### Method-based JIT の意味

YJITのLBBVとは異なり、ZJITはメソッド全体をコンパイルの単位として扱います。この設計により、メソッド全体の制御フローを分析し、より効果的な最適化パスを適用することが可能になります。

### SSA-based IR（Static Single Assignment）

ZJITの核となるのが**SSA-based IR** です。これは、各変数が**一度だけ代入される**という形式に中間表現を変換します。例えば以下のようなコードは：

```ruby
x = 1
x = 2
y = x + 1
```

SSA形式では以下のように変数をバージョニングします：

```
x_1 = 1
x_2 = 2
y = x_2 + 1
```

このアプローチは、LLVM、GCC、HotSpot JVM など、最先端のJITコンパイラで採用されているものです。SSAの利点は、最適化パスが直交的（独立）に設計できることにあります。

### コンパイルパイプライン

```
YARV bytecode
    ↓
SSA HIR（High-level IR）
    ↓
最適化パス群
(CSE, DCE, SCCP等)
    ↓
機械語コード生成
```

このパイプラインにより、各最適化パスが独立して機能し、新しい最適化の追加も容易になります。

### YJITとZJITの比較

| 項目 | YJIT | ZJIT |
|------|------|------|
| コンパイル単位 | 基本ブロック (LBBV) | メソッド全体 |
| 中間表現 | なし（直接機械語） | SSA-based IR |
| 最適化範囲 | ブロック内 | メソッド全体 |
| 成熟度 | 本番環境対応 | experimental |
| 有効化 | デフォルト (Ruby 3.2+) | opt-in (--zjit) |

## 03 なぜLBBVから離れるのか

LBBV（Lazy Basic Block Versioning）は、多くの場面で有効であることが実証されています。実行されるブロックのみをコンパイルし、型情報に基づいて特殊化することで、メモリ効率と高速なコンパイルを両立していました。

### LBBVの強みと弱み

**メリット:**
- コンパイル時間が短い
- メモリ使用量が効率的
- 実行されるコードパスのみを最適化

**デメリット:**
- ブロックを超えた最適化ができない
- 中間表現（IR）がないため、高度な最適化パスの適用が難しい
- メソッド全体の制御フロー解析が不可能

### ZJITがもたらす改善

ZJITはメソッド全体を見渡すことで、以下のような最適化が可能になります：

- **ロード/ストア冗長性除去（Load-Store Optimization）:** 同じフィールドへの複数アクセスを検出し、不要なメモリ操作を削除
- **インライン化の強化:** メソッド間の呼び出しを分析し、より積極的なインライン化を実施
- **多段JITコンパイルの基盤:** 将来的にC1→C2相当の段階的コンパイルが可能

> **ヒント:** ZJITのIRベースのアプローチにより、コンパイラの保守性と拡張性が飛躍的に向上しました。新しい最適化パスの追加が容易になったため、コミュニティの貢献もしやすくなります。

## 04 パフォーマンスの現状

ZJITのパフォーマンスは、現在測定可能な段階に達しました。以下はRubyKaigi 2025で発表された数字です。

### CRuby呼び出しの高速化

C言語で書かれたRuby内部のメソッド呼び出し（CRuby calls）は、複雑で遅くなりやすい処理です。従来のYJIT呼び出しでもCRuby比で8～12倍速いのですが、ZJITではjmp命令からcall/ret命令への変更により、CPUの分岐予測機能をより効果的に活用できるようになりました。

具体例として、あるマイクロベンチマークでの時間変化：

```
純粋CRuby:  1ms
YJIT最適化: 0.016ms (62倍高速化)
ZJIT最適化: 0.01ms   (100倍高速化)
```

### マイクロベンチマーク結果

再帰的なフィボナッチ計算（Fib(40)）を実行したマイクロベンチマークでは、ZJITはYJITより約60%高速化を達成しました。

### 実アプリケーションでの結果

ただし、Rails などの実アプリケーションを計測すると、現段階ではYJITのほうがまだ全体的に速いという結果も出ています。その理由は、ZJITのSSA構築と最適化解析のオーバーヘッドがまだ大きいからです。チームは継続的にコンパイル効率を改善中です。

> **重要:** 現在のZJITは最適化に注力しており、コンパイル速度の改善は今後の課題です。Ruby 4.1でYJITとの性能パリティ達成を目標としています。

### 型の特殊化

Ruby のメソッド呼び出しの80～90%は monomorphic（単一の型のみが渡される）です。ZJITはこの特性を活かし、型情報に基づいた積極的な特殊化を行い、不要な型チェックを削減します。

## 05 冗長ロード・ストア除去の仕組み

ZJITの実装で特に注目される機能が、冗長ロード・ストア除去です。

### オブジェクトフィールドアクセスの最適化

Ruby オブジェクトのインスタンス変数にアクセスする際、ZJITは以下のような処理を行います：

- **LoadField 命令:** オブジェクトのフィールドをメモリから読み込み
- **StoreField 命令:** フィールドに値を書き込み

同じフィールドへの連続したロード操作が検出された場合、ZJITは既に持っている値を再利用し、不要なメモリアクセスを排除します。

```ruby
# オリジナルコード
class User
  def process
    @name = fetch_name()
    puts @name
    puts @name
    puts @name
  end
end

# ZJIT最適化後
# @name は1回のみロード、以後はキャッシュを使用
```

### GuardType命令とSide-Exit

ZJITは実行時に型チェックを行う GuardType 命令を挿入します。型チェックに失敗した場合、処理は自動的にインタプリタにフォールバック（side-exit）します。

開発者は、以下のコマンドラインオプションでside-exitの発生箇所をトレースできます：

```bash
ruby --zjit --zjit-trace-exits-sample-rate=N script.rb
```

このオプションにより、予期しない型変化やデプロイ後の性能低下原因を特定できます。

### 現在の実装範囲

現時点では、冗長除去はブロックローカルな最適化（ブロック内のみ）に限定されています。しかし、ZJITの設計により、今後メソッド全体への拡大が容易に実現可能です。

> **ベストプラクティス:** インスタンス変数の頻繁なアクセスが性能ボトルネックの場合、ZJITはそれを自動的に最適化します。コード側での特別な工夫は不要です。

## 06 デプロイとウォームアップの最適化

Shopify は世界最大級のRuby on Rails 運用企業です。数千台のサーバーで同じコードを実行しており、デプロイとウォームアップに関する課題が尖鋭化しています。

### 現在の課題

- **冗長なコンパイル:** 同じコードが数千台のサーバーで繰り返しコンパイルされている
- **ウォームアップ時間:** デプロイのたびに全サーバーでJITコンパイルの再開始が必要
- **初期レイテンシ:** デプロイ直後のリクエスト処理が顕著に遅い

### Persistent Code Caching の計画

ZJITの将来目標は、コンパイル済みコードの永続化（Persistent Code Caching）です。実装の鍵となるのが**ミニリンカー**です。

**デプロイフロー（将来像）**

デプロイ前：
```
Ruby プロセス起動 → JIT コンパイル → コード保存
```

デプロイ後：
```
Ruby プロセス起動 → キャッシュから読み込み → 即座に実行開始
```

> 99.99% のコードは変わらないため、キャッシュ再利用率が非常に高い

### ウォームアップコストの削減

コンテナベースのデプロイメント環境では、このキャッシング戦略が特に有効です。

**数字で見る効果:** Shopifyの内部測定では、Persistent Code Cachingの導入により、デプロイ後のレイテンシ改善が平均30～40%期待されています（Python での実装例から推定）。

開発チームは、既にRust での実装プロトタイプを作成中です。Ruby 4.1 でのマージを目指しています。

## 07 Ruby 4.0 でのZJITの状態

Ruby 4.0（2025年12月25日リリース予定）には、ZJITが**experimental**（実験的）として搭載されます。

### 有効化の方法

デフォルトではコンパイルされるものの、有効化されません。ユーザーはいくつかの方法で明示的に有効化できます。

**コマンドラインオプション:**
```bash
ruby --zjit script.rb
```

**環境変数:**
```bash
export RUBY_ZJIT_ENABLE=1
ruby script.rb
```

**プログラム内:**
```ruby
RubyVM::ZJIT.enable
# 以後のコードはZJITで実行
```

### ビルド要件

ZJITはRustで実装されているため、Rust 1.85.0以上が必要です。Ruby のコンパイル時に自動的に検出されます。

### 詳細なコマンドラインオプション

| オプション | 説明 | デフォルト値 |
|-----------|------|-----------|
| `--zjit` | ZJIT を有効化 | 無効 |
| `--zjit-call-threshold=num` | コンパイル開始の呼び出し回数閾値 | 30 |
| `--zjit-stats[=quiet]` | コンパイル統計を表示（quiet で抑制） | 表示 |
| `--zjit-trace-exits[=counter]` | side-exit をトレース（counter でカウント） | 無効 |

> **重要:** ZJITは現時点でexperimentalです。本番環境ではYJITの使用を推奨します。ZJITはRuby 4.1でYJITとの性能パリティを目指して開発中です。安定性が確認されるまでは、テスト環境での検証をお勧めします。

## 08 今後のロードマップ

### Ruby 4.1 — パリティ達成

開発チームの最大の目標は、Ruby 4.1（2026年3月リリース予定）でZJITがYJITと同等以上のパフォーマンスを発揮することです。

優先項目：
- SSA構築の高速化
- 最適化パスの並列実行
- 型推論のより正確な実装

### Persistent Code Caching

Ruby 4.1～4.2の時間軸で、Persistent Code Caching の実装が予定されています。これにより、大規模デプロイメント環境での効率が飛躍的に向上します。

### 多段コンパイル（Tiered Compilation）

将来的には、HotSpot JVMやV8で採用されている多段コンパイルの導入も検討されています。

- **Tier 1（C1相当）:** 軽量・高速なコンパイル
- **Tier 2（C2相当）:** 深い分析に基づく最適化

これにより、初期のコンパイルオーバーヘッドを最小化しつつ、長時間実行されるメソッドをさらに最適化できます。

### YJITとの共存期間

YJITとZJITの共存は当面継続されます。ユーザーは必要に応じて選択できます。ZJITが完全に成熟し、安定性が確保されたタイミングで、YJITは段階的に非推奨化される予定です。

### アップストリーム状況

ZJITはFeature #21221 として正式にRuby のアップストリームにマージされています。開発は Ruby Development チームとShopify の協力の下、進行中です。

> **ヒント:** Ruby 4.0 のreleaseから、ZJITの開発速度は加速するでしょう。コミュニティからの早期フィードバックが、最適化の優先順位を決める重要な要素になります。

## まとめ

ZJITは、YJITの成功を基盤としながら、SSA-based IRの導入によって、より強力で拡張性の高いJITコンパイラを目指しています。

**主要な特徴:**
- Method-based JIT による全メソッド最適化
- SSA-based IR による直交的な最適化パス
- 冗長除去やメモリ最適化の強化
- Persistent Code Cachingによる大規模デプロイの効率化

**開発スケジュール:**
- **Ruby 4.0（2025年12月）:** Experimental搭載
- **Ruby 4.1（2026年3月）:** YJITとの性能パリティ達成を目標
- **Ruby 4.1～4.2:** Persistent Code Caching実装

コミュニティからのフィードバックが、ZJITの最適化優先順位を決める重要な要素となります。
