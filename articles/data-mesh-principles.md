## 第1章：Data Mesh とは何か（ナットシェル） {#第1章}

### コア定義

**Data Mesh** = 複雑・大規模環境において、分析用データの共有・アクセス・管理を行うための **分散型ソシオテクニカル（社会技術的）アプローチ**。

ここで言う「分析用データ（Analytical Data）」とは、レポーティング・診断・機械学習に使われるデータのことであり、トランザクションや業務系の運用データ（Operational Data）とは区別される。

### Data Mesh が目指す3つのアウトカム

| アウトカム | 説明 |
|---|---|
| **Agility at Scale（規模でのアジリティ）** | 大規模環境でもスピードを維持できる組織的俊敏性 |
| **Data-Driven Innovation（データ駆動のイノベーション）** | データを活用した新しい価値創造の継続 |
| **Data Democratization（データの民主化）** | 組織全体でデータにアクセス・活用できる状態 |

### 3つのパラダイムシフト

Data Mesh は、以下の従来の考え方を根本から転換する。

1. **中央集権型** → **ドメインオーナーシップへ分散**
2. **データを副産物** → **データをプロダクト**として設計・提供
3. **中央集権型ガバナンス（アドホック）** → **連合型計算ガバナンス（自動化）**

さらに、個別に構築されたビスポーク（一点物）パイプラインから、**セルフサーブ・プラットフォーム**へと移行する。

### 4つの相互依存する原則

Data Mesh は以下の4原則が**相互依存**している点が重要。どれか1つを欠いても機能しない。

1. **ドメインオーナーシップの原則**（第2章）
2. **データをプロダクトとして扱う原則**（第3章）
3. **セルフサーブ・データプラットフォームの原則**（第4章）
4. **連合型計算ガバナンスの原則**（第5章）

### Data Product Quantum（DPQ）

**DPQ（データプロダクト量子）** は、Data Mesh における最小の独立したデプロイ可能ユニットで、以下の3要素から構成される。

- **コード（Code）**: データプロダクトのロジック・変換処理
- **データとメタデータ（Data/Metadata）**: 実際のデータとその記述
- **インフラストラクチャ（Infrastructure）**: 動作に必要な基盤

### Data Mesh の起源：3世代の失敗

Dehghani は、データアーキテクチャの歴史的失敗から Data Mesh を導出している。

| 世代 | アーキテクチャ | 主な問題 |
|---|---|---|
| 第1世代 | データウェアハウス（DWH） | 中央チームへのボトルネック、スケール不可 |
| 第2世代 | データレイク | ガバナンス欠如、「データの沼」化 |
| 第3世代 | クラウドプラットフォーム | テクノロジーは進化したが組織問題は未解決 |

**根本原因は技術的ではなく、組織的なものだった。** テクノロジーだけを変えてもデータサイロや中央集権的ボトルネックは解消されない。

### DDD とマイクロサービスの応用

Data Mesh は、ソフトウェアエンジニアリングで実証済みの2つの概念を分析データ領域に適用している。

- **ドメイン駆動設計（DDD）**: 業務ドメインを中心とした設計
- **マイクロサービスアーキテクチャ**: 責任の分散と自律性

## 第2章：ドメインオーナーシップの原則 {#第2章}

### 基本思想

データの責任を**そのデータを最もよく知る人たち**（= データを生み出すドメイン）へ分散・移譲すること。「データを理解している人」が「データの責任」も持つ。

### 分解の軸：テクノロジーではなくビジネスドメイン

従来のデータチームは「インジェスチョン」「変換」「提供」といった**技術的な機能**で縦割りされていた。Data Mesh では、**ビジネスドメインの境界**に沿って責任を分割する。

これはDDD（ドメイン駆動設計）の考え方を直接適用したもので、**バウンデッドコンテキスト（Bounded Context）** ごとにデータの所有権を定義する。

### 3種類のドメインデータアーキタイプ

| アーキタイプ | 定義 | 例 |
|---|---|---|
| **Source-Aligned（ソース整合）** | 業務システムの実データに近い形で提供 | 注文ドメインの生注文データ |
| **Aggregate（集約型）** | 複数ソースを統合・集約したデータ | 全社売上サマリー |
| **Consumer-Aligned（コンシューマー整合）** | 特定のコンシューマーのユースケースに特化 | MLモデル学習用の特徴量データ |

### データオーナーシップの上流化

データオーナーシップを「上流」（= データを生み出す側）に押し上げることが重要。中央のデータチームに押し付けるのではなく、各ドメインが自分たちのデータに責任を持つ。

### 単一真実源への執着を捨てる

Data Mesh は「唯一の真実（Single Source of Truth）」という概念を否定する。代わりに、**複数のコンテキスト的な真実**を受け入れる。異なるドメインが異なる視点から同じ現実を表現することを許容する。

### データパイプラインを内部実装として隠す

各ドメインのデータパイプラインは、そのドメインの**内部実装の詳細**として扱う。外部（他ドメイン）はパイプラインの存在や実装を気にせず、公開されたデータプロダクトのみを利用する。

### 基盤となる原則

ドメインオーナーシップは**基盤原則**であり、これが機能しないと残りの3原則も成立しない。他の原則はドメインオーナーシップを前提として機能する。

## 第3章：データをプロダクトとして扱う原則 {#第3章}

### 課題の定義：フリクション問題

Anaconda 2020年のレポートによると、データサイエンティストの約**50%の時間**がデータの前処理に費やされている。データへのアクセスが困難、品質が不明確、文書化が不十分 — これらが「データフリクション」を生み出す。

ドメインオーナーシップだけでは、この問題を解決するどころか**データサイロをより悪化させる**可能性がある。そこで必要なのが「データをプロダクトとして扱う」という原則。

### プロダクト思考の適用

- データコンシューマー = **顧客**
- 各ドメインは**データプロダクトオーナー**を置く
- データは「副産物」ではなく「プロダクト」として意図的に設計・提供される

### 8つのユーザビリティ特性

高品質なデータプロダクトは以下8つの特性を備える。

| # | 特性 | 説明 |
|---|---|---|
| 1 | **Discoverable（発見可能）** | データカタログや検索で見つけられる |
| 2 | **Addressable（アドレス可能）** | 一意のアドレスでアクセスできる |
| 3 | **Understandable（理解可能）** | データの意味・文脈が明確に文書化されている |
| 4 | **Trustworthy & Truthful（信頼可能・真実）** | SLOが定義され、品質が保証されている |
| 5 | **Natively Accessible（ネイティブアクセス可能）** | 標準的な方法で直接アクセスできる |
| 6 | **Interoperable & Composable（相互運用可能・組み合わせ可能）** | 他のデータプロダクトと連携できる |
| 7 | **Valuable on Its Own（単独で価値がある）** | 単体で意味のある価値を提供する |
| 8 | **Secure（セキュア）** | アクセス制御が適切に実装されている |

### Data Product Quantum（DPQ）の再確認

データプロダクトの最小デプロイ可能ユニット。コード・データ/メタデータ・インフラの三位一体で独立して機能する。

### データプロダクトオーナーの役割

- データプロダクトのライフサイクル全体の責任者
- 品質基準の定義と維持
- コンシューマー体験の向上
- 継続的な改善の推進

### 変容：副産物からプロダクトへ

この原則によって、データは「業務システムの副産物」から「オーナーシップ、品質基準、継続的改善を伴うプロダクト」へと変容する。

## 第4章：セルフサーブ・データプラットフォームの原則 {#第4章}

### 目的

ドメインチームが**自律的に**データプロダクトを構築・デプロイ・維持できるよう、インフラの複雑性を抽象化すること。

プラットフォームなしにドメインオーナーシップを要求することは、ドメインチームに過大な負担を強いることになる。

### 設計目標

- ドメインの自律性を実現する
- 複雑性を隠蔽する
- 専門化の障壁を下げる
- ドメイン非依存（汎用的）であること

### 3層のケイパビリティ構成（マルチプレーンアーキテクチャ）

```
┌─────────────────────────────────────────────────────┐
│  Plane 3: Data Mesh Supervision Plane（最上位）       │
│  発見・セマンティッククエリ・カタログ・セキュリティ・監視  │
├─────────────────────────────────────────────────────┤
│  Plane 2: Data Product Developer Experience Plane   │
│  宣言的インターフェース・スキーマレジストリ・              │
│  パイプラインオーケストレーション・リネージュ              │
├─────────────────────────────────────────────────────┤
│  Plane 1: Data Infrastructure Provisioning Plane    │
│  分散ストレージ・ACL・オーケストレーション・クエリエンジン  │
└─────────────────────────────────────────────────────┘
```

| プレーン | 層 | 主な機能 | 利用者 |
|---|---|---|---|
| **Data Infrastructure Provisioning Plane** | 最下位 | 分散ストレージ、ACL、オーケストレーション、クエリエンジン | 他プレーンまたは上級開発者 |
| **Data Product Developer Experience Plane** | 中間 | 宣言的インターフェース、スキーマレジストリ、パイプラインオーケストレーション、リネージュ | ドメインチームの開発者 |
| **Data Mesh Supervision Plane** | 最上位 | グローバルな発見、セマンティッククエリ、カタログ、セキュリティ、監視 | 全組織・ガバナンスチーム |

### プラットフォームチームの役割

プラットフォームチームは**プロダクトチームとして運営**する。ドメインチームが「顧客」であり、プラットフォームの開発優先順位はドメインのニーズに基づく。

### 現状の課題

運用系テックスタック（Docker/Kubernetes）とデータ系テックスタック（Spark/Databricks）の乖離が現在の主要な技術的課題となっている。この2つの世界の統合が、セルフサーブ実現の鍵。

## 第5章：連合型計算ガバナンスの原則 {#第5章}

### 中央集権vs無政府状態のジレンマを解く

- **中央集権型**: ボトルネック、遅延、スケール不可
- **無政府状態**: データアナーキー、互換性なし、コンプライアンス違反

Data Mesh の解答：**連合型計算ガバナンス**

### 「連合型（Federated）」の意味

意思決定を「**ドメインのデータプロダクトオーナーとプラットフォームプロダクトオーナーの連合体**」が行う。中央チームが独裁的に決定するのではなく、各ドメインの代表者が集まって決定する（ギルドモデル）。

### 「計算型（Computational）」の意味

ポリシーを**コードとして実装**し、プラットフォームが自動的に適用・強制する。人手による文書ベースのガバナンスではなく、機械が実行可能なポリシーとして表現する。

### グローバル基準 vs ローカル決定

| 種別 | 内容 |
|---|---|
| **グローバル基準（全ドメイン共通）** | 相互運用性、セキュリティ、コンプライアンス、SLO定義、識別子体系 |
| **ローカル決定（各ドメイン自律）** | ドメインデータモデル、内部ロジック、スキーマ設計 |

### ガバナンスチームの構成

各ドメインの代表者から構成される**ギルドモデル**を採用。中央の専任チームではなく、各ドメインからの参加者が集まって意思決定を行う。

### 成功指標

成功は「管理されたデータの量」ではなく、**データ消費の接続数（ネットワーク効果）** で測定する。データプロダクトが広く使われているか、組織全体でデータの流れが増えているかが指標。

### エラー対応の哲学

- **旧来**: 予防主義（手動チェック、承認プロセス）
- **Data Mesh**: 検出と回復（自動化された検知・修正）

### ポリシーの管理方法

ポリシーは**Gitで管理可能**：

- バージョン管理（変更履歴の追跡）
- コードレビュー（変更の審査プロセス）
- テスト可能（ポリシーの自動検証）

### Pre-Data-Mesh vs Data Mesh のガバナンス比較

| 観点 | 従来（Pre-Data-Mesh） | Data Mesh |
|---|---|---|
| **チーム構成** | 中央集権チーム | 連合型チーム（各ドメイン代表） |
| **適用方法** | 手動（文書ベース） | 自動化（コードベース） |
| **ポリシー** | 静的ドキュメント | 動的コード（Git管理） |
| **意思決定** | トップダウン | ボトムアップ＋合意形成 |
| **エラー処理** | 予防重視 | 検出と回復 |
| **スケーラビリティ** | スケールしない | スケール可能 |
| **変更速度** | 遅い（承認待ち） | 速い（自動適用） |

## 4原則の相互関係まとめ {#まとめ}

Data Mesh の4原則は独立して機能するものではなく、**相互に依存する一つのシステム**を形成している。

```
┌─────────────────────────────────────────────────┐
│                  Data Mesh                       │
│                                                  │
│  ①ドメインオーナーシップ ←→ ②データプロダクト   │
│           ↕                        ↕             │
│  ③セルフサーブプラットフォーム ←→ ④連合ガバナンス │
└─────────────────────────────────────────────────┘
```

| 依存関係 | 説明 |
|---|---|
| ①→② | ドメインが責任を持つことで、はじめてデータをプロダクトとして扱えるようになる |
| ①→③ | プラットフォームなしにドメインに責任を持たせるのは無理 |
| ②→④ | データプロダクトが存在してはじめてガバナンスが機能する |
| ③→④ | プラットフォームがガバナンスポリシーの「計算的」な実行を担保する |

### 学習から得た洞察

Data Mesh の本質は**テクノロジーの問題ではなく、組織とカルチャーの問題**への回答である。  
3世代のデータアーキテクチャが失敗した根本原因（中央集権的な組織構造）に対して、ソフトウェアエンジニアリング（DDD、マイクロサービス）の知見を援用して解決しようとしている点が革新的。
