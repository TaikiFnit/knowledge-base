01
Introducing Data Governance

データガバナンスとは

"Data is the new oil." — Clive Humby

第1章は、データがなぜ「新しい石油」と呼ばれるほど価値を持つようになったのか、そしてその価値あるデータを適切に管理しなかった場合に何が起きるかを実世界のインシデントで示すところから始まる。

データガバナンスの定義

// Core Definition

Data governance とは、組織のデータを管理するための包括的なアプローチであり、データの可用性（availability）、使用可能性（usability）、完全性（integrity）、セキュリティ（security）を確保する。

データガバナンスの主要コンポーネント

COMPONENT 01

ポリシーと標準

Policies and Standards

データの取り扱いルール・組織横断的な標準を策定し、全従業員が遵守すべき指針を明確にする。

COMPONENT 02

データ品質管理

Data Quality Management

データの正確性・完全性・一貫性を継続的に監視・改善する仕組み。

COMPONENT 03

メタデータ管理

Metadata Management

データに関するデータ（スキーマ、所有者、作成日時、ビジネス定義など）を一元管理する。

COMPONENT 04

アクセス制御とセキュリティ

Access Controls & Security

暗号化、ファイアウォール、侵入検知、データ最小化、定期監査、インシデント対応計画を含む。

COMPONENT 05

データライフサイクル管理とリネージ

Lifecycle Management & Lineage

データの生成から廃棄までの全過程を追跡し、変換履歴・依存関係を可視化する。

COMPONENT 06

監査とコンプライアンス監視

Auditing & Compliance Monitoring

すべてのデータ操作を記録し、規制要件への準拠を継続的に確認する。

効果的なデータガバナンスの利点

  • 人的エラーの削減 — 明確なポリシーと制御により、データライフサイクル全体を通じた人的エラーを軽減
  • 強力なアクセス制御 — 認証とアクセス制御の統合的な実装
  • データ分類とリスク評価 — データの機密レベルに基づく適切な分類と保護
  • データ取扱ポリシー — 保存、処理、転送、廃棄の各段階でのルール策定
  • データ品質と信頼性の向上 — 意思決定に使用されるデータの正確性を保証
  • 規制コンプライアンスの強化 — GDPR、CCPA等の法規制への準拠を促進
  • 運用効率の向上とコスト削減 — 重複作業・データ管理コストの低減
  • 消費者からの信頼向上 — データ保護に対する信頼の獲得

また、ガバナンスは人間中心（human-centric）であることが重要だ。定期的な従業員トレーニング、明確な役割と責任の定義が、82%のセキュリティ侵害の原因となっている人的要素への対策となる。

02
Data Lifecycle

データのライフサイクル

データには4つの主要なライフサイクルステージがある。各ステージでガバナンスが適用され、データの品質・セキュリティ・コンプライアンスが維持される必要がある。

// データライフサイクルの4段階（Figure 1-1）
  • 01
    Stage 1 — Creation / Ingestion

    データの作成・取得

    データは3つの方法で生まれる：収集（ユーザーサインアップ等）、生成（マシンシグナル・IoTセンサー等）、取り込み（サプライヤーカタログ等の外部データ）。データの「誕生」の段階から、所有者・分類・アクセス権限の定義が始まる。

  • 02
    Stage 2 — Processing

    データの処理

    取得された生データに対して、クレンジング、結合・マージ、集計といった処理が行われる。この段階ではデータの品質向上・標準化が主な目的であり、コンプライアンスに関わる匿名化処理もここで実行される。

  • 03
    Stage 3 — Active Duty

    アクティブな利用

    処理されたデータがアプリケーション、レポーティング、高度なアナリティクスで積極的に利用されるフェーズ。データの価値が最大限に発揮される段階であり、同時にアクセス制御と監視が最も重要になる。

  • 04
    Stage 4 — Offboarding

    データの退役（パージ/アーカイブ）

    規制要件とコストに基づき、データをパージ（完全削除）するかアーカイブ（長期保存）するかを決定する。GDPRの「忘れられる権利」等、法規制がこの段階の判断に直接影響する。

処理段階の重要な概念

Entity Resolution（エンティティ解決）

複数のシステムから同一のデータを統合し、冗長性を排除する処理。例：同一顧客が異なるシステムで異なるIDを持っている場合、それらを一つに統合する。

Data Harmonization（データ調和）

様々なソースからのデータを一貫性のある標準化されたフォーマットに統一する処理。テーブル結合（join）を伴い、異なる命名規則・データ型の統一を行う。

Aggregation（集約）

大規模データセット（例：IoTセンサーデータ）から意味のある情報を抽出する処理。個別レコードの詳細を集約し、傾向やパターンを明らかにする。

Anonymization（匿名化）

コンプライアンス要件に基づくデータのマスキング処理。墨消し（redaction）または仮名化（pseudonymization）の手法を用いる。

// PII と PHI の区別

PII（Personally Identifiable Information）: 個人を特定できるあらゆる情報。氏名、住所、社会保障番号、メールアドレス等。

PHI（Protected Health Information）: PIIのサブセットであり、医療分野における個人の健康情報。米国ではHIPAA（Health Insurance Portability and Accountability Act）により保護される。

03
Governing the Ungoverned

非構造化データとAIアセットのガバナンス

従来のガバナンスは構造化データ（テーブル形式のデータ）を主な対象として設計されてきた。しかし現実には、組織データの大部分は構造化されていない。

// 非構造化データの現実
>80%
組織データのうち非構造化データの割合
（メール、テキストファイル、ログ、メディアファイル等）
Data
Swamp
管理されない非構造化データが蓄積され、利用不能な「データ沼」が形成される

AIの台頭により、非構造化データの価値が急速に高まっている。組織は非構造化データを競争優位に活用し始めており、ガバナンスの対象範囲を拡大する必要性が高まっている。

AIが拡大するガバナンス対象

AIは以下の技術領域をカバーする：

  • 予測分析（Predictive Analytics） — 過去データから将来を予測
  • 処方分析（Prescriptive Analytics） — 最適なアクションを推奨
  • 従来型ML（Traditional Machine Learning） — 分類・回帰等の教師あり/なし学習
  • 生成AI（Generative AI） — テキスト・画像・コードの自動生成
// AIアセットという新しいガバナンス対象

データセットを用いて学習されたMLモデルは、データそのものとは異なる新しい種類のアセット（AI assets）だ。これらのAIアセットにもガバナンスが必要だが、既存のガバナンスソリューションの多くはAIアセットのガバナンスに対応できていない。

Unity Catalogが目指すのは、データアセットとAIアセットの両方を統合的にガバナンスすることだ。

04
Regulatory Compliance

規制への対応

ChatGPTがAIを一般大衆に広く普及させたことで、AIに関する規制の動きが世界的に加速している。データガバナンスはもはや「あれば良い」ではなく、法的に必須な要件になりつつある。

EU
AI Regulation

EU AI Act

2024年7月に公布されたAIの利用を規制する法律。AIシステムをリスクレベル別に分類し、高リスクAIには厳格な要件を課す。世界初の包括的AI規制として、グローバルな影響力を持つ。

→ 2024年7月公布 — 世界初の包括的AI規制
EU
Data Protection

GDPR（一般データ保護規則）

EU域内の個人データ保護に関する規制。データ主体の権利（アクセス権、削除権「忘れられる権利」、ポータビリティ権）を包括的に定義。データガバナンスの基盤となる規制。

→ EU域内 — 個人データ保護の金字塔
US
Privacy

CCPA（カリフォルニア消費者プライバシー法）

カリフォルニア州のプライバシー保護法。消費者にデータの収集・共有・削除に関する権利を付与。米国における最も影響力のあるプライバシー法として、他州のモデルとなっている。

→ 米国 — カリフォルニア州のプライバシー保護

モダンなガバナンスソリューションは、データだけでなくAIアセットを含む拡大した規制範囲に包括的に対応する必要がある。第10章で詳細に解説される。

05
The Dawn of the Lakehouse

レイクハウスの夜明け

データガバナンスの必要性を理解したところで、次はデータをどのように保存・処理し、価値を生み出すかというアーキテクチャの話に移る。人間、マシン、その他のソースからデータを取得し、ダッシュボード・レポート・MLモデルを生成するまでの道のりだ。

BI vs 高度なアナリティクス

BI（ビジネスインテリジェンス）
過去・現在からのインサイト抽出
レポート、ダッシュボードの作成
構造化された表形式データが対象
主にデータウェアハウスで実行
ADVANCED ANALYTICS（高度なアナリティクス）
予測・処方的なユースケース
MLモデル、統計分析、AI推論
半構造化・非構造化データも対象
データレイクで実行されることが多い
// データの構造レベル

構造化データ: テーブル形式（行と列）。RDBに格納。

半構造化データ（Semi-structured）: パターンはあるがテーブル形式ではない。ログファイル、JSON、XMLなど。

非構造化データ（Unstructured）: メディアファイル、画像、プレーンテキスト。一貫した構造を持たない。

ビッグデータの3V

Doug Laney（Gartner, 2001）がビッグデータの3つのV——Volume（量）、Velocity（速度）、Variety（多様性）——を定義した。 — Doug Laney, 3D Data Management (2001)

VolumeとVelocityはクラウドの進化によりコモディティ化された。現在最も挑戦的なのはVariety（多様性）——異種混合のデータソースをいかに統合・管理するかという課題だ。

データウェアハウス vs データレイク

観点 Data Warehouse Data Lake
主な用途 BIとレポーティング 高度なアナリティクス、ML
データ型 構造化データ 半構造化・非構造化データ
代表例 Azure Synapse, Amazon Redshift, Google BigQuery Amazon S3, Azure Data Lake Storage, Google Cloud Storage
処理の流れ 収集 → 格納 → 変換 → 提供 収集 → 格納 → 変換 → 提供
// Data Mart（データマート）とは

単一のビジネスユニット（マーケティング、営業等）向けにキュレーションされたデータベース。データウェアハウスのサブセットとして機能する。

"If you think of a data mart as a store of bottled water — cleansed and packaged, and structured for easy consumption — the data lake is a large body of water in a more natural state." — James Dixon（Pentaho元CTO）

Hadoop: 偽りの夜明け

著者Karthikは2010〜2011年頃、フィンテックスタートアップでHadoopに携わったキャリアを持つ。HDFS + MapReduceは分析データのパラダイムシフトとなり、Cloudera、MapR、Hortonworksが設立された。LinkedIn、Yahoo、Facebookが採用した。

しかし、ディスクベースであるHadoopは現代のシステムと比較して低速であり、レガシーとなった。良い結果として、UC BerkeleyでApache Sparkが誕生した——これがレイクハウスの基盤技術となる。

クラウドデータレイクとデータサイロ

クラウドのconsumption-based model（従量課金モデル）により、あらゆる種類のデータを安価でスケーラブルなオブジェクトストレージに保存できるようになった。

しかし、複数の事業部門が異なるツール・プラットフォームを選択することでデータサイロが形成される。著者はこれをベンガルール（バンガロール）の交通に例える——素晴らしい都市だが、インフラが成長に追いついていない。

// フラクチャードガバナンスの問題

2つの分離されたスタック（ウェアハウス + レイク）を運用することは、分断されたガバナンス（fractured governance）を生む。各スタックに異なるアクセス制御、異なるメタデータ管理、異なる監査ログが存在し、組織全体の一貫したガバナンスが不可能になる。

ビルド vs バイの意思決定が問題をさらに悪化させ、技術的負債（technical debt）が蓄積される。

06
The Lakehouse Paradigm

レイクハウスパラダイム

// Lakehouse の定義

Lakehouse とは、データレイクとデータウェアハウスの最良の要素を組み合わせたオープンなデータ管理アーキテクチャであり、単一のプラットフォーム上で統合ガバナンスとAI駆動アナリティクスを実現する。

目標：① すべてのデータワークロード（BI、高度なアナリティクス等）② すべてのデータ型（構造化、半構造化、非構造化）③ 統合されたセキュリティとガバナンス

「data lakehouse」という用語が初めて文書化されたのは2017年。先駆者としてImpala（Cloudera）とPresto（Facebook/Meta）がある。

レイクハウスアーキテクチャ

レイクハウスの構築は、クラウドオブジェクトストレージ（S3、ADLS、GCS）を起点とする。データレイク上でACIDトランザクションを実現するために、オープンテーブルフォーマットが不可欠だ。

H
Open Format

Apache Hudi

Uberが開発。ACIDトランザクション、スキーマ進化、タイムトラベル、効率的なメタデータ管理を提供するオープンソースのテーブルフォーマット。

→ Uber発のインクリメンタル処理に強み
I
Open Format

Apache Iceberg

Netflixが開発。大規模データセット向けのテーブルフォーマット。パーティション進化、隠しパーティション等の高度な機能を提供。

→ Netflix発の大規模データ向け設計
Δ
Open Format

Delta Lake

Databricksが開発。Apache Spark上で動作するオープンソースストレージレイヤー。ACIDトランザクション、スキーマ強制、タイムトラベルを提供。Databricksエコシステムの中核。

→ Databricks/Sparkエコシステムの基盤

ACID vs BASE

特性 ACID BASE
正式名称 Atomicity, Consistency, Isolation, Durability Basically Available, Soft state, Eventually consistent
一貫性モデル 強整合性（Strong Consistency） 結果整合性（Eventual Consistency）
適用場面 トランザクション処理、金融データ、ガバナンスが必要な場面 高可用性が優先される分散システム
レイクハウスでの位置付け オープンテーブルフォーマット（Delta Lake等）がデータレイク上でACIDを実現 従来のデータレイクのデフォルト

コンピュートレイヤーにはApache Sparkがストレージから分離（デカップル）されて配置される。データ処理レイヤーではバッチ処理、ストリーム処理、ML、高度なアナリティクスが実行される。

メダリオンレイクハウスアーキテクチャ

Medallion architecture（メダリオンアーキテクチャ）はマルチホップアーキテクチャとも呼ばれ、データの品質が段階的に向上するレイヤード処理パターンだ。

// メダリオンアーキテクチャ（Figure 1-7）
  • B
    Bronze Layer — 生データ

    ブロンズレイヤー

    ソースシステムからの生データをそのまま格納。データの最初の着地点。変換は最小限で、元データの忠実な保存が目的。

  • S
    Silver Layer — クレンジング済み

    シルバーレイヤー

    検証・変換・エンリッチメントを経たデータ。重複排除、スキーマ適用、ビジネスルール適用が行われる。ACIDトランザクションが品質を保証。

  • G
    Gold Layer — ビジネス最適化

    ゴールドレイヤー

    読み取り最適化された、効率的なクエリパフォーマンスのためのデータ。BIダッシュボード、KPIレポート、MLモデルのフィーチャーストアなど、最終消費向けのキュレーションされたデータセット。

Databricks Data Intelligence Platform

DatabricksはApache Sparkに起源を持ち、すべてのアナリティクスワークロードに拡大した。統合的・オープン・インテリジェントなプラットフォームとして、以下のコアコンポーネントを持つ。

COMPONENT 01

コスト効率の高いオブジェクトストレージ

Object Storage for All Data/AI Assets

すべてのデータおよびAIアセットを安価で耐久性の高いクラウドストレージに保存。

COMPONENT 02

オープンソースストレージレイヤー

Delta Lake — ACID, Schema Evolution, Time Travel

Delta Lakeによりデータレイク上でACIDトランザクション、スキーマ進化、タイムトラベルを実現。

COMPONENT 03

統合カタログ

Unity Catalog — Metastore, ACL, Audit, Lineage

メタストア、アクセス制御、監査、リネージを提供する統合カタログ。

COMPONENT 04

インテリジェンスエンジン

Intelligence Engine

データ、コード、使用状況に関するコンテキスト情報を活用してインテリジェントな機能を提供。

COMPONENT 05

全ワークロード対応

BI, Data Eng, Data Science, ML, Real-time

BI、データエンジニアリング、データサイエンス、ML、リアルタイムアナリティクスを単一プラットフォームで実行。

Databricksの主要な能力として、データサイエンス/ML（MLflow）、データウェアハウジング（Databricks SQL）、ガバナンス（Unity Catalog）が統合されている。

07
Unity Catalog: Unified Governance

Unity Catalog: 統合ガバナンスの実現

Unity Catalogのタイムライン

// Unity Catalog の歩み
  • 21
    2021 — Data and AI Summit

    Unity Catalog発表

    Databricksの年次カンファレンスData and AI Summit 2021でUnity Catalogが初めて発表された。

  • 22
    2022 — Summit / Public Preview

    ゲーテッドパブリックプレビュー開始

    Summit 2022でAzure/AWS向けにゲーテッドパブリックプレビューが提供された。

  • GA
    August 2022 — General Availability

    一般提供開始（GA）

    2022年8月にUnity Catalogが一般提供（Generally Available）となった。

なぜUnity Catalogが必要なのか

Unity Catalog以前は、Hive Metastore（HMS）がメタデータリポジトリとして使用され、アクセス制御はワークスペースごとに管理されていた。組織が数十〜数百のワークスペースを運用する段階になると、この方法は複雑で、エラーが発生しやすく、スケーラブルではないという限界を露呈した。

// アネクドート：「Bob」の教訓

大規模小売企業の「Bob」は、統合ガバナンスがないことの影響を想像できなかった。ワークスペースごとに分断されたアクセス制御、一貫性のないメタデータ、組織全体で追跡不能なデータリネージ——これらの問題は、ワークスペース数の増加とともに指数関数的に悪化する。

Unity Catalogの特性

O
Characteristic

Operational（運用カタログ）

Unity Catalogは単なるメタデータカタログではなく、アクセス制御・監査・リネージを実行する運用カタログだ。統合的かつオープンな設計で、あらゆる主要クラウドプロバイダー上で動作する。

→ メタデータ管理を超えた統合運用
U
Characteristic

Unified（統合）

すべてのデータおよびAIアセットに対する単一のガバナンスモデル。Delta、Iceberg、Hudi、CSV、JSON、Parquet、テキスト、バイナリなど複数のフォーマットをサポート。

→ 単一モデルで全アセットをガバナンス
O
Characteristic

Open（オープン）

オープンソースでベンダーロックインなし。Credential vending（外部アプリ向け一時的アクセス）とCatalog federation（外部カタログとのメタデータ交換）をサポート。

→ オープンソース、ベンダーロックインなし
// UC OSS vs Databricks UC

UC OSS（オープンソース版）= スタンドアロンの統合データ/AIカタログ。

Databricks UC = Databricksプラットフォームに組み込まれたガバナンスソリューション。OSS版の機能に加え、Databricksの管理機能が統合されている。

データ/AIアセットの種類

アセット種別 英語名 説明
テーブル Tables 行と列で構成される構造化データ
ビュー Views テーブルやビューから作成される読み取り専用のデータ
ボリューム Volumes 任意のファイルの論理エンティティ（非構造化データ対応）
モデル Models AIモデル（MLflowで管理）
関数 Functions ユーザー定義ロジック（UDF）
外部カタログ Foreign Catalogs 外部DB、ウェアハウス、カタログ（例：AWS Glue）

コアガバナンス機能（6つ）

CORE 01

ID管理

Identity Management

Entra ID、Google、Okta等の外部IDプロバイダーと統合。組織の既存ID基盤をそのまま活用できる。

CORE 02

カタログ

Catalog (Metastore)

すべてのデータおよびAIアセットのメタストア。統一された名前空間で組織のデータ資産を管理する。

CORE 03

アクセス制御

Access Control

アイデンティティに対する認可。誰がどのデータに何を行えるかを細粒度で制御する。

CORE 04

検索と発見

Search and Discovery

カタログに登録されたアセット、メタデータ、タグに対する自然言語検索。データの発見可能性を大幅に向上。

CORE 05

監査

Audit

すべてのデータ操作を自動的にログ記録。コンプライアンス監査・インシデント調査に不可欠。

CORE 06

リネージ

Lineage

テーブル、ビュー、ボリューム、モデルのリネージを自動生成。データの出所と変換履歴を追跡可能にする。

高度なガバナンス機能（3つ）

A1
Advanced

行レベルフィルタリング & 列レベルマスキング

Row-level filteringとColumn-level masking（CLM）により、機密データへのアクセスを細粒度で制御。同一テーブルに対して、ユーザーの権限に応じて異なるデータビューを提供できる。

→ 機密データのきめ細かな保護
A2
Advanced

属性ベースアクセス制御（ABAC）

Attribute-Based Access Control: アイデンティティ属性、リソース属性、リクエスト属性に基づく条件付きアクセス制御。従来のRBAC（ロールベース）を超えた柔軟なポリシー定義が可能。

→ 動的な条件に基づくアクセス制御
A3
Advanced

データモニタリング

テーブルレベルでの統計的特性とデータ品質の自動モニタリング。データドリフト、異常値、品質劣化を自動的に検出し、プロアクティブなデータ管理を実現する。

→ テーブルレベルの自動品質監視
08
Platform Architecture

Databricksプラットフォームアーキテクチャ

Databricksプラットフォームは4つの主要コンポーネントから構成される（Figure 1-10）。

// Databricks プラットフォーム 4コンポーネント
  • 01
    Component 1 — Users & Applications

    ユーザーとアプリケーション

    プラットフォームにアクセスするエンドユーザー、サービスプリンシパル、外部アプリケーション。

  • 02
    Component 2 — Control Plane（Databricks管理）

    コントロールプレーン

    Databricksが管理するサービス群：Web UI、クラスター管理、ワークフロー管理、Unity Catalogサービス、ノートブックサーバー。ユーザーから直接アクセスされることはない。

  • 03
    Component 3 — Compute Plane

    コンピュートプレーン

    実際のデータ処理が行われる場所。サーバーレス（Databricks管理）とクラシック（顧客管理）の2つのデプロイメントモデル。Databricks Runtime（DBR）が最適化されたApache Sparkと追加ライブラリ/統合機能を提供。

  • 04
    Component 4 — Storage

    ストレージ

    すべてのデータおよびAIアセットが永続化される場所。クラウドオブジェクトストレージ（S3、ADLS、GCS）を使用。

// Databricks Runtime（DBR）について

DBRは最適化されたApache Sparkに追加のライブラリと統合機能を組み合わせたランタイム環境だ。Unity Catalogとの連携にはDBR 11.3以上が必要で、最新のLTS（Long-Term Support）バージョンの使用が推奨される。

注意：Databricks SQL Serverlessなど、一部の機能はSparkを直接使用しない。

Unity Catalogの3コンポーネント

UI
UC Component

Unity Catalog UI

Databricks UIに埋め込まれたインターフェース。カタログの閲覧、アクセス権の管理、リネージの可視化、データプロファイリングなどをグラフィカルに操作できる。

→ Databricks UIに統合
SV
UC Component

Unity Catalog Service

コントロールプレーン上で稼働し、ガバナンス機能（アクセス制御、監査、リネージ）を実行するバックエンドサービス。REST APIを通じてプログラマティックなアクセスも提供。

→ コントロールプレーンで稼働
RT
UC Component

Databricks Runtime（DBR）統合

DBRがUnity Catalogと統合され、セキュアなデータアクセスを実現。クエリ実行時にUCサービスと連携し、アクセス権限の確認・監査ログの記録を自動的に行う。

→ DBR 11.3+ 必須、最新LTS推奨
09
Data Sharing & Collaboration

データ共有とコラボレーション

データの価値はその共有によって倍増する。Unity CatalogはDelta Sharingプロトコルを通じて、組織内外でのセキュアなデータ共有を実現する。

// Delta Sharing

Delta Sharingは、セキュアなデータ共有のためのオープンプロトコルだ。オープンソースであり、スタンドアロンでの使用も可能。Databricks上では最小限の設定で利用できるマネージドソリューションとして提供される。

プロトコルはデータプロバイダーとレシピエント（受信者）の間に信頼関係を確立し、Unity Catalogがアクセス制御とガバナンスを適用した上でデータ共有を可能にする。

Delta Sharingの上に構築されたサービス

Databricks Marketplace

データプロダクトの公開・消費のためのオープンフォーラム。組織はデータセット、ノートブック、MLモデルなどを安全に公開・発見・取得できる。外部パートナーとのデータエコシステムを構築。

Databricks Clean Rooms

機密データを共有するためのプライバシー保護環境。複数の組織が互いの生データを公開することなく、共同分析を実行できる。広告・マーケティング分析、医療研究等のユースケースに対応。

データ共有とコラボレーションの詳細は本書の第8章で深く掘り下げられる。

Chapter 1 Summary

まとめ — Ch.1の要点

第1章「The Modern Governance Stack」を通じて明らかになった核心的な洞察を整理する。

  • 01
    ガバナンスはデータの量と多様性の増大に比例して重要になる

    Securitas、Capital Oneのインシデントが示すように、クラウドの設定ミスや人的エラーが大規模なデータ漏洩につながる。82%のセキュリティ侵害に人的要素が関与しているという事実は、技術的対策だけでなくhuman-centricなガバナンスの必要性を示している。

  • 02
    データには明確なライフサイクルがあり、各段階でガバナンスが必要

    作成・取得 → 処理 → 利用 → 退役の4段階それぞれで、アクセス制御、品質管理、コンプライアンス遵守が求められる。PII/PHIの匿名化、エンティティ解決、データ調和はすべて処理段階のガバナンス活動だ。

  • 03
    非構造化データとAIアセットがガバナンスの新たなフロンティア

    組織データの80%以上が非構造化であり、AIの台頭によりMLモデル等のAIアセットもガバナンス対象となった。従来のガバナンスソリューションの多くはこの拡大した範囲に対応できていない。

  • 04
    2つの分離されたスタックは分断されたガバナンスを生む

    データウェアハウスとデータレイクの並行運用はデータサイロを形成し、一貫したガバナンスを不可能にする。Hadoopの「偽りの夜明け」から学んだ教訓は、統合されたアーキテクチャの必要性だ。

  • 05
    レイクハウスパラダイムが統合データ管理の解答

    レイクハウスはデータレイクの柔軟性とデータウェアハウスのACIDトランザクション・パフォーマンスを統合する。メダリオンアーキテクチャ（Bronze → Silver → Gold）により、データ品質が段階的に向上する。

  • 06
    Unity Catalogはレイクハウスの統合ガバナンスレイヤー

    運用的・統合的・オープンなカタログとして、すべてのデータとAIアセットに対して単一のガバナンスモデルを提供する。ID管理、アクセス制御、監査、リネージ、検索・発見の6つのコア機能と、行/列レベルの保護、ABAC、データモニタリングの3つの高度な機能を持つ。

  • 07
    Delta Sharingがセキュアなデータコラボレーションを実現

    オープンプロトコルとしてのDelta Sharingにより、Marketplace（データプロダクトの公開・消費）とClean Rooms（プライバシー保護環境での共同分析）が構築されている。データの価値は共有によって倍増する。

// Ch.1 の本質的なメッセージ

データガバナンスは単なるセキュリティ対策ではなく、組織のデータ資産から価値を引き出す能力そのものを決定する。レイクハウスパラダイムはデータの格納・処理の統合を実現し、Unity Catalogはその上で統合ガバナンスを適用する。

データの量・多様性の増大、AIアセットの出現、規制の拡大——これらすべてに対応するには、統合的なガバナンスが不可欠だ。
