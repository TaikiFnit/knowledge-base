Part III Overview

Part III 概要

Part IIIは、Data Meshの「どう設計するか」を扱う。Part I（4原則）・Part II（なぜ今？）を受けて、本パートでは原則を具体的なアーキテクチャ構造に落とし込む方法を提示する。

Part III のスコープ

第9章「The Logical Architecture」では、4原則それぞれをアーキテクチャコンポーネントに対応させた技術中立な論理設計を解説。第10章「The Multiplane Data Platform Architecture」では、ドメインが自律的にデータプロダクトをビルド・デプロイ・発見・消費できるようにするプラットフォームの3プレーン構造を詳述する。

技術非依存のブループリント。Data Product Quantum、ポート設計、埋め込み型ガバナンスを定義する。

Chapter 10

マルチプレーンデータプラットフォーム

セルフサーブを実現する3層プラットフォームアーキテクチャ。開発者・コンシューマーそれぞれのジャーニーを設計ドライバーとする。

Design Philosophy

「プロダクト→API→実装」の順序

まずデータプロダクトを設計し、次にプラットフォームAPIを考え、実装は最後に決める。過度なツール依存を避けるための原則。

Chapter 9

第9章：論理アーキテクチャ

第9章は、Data Meshの技術非依存な論理アーキテクチャのブループリントを提示する。Dehghaniはあえて特定ツール・実装を規定せず、設計の自由度を保つ。各アーキテクチャコンポーネントはData Meshの4原則のいずれかに対応している。

// 設計哲学

Dehghaniが繰り返し強調するのは、「このアーキテクチャは特定の技術スタックを強制しない」という点。論理設計として読み、実装時にチームが創造性を発揮できる余地を意図的に残している。

02
Ch.9 — The Great Divide

データの大分裂（The Great Divide of Data）

現代の組織データは大きく2つの面に分裂している：オペレーショナルプレーン（トランザクション処理）とアナリティカルプレーン（ウェアハウジング・BI・ML）だ。この2プレーンはテクノロジー・組織・トポロジーすべての面で乖離してきた。

// THE GREAT DIVIDE — データの2プレーン構造

アナリティカルプレーン自体もさらに内部で分裂している。データウェアハウス（BI/レポーティング向け）とデータレイク（データサイエンス向け）という2系統がそれぞれ独自進化し、ETLパイプラインが両者と運用システムをつなぎ続けている。

// ETLパイプラインの脆弱性

組織が成長するにつれて、ETLパイプラインは「継続的に壊れ、複雑さが際限なく増大する」構造となる。Data Meshはこの問題を技術的パッチで解決するのではなく、ドメインベースの逆転トポロジーという根本的なアーキテクチャ変更で対応する。

ドメイン指向の分析データ共有

Data Meshの論理アーキテクチャは、データをテクノロジースタックではなくビジネスドメインを基準に配置する。各ドメインはオペレーショナルAPI（「新しいエピソードを作成する」）とアナリティカルデータエンドポイント（「過去N月のエピソードデータ」）の両方を公開する。

// ドメイン指向アーキテクチャの特徴
双方向
各ドメインが運用データも分析データも両方公開する
自律
運用系・分析系を独立してリリース・デプロイ可能
消費
一つのドメインが別ドメインの分析データを消費できる
03
Ch.9 — Architectural Quantum

Data Product Quantum — アーキテクチャの量子

Data Meshの中核アーキテクチャ概念がData Product Quantum（DPQ）だ。「Quantum」という言葉は進化的アーキテクチャから借用したもので、高い機能凝集性を持ちながら独立してデプロイ可能な最小アーキテクチャ単位を意味する。データプロダクトはメッシュ上の1つのノードであり、その機能に必要なすべての構造要素を内包する。

"The data product is the architectural quantum of the data mesh — the smallest unit of architecture that can be independently deployed with high functional cohesion." — Zhamak Dehghani, Data Mesh (O'Reilly, 2022)

3つの構造コンポーネント

// DATA PRODUCT QUANTUM — 3つの構造コンポーネント
1
Component 01

コード（Code）

データパイプラインのコード（上流データの消費・変換・サービング）、APIのコード（データアクセス・スキーマ・可観測性メトリクス）、そしてトレイト強制のコード（アクセス制御・コンプライアンス・プロベナンス）の3つから構成される。

→ 4原則の「ドメインオーナーシップ」と「データプロダクト」を実装
2
Component 02

データ・メタデータ

分析データはポリグロット形式（イベント・バッチファイル・リレーショナル・グラフ — 同一セマンティクス）で格納。メタデータには計算ドキュメント、セマンティック/構文宣言、品質メトリクスが含まれる。

→ 4原則の「セルフサーブ」と「連合ガバナンス」を実装
3
Component 03

インフラストラクチャ

コードのビルド・デプロイ・実行を可能にする基盤。ビッグデータとメタデータへのストレージとアクセスを提供する。他のデータプロダクトへの影響なしに独立セグメント（論理的分離）で動作する。

→ 4原則の「セルフサーブプラットフォーム」を実装
// 過去のパラダイムとの根本的な違い

過去： パイプラインはデータから独立して管理。インフラ（ウェアハウス・レイクのストレージ）は共有。

Data Mesh： データプロダクト = コード ＋ データ ＋ インフラ の合成物。ドメインの境界コンテキストの粒度で一体化される。

04
Ch.9 — Ports & Interfaces

データプロダクトのポート

データプロダクトは外部と通信するための標準化されたインターフェース「ポート」を持つ。ポートは4種類あり、それぞれ明確な役割を持つ。

INPUT DATA PORT OUTPUT DATA PORT CONTROL PORT DISCOVERY PORT
// DATA PRODUCT QUANTUM — ポート構造とメッシュ接続
①
Input Data Port

入力データポート

上流データ（他のデータプロダクトまたはドメインの運用システム）を受信・プル取得するためのAPI/メカニズムのセット。

データフローの入口 → リネージを形成
②
Output Data Port

出力データポート

分析データとメタデータを提供するインターフェース。読み取り専用・時間軸変異型・イミュータブル。SQL・Blob・イベント・グラフ・カラムナーファイルのポリグロット対応。読み取り関数は常に時間の関数（from-toウィンドウ）として定義される。

マルチモーダル: BI用SQL / データサイエンス用カラムナー / アプリ用Pub/Sub
③
Control Port

コントロールポート

ガバナンスポリシーを設定するための特権APIセット。ポリシーは集中的にプッシュされるか、ローカルで作成される。受信・変更・実行はすべてローカルで行われる。

→ 連合ガバナンスモデルの実行点
④
Discovery Port

ディスカバリーポート

出力ポート情報・SLO・ドキュメント・データアクセス・認可情報を提供するルートAPI（アグリゲート）。コンシューマーがデータプロダクトを発見・理解するための入口となる。

→ メッシュカタログとの接続点

メッシュの3層相互接続

データプロダクト間の接続は3つの層から成る：

  1. データフロー層（Data Flow Layer）：入力ポート⇔出力ポート間のデータの流れ。これがリネージになる。
  2. スキーマセマンティックリンク（Schema Semantic Links）：標準スキーマ言語によりプロダクト横断でのJOINを可能にする。
  3. データリンク（Data Links）：実際のデータ関係（例：「このリスナーはこのユーザーと同一人物である」）。
05
Ch.9 — Federated Governance

埋め込み型コンピュテーショナルポリシー

第9章はガバナンスのアーキテクチャも扱う。Data Meshのガバナンスモデルは連合型コンピュテーショナルガバナンスと呼ばれ、中央集権と分散のバランスを取る。

連合ガバナンスモデル

// 連合ガバナンスの核心

意思決定はドメインデータプロダクトオーナー ＋ データプラットフォームプロダクトオーナーの連合体が行う。自律性・ドメインローカルの意思決定と、健全で相互運用可能なエコシステムのためのグローバルルールのバランスをとる。

グローバル決定の目的：相互運用性の実現 ＋ 発見と組み合わせを通じた複合的ネットワーク効果の創出。

サイドカーパターンによるポリシー実行

ポリシーはサービスメッシュのサイドカーと類比して、各データプロダクトにサイドカーまたはライブラリとして埋め込まれる。アクセス制御・匿名化・暗号化・差分プライバシー・合成データなどの機能をプログラマティックに構築し、プラットフォームに自動化する。コントロールポートを通じてポリシーを設定する。

Policy Type

アクセス制御

誰がどのデータにアクセスできるかをデータプロダクト自身がローカルに強制する。

Policy Type

匿名化・暗号化

個人情報保護のため、データプロダクトが自律的に匿名化・暗号化処理を適用する。

Policy Type

合成データ・差分プライバシー

センシティブデータへのアクセスを合成データや差分プライバシーで代替する。

ガバナンス対比表：従来 vs Data Mesh

観点 従来のデータガバナンス Data Mesh ガバナンス

Chapter 10

第10章：マルチプレーンデータプラットフォーム

第10章はData Meshのセルフサーブデータプラットフォームアーキテクチャを詳細に解説する。ドメインが自律的にデータプロダクトをビルド・デプロイ・発見・消費できるインフラが本章のテーマだ。

なぜマルチプレーン？

// 「プレーン」の概念

「プレーン」とはネットワーキングのコントロールプレーン・データプレーンのように、統合されているが独立した存在の層を意味する。階層的なアクセスモデルではなく、異なるユーザープロファイル/ペルソナにそれぞれ対応する別個の存在として設計する。

マルチプレーンモデルの目的は、高レベルの抽象化によって複雑さとフリクションを除去することにある。各プレーンは異なるユーザーペルソナに対して、それぞれ適切な抽象レベルでインタラクションを提供する。

07
Ch.10 — Three Planes

3つのプレーン

// MULTIPLANE DATA PLATFORM — 3プレーン構造
1
Plane 01 — Utility Plane

インフラプロビジョニングプレーン（ユーティリティプレーン）

最も低レベルの層。生インフラのライフサイクル管理を担う。分散ファイルストレージ、ストレージアカウント、アクセス制御管理、データプロダクトの内部コードのオーケストレーション、データプロダクトのグラフ上の分散クエリエンジンをプロビジョニングする。

インフラアズコード（IaC）のAPIを提供し、スペック/API/コマンドでプロビジョニングする。他のプレーンや高度な開発者が直接使用する。

Spark Jobs Kafka Topics Storage Accounts RBAC Query Engine Orchestration
2
Plane 02 — Developer Experience

データプロダクト開発者体験プレーン

典型的なデータプロダクト開発者のためのメインインターフェース。データプロダクトワークフローの複雑さを抽象化する。プロビジョニングプレーンより高い抽象レベルで、ライフサイクル管理のためのシンプルな宣言的インターフェースを提供する。

開発者はスペックファイル（入力/出力・ポリシー・モデルを宣言的に記述）を提供するだけでよい。リポジトリにチェックイン → API/コマンドに渡す → プラットフォームがテクノロジーを自動的に連結する。

Declarative Spec CI/CD Pipeline Auto Standards Full Lifecycle OAM Support
3
Plane 03 — Mesh Experience

メッシュ監督プレーン（メッシュ体験プレーン）

メッシュレベルで提供されるグローバルな能力。接続されたデータプロダクトのグラフを扱う。検索・ブラウズ（ユースケースに合ったデータプロダクトを発見）、セマンティッククエリ（メッシュを横断した複数データプロダクトの相関）、ナレッジグラフ（セマンティックリンク付きのスキーマシステム）を提供する。

集中化を避けながら、ローカル定義のスキーマからグローバル標準モデルとして出現する。主にデータコンシューマー向けのインターフェース。

Search & Browse Semantic Query Knowledge Graph Cross-product Join

ThoughtWorksの実装例

// OAM (Open Application Model) による実装

スペックファイル： data_product_spec.yml + data_product_deploy.yml としてリポジトリにスキャフォールド。

Bootstrap： ドメインごとにSnowflakeウェアハウスをプロビジョニング、データプロダクトごとにスキーマ作成、モニタリング名前空間・カタログコミュニティ・RBACロール（Consumer/Service/Owner）を設定。

Deploy： CI/CDパイプラインがスペックを検証し、パイプラインをデプロイ、SLOモニターを作成し、カタログに公開、ポリシーを適用する。

Retire： 利用状況メトリクスを追跡し、非アクティブなプロダクトを削除する。

コンシューマー vs 開発者の視点

Dehghaniはプラットフォーム設計をユーザージャーニー駆動で行うことを推奨する。2つの主要ペルソナが存在する：

Consumer Persona

データコンシューマー

データを発見・利用する人。メッシュ体験プレーンが主たるインターフェース。

ジャーニー：発見 → 評価 → アクセス申請 → 消費 → フィードバック提供

Developer Persona

データプロダクト開発者

データプロダクトをビルド・維持する人。開発者体験プレーンが主たるインターフェース。

ジャーニー：Incept → Explore → Bootstrap → ... → Retire（全11フェーズ）

// 設計の優先順位

ブッククラブ参加者の議論では「コンシューマーから設計を始めるべき」という意見が強調された。コンシューマーが何を必要とするかが、プラットフォームの要件を逆算的に決定する。

Summary

まとめ — Part IIIの要点

Part IIIはData Meshを「どう作るか」の具体的な設計指針を提供する。以下に第9章・第10章の要点を整理する。

第9章の要点

A
Key Concept

Data Product Quantum

データプロダクトは「コード + データ/メタデータ + インフラ」の3要素をドメイン境界で一体化したアーキテクチャ量子。過去のパラダイムとの最大の違いはこの一体化にある。

B
Key Concept

4種類のポート設計

Input（受信）・Output（サービング）・Control（ガバナンス設定）・Discovery（発見）の4ポートが、データプロダクトの外部インターフェース全体を定義する。

C
Key Concept

サイドカーガバナンス

ポリシーはサイドカーとしてデータプロダクトに埋め込まれ、ローカルに実行される。「防止」ではなく「自動検知・回復」へのパラダイムシフト。

第10章の要点

D
Key Concept

3プレーン構造

インフラプロビジョニング（Plane 1）・開発者体験（Plane 2）・メッシュ監督（Plane 3）の3プレーンが、それぞれのペルソナに適した抽象レベルを提供する。

E
Key Concept

宣言的スペックファイル

開発者はYAMLスペックファイルを書くだけ。プラットフォームが残りを自動化する。「原始的なスクリプトから始め、洗練させていく」反復的アプローチ。

F
Key Concept

設計の優先順位

①データプロダクトを設計する → ②プラットフォームAPIを設計する → ③実装を決める。この順序がDehghaniの推奨する設計アプローチ。

// Part III 全体の核心メッセージ

Data Meshのアーキテクチャは特定ツールを強制しない。論理設計として原則を理解し、チームの文脈で創造的に実装することが求められる。DPQというアーキテクチャ量子の概念と、3プレーンのプラットフォーム設計が、4原則を現実のシステムに翻訳するための中心的な道具立てとなっている。
