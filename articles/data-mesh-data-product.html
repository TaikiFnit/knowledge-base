Part IV Overview

Part IV 概要

Part IVは、Data Meshの「データプロダクトをどう設計するか」という実装レベルの問いに答える。Part III（アーキテクチャ論理設計・マルチプレーンプラットフォーム）を受けて、本パートでは個々のデータプロダクトが持つべき機能（アフォーダンス）と、その設計プロパティを章ごとに詳述する。

Part IV のスコープ

第11章「アフォーダンスによる設計」では、データプロダクトが提供すべき能力を列挙し、複雑適応系の哲学を導入する。第12〜14章では、各アフォーダンスの具体的な設計プロパティ（Serve・Consume・Transform・Discover・Understand・Compose・Manage・Govern・Observe）を掘り下げる。

Chapter 11

アフォーダンスによる設計

データプロダクトが提供すべき能力（アフォーダンス）を定義する。複雑適応系からの設計インサイトを通じ、シンプルなローカルルールがメッシュ全体の複雑な振る舞いを生み出す設計哲学を説く。

Chapter 12

取得・変換・提供の設計

データをどう受け取り（Consume）、どう変換し（Transform）、どう提供するか（Serve）の設計プロパティ。ポリグロットアクセス、バイテンポラル、MLとしての変換などを扱う。

Chapter 13

発見・理解・合成の設計

自己登録によるDiscovery、グローバルURI、セマンティックモデル、SLOによる信頼構築、そしてポリセームとスキーマリンクによるCompose設計。

Chapter 14

管理・ガバナンス・監視の設計

ライフサイクル管理とマニフェスト、ポリシーコードとしてのガバナンス、そしてメトリクス・ログ・トレースという可観測性の3つの柱。

01
Chapter 11

第11章：アフォーダンスによるデータプロダクト設計

第11章では、データプロダクトを設計する際の出発点として「アフォーダンス（affordance）」という概念を導入する。アフォーダンスとは、HCI（ヒューマン・コンピュータ・インタラクション）の理論家Don Normanが提唱した概念で、「あるオブジェクトがユーザーに対して提供する行為の可能性・能力」を指す。

"Design a data product by what it enables its users to do — its affordances." — Zhamak Dehghani, Data Mesh (O'Reilly, 2022)

データプロダクトのアフォーダンス設計とは、データプロダクトが利用者に対して何を可能にするかから出発する設計アプローチだ。実装技術ではなく、ユーザーの行為能力を最初に定義する。

02
Ch.11 — Affordances

データプロダクトのアフォーダンス

Dehghaniは、データプロダクトが持つべきアフォーダンスを8つに分類する。これらは「8つのユーザビリティ属性」とも対応し、各アフォーダンスが後の章での設計の章立てとなる。

// DATA PRODUCT AFFORDANCES — 8つのアフォーダンス構造
1
Affordance 01

Serve Data — データの提供

複数モードでデータコンシューマーに分析データを提供する能力。SQL、API、ファイル、イベントストリームなど複数のアクセス形式（ポリグロット）をサポートする。

→ 第12章で詳述
2
Affordance 02

Consume Data — データの取得

上流ソース（運用システム、他のデータプロダクト、外部ソース）からデータを受け取る能力。入力ポートを通じてプッシュまたはプル方式で取得する。

→ 第12章で詳述
3
Affordance 03

Transform Data — データの変換

データを処理・クレンジング・集計・整形する能力。プログラマティック（Python/Spark）または宣言的（dbt）なアプローチで実装される内部ロジック。

→ 第12章で詳述
4
Affordance 04

Discover, Understand, Trust

データプロダクトを発見可能にし、セマンティック・構文モデルで理解を促し、SLOと透明性でコンシューマーの信頼を獲得する能力。

→ 第13章で詳述
5
Affordance 05

Compose — データの合成

他のデータプロダクトとデータを組み合わせ・結合できる能力。ポリセーム（標準識別子）とスキーマリンクにより、分散した状態でも相互運用性を確保する。

→ 第13章で詳述
6
Affordance 06

Manage Lifecycle — ライフサイクル管理

ブートストラップ・ビルド・デプロイ・実行・保守・進化・リタイアというフルライフサイクルを自律的に管理する能力。マニフェストにより宣言的に定義する。

→ 第14章で詳述
7
Affordance 07

Govern — ガバナンス

アクセス制御・暗号化・データ分類・保持ポリシー・プライバシー（GDPR/CCPA）などのポリシーをデータプロダクト自身が埋め込み、ローカルで強制する能力。

→ 第14章で詳述
8
Affordance 08

Observe — 可観測性

メトリクス・ログ・トレースという3つの観点からデータプロダクトの動作を監視・デバッグ・監査する能力。SLIによるSLO達成度の継続的な可視化を含む。

→ 第14章で詳述
03
Ch.11 — Design Philosophy

複雑適応系から学ぶ設計哲学

第11章のもう一つの重要な貢献は、複雑適応系（Complex Adaptive Systems / CAS）の原理をData Meshの設計哲学に取り込んだ点だ。CASとは、鳥の群れや生態系のように、個々のエージェントがシンプルなローカルルールに従うだけで、全体として複雑で適応的な振る舞いが創発するシステムを指す。

// 複雑適応系とData Meshの対応

CASの原理： シンプルなローカルルール → 複雑な創発的振る舞い → 中央オーケストレーター不要

Data Meshへの適用： 各データプロダクトが同じアフォーダンスセット（ローカルルール）に従うだけで、メッシュ全体の価値（グローバルな振る舞い）が創発する。誰も「全体を管理」しなくてよい。

// CAS設計原理のData Meshへの適用
自律
各データプロダクトは自律的に動作する独立ユニット
創発
シンプルなルール（アフォーダンス）から複雑な価値が生まれる
適応
中央制御なしに変化する環境へ個別に適応できる

設計の実践的含意

  • シンプルさを優先： ドメインチーム（データ専門家でない汎用エンジニア）でも構築・維持できる設計シンプルさを維持する
  • プラットフォームが複雑さを吸収： 各データプロダクトが複雑な実装を担うのではなく、プラットフォームが共通の複雑さを抽象化する
  • 中央オーケストレーターの排除： データプロダクトは自律的に動作し、プラットフォームはツールを提供するが中央制御はしない
04
Chapter 12

第12章：データの取得・変換・提供の設計

第12章は最もボリュームのある章で（p.201–231）、データプロダクトの中核的な機能である「Serve（提供）」「Consume（取得）」「Transform（変換）」の3つについて、具体的な設計プロパティと実装アプローチを詳述する。

// 第12章の構成

Dehghaniはあえて「Serve → Consume → Transform」の順で議論する。まず出力（何を提供するか）を定義し、次に入力（何を受け取るか）を決め、最後に変換（どう処理するか）を設計する。この「出力優先」の思考順序は、ユーザー価値からの逆算設計を体現している。

05
Ch.12 — Serve Data

Serve Data — ポリグロットアクセス設計

ユーザーペルソナ別のアクセスニーズ

データプロダクトを「使う」ペルソナはひとりではない。異なるペルソナが異なる形式でデータを必要とするため、データプロダクトは単一の形式を強要するのではなく、同一のセマンティックデータを複数の形式で提供する必要がある。

ペルソナ 必要なアクセス形式 ユースケース
データアナリスト SQLクエリ、BIツール、スプレッドシート ダッシュボード作成、アドホック分析
データサイエンティスト カラムナーファイル（Parquet）、Notebookアクセス 探索的分析、特徴量エンジニアリング
MLエンジニア Feature Store、ストリーミング、トレーニングデータセット モデルトレーニング、オンライン推論
アプリ開発者 REST API、イベントストリーム、Pub/Sub アプリケーション統合、リアルタイム処理
ビジネスユーザー レポート、ダッシュボード、集計ビュー KPI監視、意思決定サポート
// SERVE DATA — ポリグロットアクセス設計

Serve Data の5つの設計プロパティ

1
Property 01

ポリグロットデータアクセス

同一セマンティックデータを複数形式（SQL、ファイル、イベント、API）で提供する。異なるペルソナが異なるインターフェースで同じデータにアクセスできる。

→ 8属性の「ネイティブアクセス可能（Natively Accessible）」を実現
2
Property 02

時間変異（Time-Variant）

データは常に時間の関数として提供される。すべての読み取り関数はf(from_time, to_time)として定義され、任意の時点のデータにアクセスできる。

→ 正確な履歴分析・イベント時間処理を可能にする
3
Property 03

イミュータブル（Immutable）

スナップショットは遡及的に変更されない。データはイミュータブルに保持され、過去に遡った修正は新しいバージョンとして追記される。カスケードエラーを防止する。

→ 分析結果の再現性と信頼性を確保
4
Property 04

読み取り専用（Read-Only）

出力ポートは読み取り専用インターフェース。コンシューマーはデータを読むことはできるが、直接書き込むことはできない。データプロダクトの整合性を保護する。

→ データ品質の一元管理とセキュリティの強化
5
Property 05

バイテンポラル（Bitemporal）

ビジネス時間（イベントが実際に発生した時間）と処理時間（データが処理・記録された時間）の両方を軸にデータをクエリできる。遅延到着データの正確な処理に不可欠。

→ 「いつ発生したか」と「いつ記録されたか」を区別した正確な分析
06
Ch.12 — Consume Data

Consume Data — 3つのソースアーキタイプ

データプロダクトが受け取るデータのソースには3つのアーキタイプがある。各アーキタイプは結合度（カップリング）の強度が異なり、適切な入力ポート設計を要する。

①
Archetype 01

同ドメインの運用システム

ドメイン内の自分自身の運用データベース・サービス。最も密結合で、しばしば同期的。同一チームが管理するため、スキーマ変更への対応が比較的容易。

→ ローカル消費（Local Consumption）
②
Archetype 02

上流データプロダクト

他のドメインが提供するデータプロダクトの出力ポートから消費。疎結合で、標準インターフェース（SQL・API・イベント）を介してリモートで取得する。

→ リモート消費（Remote Consumption）
③
Archetype 03

外部ソース

サードパーティデータフィード、パートナーデータ、公開データセット。最も疎結合で、フォーマット変換・スキーママッピングが必要になることが多い。

→ 外部データの内部化とスキーマ正規化が重要

Push vs. Pull とストリーミング vs. バッチ

// 入力ポートの設計決定

プッシュ vs. プル： 上流が変化をプッシュ（イベントドリブン）するか、データプロダクトが定期的にプル（ポーリング）するかはソースのアーキタイプと変更頻度で決まる。

ストリーミング vs. バッチ： リアルタイム性が必要な場合はストリーミング（Kafka、Pub/Sub）、定期的な大量処理はバッチ（Spark、dbt）。混合アーキテクチャも一般的。

07
Ch.12 — Transform Data

Transform Data — データ変換設計

データ変換は、データプロダクトの内部実装詳細だ。コンシューマーは「どう変換されているか」を知る必要はなく、クリーンな出力ポートだけを見る。変換ロジックはデータプロダクトの境界内に完全にカプセル化される。

プログラマティック vs. 宣言的（Nonprogrammatic）変換

アプローチ 特徴 例 適したユースケース
プログラマティック コードベース。柔軟性が高い Python、Spark、Flink 複雑な変換、ML前処理、カスタムロジック
宣言的（Nonprogrammatic） SQL・設定ベース。読みやすく保守しやすい dbt、SQLビュー、ストアドプロシージャ 集計、結合、定型的なデータ整形

MLとしての変換（ML as Transformation）

機械学習モデルはデータプロダクト内の変換の一種として扱える。入力ポートからトレーニングデータを受け取り、出力ポートから推論・予測・特徴量を提供するという構造は、通常のデータ変換と同じパターンに当てはまる。

// MLモデルの2つの統合パターン

マイクロサービスとしてのML： モデルはオンライン推論のためのAPIサービス。出力ポートはRESTエンドポイント。リアルタイム予測に適する。

トランスフォーマーとしてのML： モデルはバッチ変換パイプラインの一部。大量のデータに対してオフラインで推論し、特徴量テーブルを出力。Feature Store等を介してデータプロダクトとして公開する。

DAGベースの変換パイプライン

変換はDAG（有向非巡回グラフ）ベースのパイプラインとして実装される。これはデータプロダクト内部のパイプラインであり、プロダクトをまたぐクロスパイプラインとは区別される。内部パイプラインは外部に露出せず、インターフェースのみが公開される。

Chapter 13

第13章：データの発見・理解・合成の設計

第13章は（p.233–252）、コンシューマーがデータプロダクトを「見つけ、理解し、信頼し、活用する」ためのアフォーダンス設計を扱う。データがいかに良質でも、発見できなければ価値はゼロに等しい。

自己登録による発見（Self-Registration）

各データプロダクトはデプロイ時に自動的にメッシュのディスカバリー層（カタログ・レジストリ）に自身を自己登録する。手動登録ではなく、デプロイパイプラインの一部として自動化される。

// 自己登録の情報コンポーネント

登録情報には以下が含まれる：データプロダクト名・所属ドメイン・オーナー・説明・出力ポート情報・SLO定義・ドキュメントURL・タグ・バージョン情報。これらはメタデータとして機械可読形式で公開される。

グローバルURI

すべてのデータプロダクトはグローバルに一意で永続的なURI（Uniform Resource Identifier）を持つ。このURIがデータプロダクトに関するすべての情報へのエントリーポイントとなる。内部実装が変わっても、URIは安定していなければならない。

// グローバルURIの要件
一意
組織全体でグローバルに一意なアドレス
永続
実装変更後も安定して変わらない
網羅
そこから全情報（ポート・SLO・ドキュメント）へアクセスできる

セマンティック・シンタックスモデル

コンシューマーがデータを理解するには2種類のモデルが必要だ：

  • セマンティックモデル（Semantic Model）：「このデータが何を意味するか」。エンティティ、関係、ビジネスコンセプトの定義。例：「customer_id は顧客の一意識別子で、CRMシステムと一致する」
  • シンタックスモデル（Syntax Model）：「このデータがどう構造化されているか」。スキーマ、データ型、フォーマット定義。標準スキーマ言語（JSON Schema、Avro、Protobuf）で記述し、機械可読にする。

SLOによる信頼構築（Establish Trust with Data Guarantees）

信頼はドキュメントでなく数値化された約束（SLO）で構築される。SLOとSLIの透明な公開により、コンシューマーはデータプロダクトの品質を客観的に評価できる。

SLOの種別 内容 例
変更間隔（Freshness） データの鮮度・更新頻度 「毎時更新、最大1時間の遅延」
完全性（Completeness） 欠損値・レコード欠落の保証 「全トランザクションの99.9%を含む」
統計的形状（Statistical Shape） データ分布・カーディナリティの特性 「アクティブユーザー数は10万〜500万の範囲」
リネージ・プロベナンス データの出所・加工履歴 「OrderServiceのDB →集計→ このプロダクト」
精度・正確性（Precision） 値の精度と正確性の保証 「金額は小数点2桁まで正確」
運用品質（Availability） 可用性・レイテンシの保証 「99.5% SLA、クエリ応答5秒以内」
09
Ch.13 — Compose Data

データの合成 — ポリセームとスキーマリンク

コンポーザビリティ（Composability）とは、複数のデータプロダクトのデータを組み合わせられる能力だ。従来の中央集権的アプローチ（正規データモデル、スターデータウェアハウス）は柔軟性・スケーラビリティに限界があった。Data Meshはフェデレーション型の標準ベースのアプローチで解決する。

従来アプローチの問題点

// 中央集権的コンポーザビリティの限界

カノニカルデータモデル（Traditional）： 単一の正規化スキーマを強制 → ボトルネック化、変更が遅い、各ドメインの自律性を損なう

データウェアハウス スタースキーマ： 中央管理で整合性は高いが、ドメイン指向ではなく、リアルタイム性に欠ける

ポリセーム（Polysemes）— グローバル標準識別子

ポリセームとは、複数のデータプロダクトをまたいで共通に使われるビジネスキー（識別子）だ。組織全体で標準化されることで、JOINが可能になる。

Polyseme例 01

customer_id

全ドメインで統一された顧客識別子。Salesforce CRMの顧客IDと一致し、注文ドメイン・マーケティングドメイン・カスタマーサポートドメインが同一キーで参照できる。

Polyseme例 02

product_id

全ドメインで統一された製品識別子。在庫ドメイン・販売ドメイン・物流ドメインが同一キーで参照。SKUや内部コードとのマッピングを一元管理する。

Polyseme例 03

order_id

注文の一意識別子。決済ドメイン・在庫ドメイン・配送ドメインが同一キーを参照することで、注文の全ライフサイクルをまたいだ分析が可能になる。

スキーマリンク（Schema Linking）

各データプロダクトは、自身のスキーマ内で他のデータプロダクトとのセマンティック接続（スキーマリンク）を宣言する。これにより、コンシューマーはどのプロダクトが結合可能かを机上で発見できる。メッシュ経験プレーン（Mesh Experience Plane）は複数プロダクトをまたぐクエリを実行する。

// コンポーザビリティは標準から、中央制御からではない

Data Meshのコンポーザビリティは、グローバルに標準化されたポリセームとスキーマリンクという規約から自然に創発する。中央の統合レイヤーがデータを統合するのではなく、各データプロダクトが自ら他プロダクトとの接続可能性を宣言する。

10
Chapter 14

第14章：データの管理・ガバナンス・監視の設計

第14章（p.255–267）は、データプロダクトの持続的な運用を支える3つの側面：ライフサイクル管理（Manage）・ガバナンス（Govern）・可観測性（Observe）の設計を扱う。

ライフサイクル管理とマニフェスト

データプロダクトは誕生から廃止まで明確なライフサイクルを持つ。このライフサイクルはマニフェスト（Manifest）と呼ばれる宣言的な設定ファイルで定義され、プラットフォームが自動的に各フェーズをサポートする。

ライフサイクルフェーズ 内容
Bootstrap（ブートストラップ） プロジェクト初期化、テンプレート適用、インフラ初期設定
Build（ビルド） コード・設定のビルド、テスト、コンテナ化
Deploy（デプロイ） インフラプロビジョニング、カタログへの自己登録、サービス起動
Run（実行） 定常運用、データ更新、SLI計測、ポリシー適用
Evolve（進化） スキーマ更新（後方互換）、ロジック改善、バージョン管理
Retire（廃止） コンシューマーへの通知、段階的廃止、アーカイブ

データプロダクト マニフェストの構成要素

// Manifest — データプロダクトの宣言的定義

マニフェストはデータプロダクトのすべての設定を宣言的に記述する仕様ファイルであり、コードとして管理（バージョン管理・テスト・デプロイ）される。含む情報：入力ポート定義・出力ポート定義（スキーマ・アクセス方式）・変換ロジック参照・ポリシー設定・SLO定義・インフラ要件・メタデータ（オーナー・ドメイン・説明・タグ）・バージョン情報。

ガバナンス設計 — ポリシーはコードとして

Data Meshのガバナンスは「外部から課せられるルール」ではなく、データプロダクト自身に埋め込まれたコードだ。ポリシーはデータプロダクトの境界でローカルに強制される。

A
Policy Type A

アクセス制御（Access Control）

RBAC（ロールベース）またはABAC（属性ベース）でデータへのアクセスを制御。「誰がどのデータを読めるか」をローカルに定義・強制する。

→ コントロールポートでポリシーを受信・適用
B
Policy Type B

暗号化・データ分類

保存時・転送時の暗号化。データの機密レベル分類（PII・機密・公開）を定義し、分類に応じたアクセス制限と匿名化処理を自動適用する。

→ 機密レベルに応じた自動処理の標準化
C
Policy Type C

プライバシー（GDPR/CCPA）

個人データの取り扱いに関するコンプライアンス。匿名化・仮名化・差分プライバシー・合成データ生成などをプログラマティックに実装する。

→ ポリシーエンジンによる動的評価・適用
D
Policy Type D

保持・品質ポリシー

データの保持期間（いつ削除するか）と最低品質基準の定義。品質閾値を下回ったデータは出力ポートから自動的に除外される。

→ サイドカーパターンで各データプロダクトに埋め込まれる

可観測性の3つの柱

各データプロダクトはデフォルトで可観測性を持つように設計される。マイクロサービスの可観測性理論に倣い、Data Meshはデータプロダクトに3つの観測軸を要求する。

// OBSERVABILITY — 可観測性の3つの柱
Observability 01

デバッグ（Debug）

データ問題を変換パイプライン全体を通じてトレースできる能力。「どのステップで値が変わったか」を特定できる。

Observability 02

監査（Audit）

誰がいつどのデータにアクセスし、何をしたかを記録・照会できる。コンプライアンス要件（GDPR・SOX等）への対応に不可欠。

Observability 03

アラーティング（Alerting）

SLIがSLOから逸脱した際に自動アラートを発報。データ品質の劣化を早期に検知し、コンシューマーへの影響を最小化する。

11
Ch.3 / Part IV 統合

8つのユーザビリティ属性まとめ

Part IVで設計されたアフォーダンスは、書籍第3章で定義された8つのベースラインユーザビリティ属性と直接対応する。すべてのデータプロダクトがこれら8属性を満たすことが、メッシュのネットワーク効果の前提条件となる。

// 8 USABILITY ATTRIBUTES — データプロダクトの必須属性
ATTR 01 / DISCOVERABLE

発見可能（Discoverable）

カタログ・検索で容易に見つけられる。デプロイ時の自己登録によって実現。

ATTR 02 / ADDRESSABLE

アドレス可能（Addressable）

グローバルに一意で永続的なURIを持つ。全ての情報へのエントリーポイント。

ATTR 03 / UNDERSTANDABLE

理解可能（Understandable）

自己記述的なセマンティック・構文モデル、ドキュメント、サンプルデータを提供。

ATTR 04 / TRUSTWORTHY

信頼性・正確性（Trustworthy & Truthful）

SLOによる数値化された約束と、SLIによる透明な達成度の継続的公開。

ATTR 05 / NATIVELY ACCESSIBLE

ネイティブアクセス可能（Natively Accessible）

ペルソナに合わせたポリグロットアクセス。SQL・API・ファイル・イベント等の複数形式。

ATTR 06 / INTEROPERABLE

相互運用・合成可能（Interoperable & Composable）

標準ポリセームとスキーマリンクにより、他のデータプロダクトと容易に結合できる。

ATTR 07 / VALUABLE ON ITS OWN

単独での価値（Valuable on Its Own）

結合なしで単独でビジネス価値を提供できる凝集したドメインコンセプト。

ATTR 08 / SECURE

セキュア（Secure）

アクセス制御・暗号化・プライバシーポリシーをデータプロダクト自身がローカルに強制。

Part IV Summary

まとめ — Part IVの要点

Part IV（第11〜14章）は、個々のデータプロダクトをどう設計するかという実装設計のハンドブックだ。以下の7つのテーマが中心を成す。

1
Key Theme 01

アフォーダンス駆動設計

「何ができるか（アフォーダンス）」からデータプロダクトを設計する。実装技術ではなくユーザーの行為可能性を起点に置くことで、ドメインチームでも構築・維持できる設計が実現する。

→ 第11章のコア
2
Key Theme 02

ポリグロットサービング

同一セマンティックデータを複数形式（SQL・API・ファイル・イベント）で提供する。アナリスト・データサイエンティスト・ML・アプリ開発者・ビジネスユーザーがそれぞれ適した形式でアクセスできる。

→ 第12章 Serve Data
3
Key Theme 03

自己登録と発見

デプロイ時に自動的にメッシュカタログに自己登録することで、グローバルURIとメタデータが常に最新状態に保たれる。手動カタログ更新の廃止が運用コストを削減する。

→ 第13章 Discover
4
Key Theme 04

標準ベースのコンポーザビリティ

ポリセーム（標準識別子）とスキーマリンクによって、中央制御なしに分散したデータプロダクト間の結合を可能にする。中央データモデルの代わりに、標準という約束でメッシュをつなぐ。

→ 第13章 Compose
5
Key Theme 05

ガバナンスはコード

ポリシーはデータプロダクトに埋め込まれたコードとして実装される。バージョン管理・テスト・デプロイされるコードとして扱うことで、ポリシーの一貫した適用と自動化が可能になる。

→ 第14章 Govern
6
Key Theme 06

フルライフサイクル管理

マニフェストによる宣言的定義で、ブートストラップからリタイアまでのフルライフサイクルを自律的に管理できる。プラットフォームが各フェーズを自動サポートする。

→ 第14章 Manage
7
Key Theme 07

デフォルトの可観測性

メトリクス・ログ・トレースの3つの柱を標準実装することで、すべてのデータプロダクトがデフォルトで監視・デバッグ・監査可能になる。SLIによるSLO達成度の継続的な可視化が信頼を構築する。

→ 第14章 Observe
// Part IV の本質的なメッセージ

Part IVは単なる技術仕様集ではない。「データプロダクトを設計するとは、使う人の可能性（アフォーダンス）を設計することだ」というDehghaniのビジョンが貫かれている。CASの原理に倣い、シンプルで一貫したアフォーダンスセットが各データプロダクトに実装されることで、Data Meshというシステム全体の複雑な価値が創発する。技術選択（Spark？dbt？Kafka？）はアフォーダンスの後で決まる、実装の詳細に過ぎない。
