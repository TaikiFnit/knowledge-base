01

前提知識 — Rubyのオブジェクト生成

Rubyでオブジェクトを生成する際、一見シンプルに見える MyClass.new(args) という呼び出しの背後には、実は複雑なプロセスが隠されています。このセクションでは、そのプロセスを詳しく解説します。

オブジェクト生成の2ステップ構造

Rubyでは、MyClass.new(args) は実際には以下の2つのステップから成り立っています：

  • allocate フェーズ：メモリ領域を確保し、オブジェクトの骨格を作成
  • initialize フェーズ：確保されたオブジェクトに初期化処理を実施

この2段階のプロセスは、Rubyの柔軟性を支える重要な設計です。ユーザーは allocate をオーバーライドしてカスタムメモリ配置を実装したり、initialize で複雑な初期化ロジックを実装したりできます。

CRuby実装の現状

CRubyのClass#newはC言語で実装されており、具体的には rb_class_new_instance という関数が担当しています。このC実装が、今回の最適化の対象となります。

Railsエコシステムへの影響

Rails で Model.new するとき、その裏側では想像以上に多数のクラスが new されています：

  • ActiveRecord::Base のサブクラス
  • Association プロキシオブジェクト
  • Query オブジェクト
  • バリデーション結果オブジェクト

このため、Class#new の小さな改善でもシステム全体に波及する可能性があります。わずか数ナノセコンドの改善が、アプリケーション全体のレイテンシーを大きく削減できるのです。

重要な前提 この最適化の根本的な背景にあるのは、Ruby → C → Ruby という往復の関数呼び出しがボトルネックであることです。言語境界を越えるたびに、引数の変換、型チェック、例外処理などのオーバーヘッドが発生します。
02

C実装の問題点

CRubyの Class#new 実装がなぜ遅いのか、その具体的な問題点を掘り下げます。

実装の流れと各段階のオーバーヘッド

現在の実装における処理フロー：

# MyClass.new(name: "Ruby", version: 3.5) の内部動作

# 1. Ruby → C: rb_class_new_instance が呼ばれる
# 2. C内部: rb_obj_alloc でメモリ確保
# 3. C → Ruby: initialize を呼び出し
#    KW引数の場合、{name: "Ruby", version: 3.5} のHashが一時生成される
# 4. Ruby: initialize 実行
# 5. Ruby → C: 戻り値をCの形式に変換
# 6. C → Ruby: 結果を返す

キーワード引数処理の落とし穴

特にキーワード引数の場合、C実装では致命的な問題が生じます。Cには KW引数/splat/デフォルト値の概念がないため、C実装では以下の処理が必須になります：

  1. Ruby側のキーワード引数を解析
  2. 一時的なHashオブジェクトを生成
  3. そのHashをC関数に渡す
  4. C側でHashを再度解析
  5. 個別の引数として Ruby 側の initialize に渡す

この流れでは、キーワード引数の個数が多いほど、一時的なHashオブジェクトの生成コストが増加します。10個のキーワード引数がある場合、その負担は劇的に増加するわけです。

caller ↓ Ruby [KW args parser] ↓ Temp Hash allocation ↓ C rb_class_new_instance ↓ C rb_obj_alloc ↓ C → Ruby vm_call initialize ↓ Ruby initialize execution ↓ Return

メモリ配置の非効率性

rb_class_alloc は直接GCヒープに領域確保を依頼します。この際：

  • GCロック取得
  • フリーリスト走査
  • メモリ初期化
  • メタデータ設定

これらは理想的には最適化できますが、現在の実装ではすべてが同期的に実行されるため、キャッシュミスの可能性も高くなります。

パフォーマンスへの影響 現在のC実装は「安全性」と「柔軟性」を最優先に設計されています。そのため、パフォーマンスは二の次になってしまっているのが実情です。
03

インラインキャッシュの仕組み

Rubyの高速化の鍵を握る技術の一つが「インラインキャッシュ（Inline Cache）」です。これを理解することが、opt_new命令の設計を理解するための基盤になります。

メソッド検索とキャッシング

Rubyではメソッドを呼び出す際、以下の処理が発生します：

  • レシーバのクラスを取得
  • メソッドテーブルを検索
  • 見つからなければ親クラスへ移動（lineage walk）
  • 見つかったら、その参照をキャッシュ

特に重要なのが「lineage walk」です。クラス A < B < C という階層構造で、a.baz を呼ぶときに baz メソッドが見つかるまで親クラスへ線形に辿ります。

Monomorphic Inline Cache

初回呼び出し時に見つかったメソッドの参照は、「monomorphic inline cache」として各呼び出しサイトにキャッシュされます。これにより、次回以降の呼び出しで検索が不要になります。

ただし、このキャッシュには落とし穴があります：

# キャッシュ効率が落ちるパターン
class A; end
class B; end

a = A.new
b = B.new

# 交互に呼び出すと、キャッシュが毎回ミスする
a.new  # Cache: A
b.new  # Cache miss → update to B
a.new  # Cache miss → update to A
b.new  # Cache miss → update to B
# 結果：最大40倍遅くなるケースもある！

Class#new内のinitialize呼び出しの問題

Class#new内で initialize メソッドを呼び出す際、C言語から Ruby のメソッドをコールする必要があります。このとき、単純なインラインキャッシュでは対応できない問題が生じます：

  • Cからインラインキャッシュにアクセスするには gcct_method_search を経由
  • Ruby VMのキャッシュはRubyオブジェクトとして保持されているため、C側からは直接参照できない
  • グローバルテーブルを走査する必要があり、毎回のメソッド検索が発生
YARV命令の利点 もし Class#new を YARV命令として実装できれば、Ruby VM内で直接インラインキャッシュにアクセスでき、C言語経由の余計な処理が不要になります。これが opt_new 命令導入の発想の源です。
04

YARV呼び出し規約（Calling Convention）

opt_new命令を実装するには、YARVの呼び出し規約を深く理解する必要があります。

YARVはスタックベース仮想マシン

YARVは「スタックベース」の仮想マシンです。つまり：

# Ruby: bar(5, 7, 9)

# YARVバイトコード：
push_int 5
push_int 7
push_int 9
call bar  # スタックから引数をポップして実行

メソッド側では、渡された引数を特定のスタック位置から取得します：

# Ruby: def bar(a, b, c); a + b + c; end

# YARVバイトコード：
getlocal -3    # a を取得
getlocal -2    # b を取得
add
getlocal -1    # c を取得
add

キーワード引数のオーバーヘッド

キーワード引数は本質的に「順序が不確定」です。bar(c: 9, a: 5, b: 7) のように順序が任意だからです。このため、YARVでも追加のオーバーヘッドが発生します：

  • 引数情報テーブル（キー名とデフォルト値を管理）
  • キーの検索・マッピング処理
  • 欠落引数への デフォルト値適用

ただし、Ruby 3.6以降での最適化が予定されているため、この状況は改善される見込みです。

C関数とRuby関数の不整合

C関数は KW/splat/default に対応していません。つまり、C関数を使う場合は：

  • Ruby側でキーワード引数を解析
  • それを C が理解できる形（通常はHash）に変換
  • C実行
  • 戻り値を Ruby 形式に変換

この変換が毎回発生するため、パフォーマンスが低下するわけです。

initialize は private method

重要な設計上の制約として、initialize は private method です。BasicObject には send メソッドがないため、直接呼び出す必要があります。これも opt_new 命令の設計に影響を与えます。

YARV命令設計の大原則 opt_new 命令は、Ruby VM内で完結する処理として設計されなければなりません。C言語への呼び出しを最小化し、インラインキャッシュを効果的に活用することが鍵です。
05

解決策 — opt_new 命令とインライン化

これまでの問題を根本的に解決するため、Aaron Patterson と Ruby コアチームが導入した解決策が opt_new YARV命令です。

opt_new 命令の基本設計

Feature #21254 と PR #13080 で提案・実装されたこの命令は、以下の特徴を持ちます：

  • Class#new をYARV命令として直接実装
  • パラメータをスタックに残したまま initialize を直接呼び出し
  • Class#new のスタックフレーム push/pop を省略
  • 各呼び出しサイトにインラインキャッシュを配置

「Ruby without Ruby」の発想

opt_new の実装の背景には「Ruby without Ruby」という概念があります。これは：

  • YARVバイトコードをコピーして、そのまま実行
  • 言語境界を越えない → C呼び出しなし
  • インラインキャッシュの効果を最大化

つまり、Rubyのセマンティクスは保ちながら、C言語の負荷を排除するということです。

Primitive.send_delegate! の追加

opt_new 命令が正しく動作するために、以下の Primitive が追加されました：

class Class
  def new(...)
    # opt_new YARV命令により、以下の処理を直接実行
    obj = allocate
    obj.initialize(...)
    obj
  end
end

monkeypatch への対応

Ruby の柔軟性を損なわないよう、allocate がmonkeypatchされている場合には以下の動作を実装しました：

  • オプティミゼーション候補をチェック
  • allocate が上書きされていない場合のみ opt_new を使用
  • 上書きされている場合は slow path にフォールバック

このため、既存コードの互換性を完全に保ちながら、最適化の恩恵を得られます。

従来的な流れ: caller ↓ [Class#new frame] (C実装) ↓ [initialize frame] (Ruby) ↓ 結果返却 opt_new を使った流れ: caller ↓ [opt_new + initialize inlined] (YARV) ↓ 結果返却 フレームが1つ減る！ 関数呼び出しオーバーヘッドが削減される！
互換性の担保 opt_new 命令は完全な互換性を保つよう設計されています。allocate や initialize をオーバーライドしているコードは、引き続き正しく動作します。
06

ベンチマーク結果

opt_new 命令の実装の効果を、複数の環境で測定しました。結果は期待以上のものでした。

パフォーマンス改善の詳細

シナリオ 改善率
（インタプリタ）		 改善率
（YJIT）
Object.new（位置引数） 1.8x —
3個のキーワード引数 3.0x 6.5x
10個のキーワード引数 6.2x+ さらに高速
Hash.new(capacity: 0) 1.72x —

改善率が引数数に比例する理由

特に注目すべきは、キーワード引数が増えるほど改善率が大きくなる点です。その理由は：

  • 従来の実装：一時的なHash生成 → Cへの変換 → Ruby側で再度解析
  • opt_new 実装：YARV内で直接処理 → Hash生成が不要

つまり、引数が多いほど Hash 生成のコストが大きくなるため、その削減がより顕著な改善につながるのです。

メモリ影響の分析

項目 従来実装 opt_new実装
allocateオブジェクト数 2個 1個
命令シーケンスサイズ ベース +0.5%
呼び出しサイトあたりのコード — 544 → 656 byte

重要な点として、メモリ使用量の増加は「呼び出しサイト」あたりであり、コード全体に占める割合はわずか 0.5% です。この小さな増加で得られる大幅なパフォーマンス改善は、明らかに価値があります。

YJIT との相乗効果 Ruby 3.5 の YJIT コンパイラと opt_new 命令が組み合わさると、キーワード引数のケースで 6.5 倍の高速化が実現します。これは、YJIT がインラインキャッシュを完全に削除して、単一の型バージョンに特化したコードを生成できるようになるためです。
Rails への波及効果 Rails のモデル操作では頻繁に new が呼び出されます。キーワード引数による初期化が一般的なため、この 6.5 倍の改善は、Rails アプリケーション全体のレイテンシー削減に大きく貢献するでしょう。
07

トレードオフと注意点

opt_new 命令の導入には多くのメリットがありますが、いくつかのトレードオフも存在します。

Stack Trace の変化

opt_new によってインライン化されるため、スタックトレースから Class#new が消えます。

# 従来の実装
from app.rb:10:in `new'
from app.rb:5:in `create_object'

# opt_new 実装
from app.rb:5:in `create_object'
# ← Class#new がバックトレースから消える

この変化は：

  • 一般的な利用：影響なし。エラー箇所の特定は十分に可能
  • デバッグ時：若干の手間が増える可能性
  • メタプログラミング：Class#new のフレームに依存していた場合は修正が必要

Ruby 3.5 への移行時に、既存テストで new をスタック最上位に期待している場合は、テストの修正が必要になる場合があります。

メモリ使用量の微増

前述の通り、呼び出しサイトあたりのコードサイズが 544 byte から 656 byte に増加します。ただし：

  • 全体への影響：+0.5%（許容範囲内）
  • 実メモリ効果：ほぼ無視できるレベル
  • キャッシュ効率の改善で相殺される可能性
Stack Trace への依存性の確認が重要 カスタムのエラー追跡システムやメタプログラミングで Class#new のフレームに明示的に依存している場合は、Ruby 3.5 への移行前に確認が必須です。例えば、「n フレーム目は常に new」というような実装があれば修正が必要になります。

デバッグビルドへの影響

デバッグビルドやプロファイリングツールによっては、opt_new 命令が正しく対応されていない可能性があります。Ruby 3.5 のリリースに向けて、主要なデバッグツールの対応が進んでいます。

08

Ruby 3.5〜4.0 での展開

opt_new 命令の導入は、Ruby の進化ロードマップの中でも重要な位置を占めています。

Ruby 3.5 での実装（2025年5月予定）

  • opt_new YARV命令が実験的に導入
  • インタプリタ環境での 1.8x ～ 6.2x の改善
  • ほぼすべてのクラスの new で恩恵
  • 互換性：100% を維持

Ruby 4.0 での統合（2025年12月予定）

Ruby 4.0 では、さらに積極的な統合が予定されています：

  • YJIT との統合：単一型のコードを生成し、6.5x 以上の改善を実現
  • ZJIT との統合：Rust ベースの新しい JIT コンパイラとも統合
  • 公式声明："Class#new is faster in all cases, but especially when passing keyword arguments"

今後の拡張可能性

opt_new 命令の成功により、他の類似の最適化も視野に入ってきています：

  • attr_reader/attr_writer のインライン化
  • 他のメタプログラミング関連の最適化
  • GC プレッシャーの削減
長期的な戦略 Ruby コアチームの戦略は明確です：「遅くて柔軟」から「速くて柔軟」へ。opt_new はその第一歩に過ぎず、今後も同様の最適化が続いていくでしょう。
09

まとめと考察

RubyKaigi 2025 で発表された Class#new の最適化は、Ruby の設計哲学そのものへの深い理解を示す革新的なセッションでした。

逆説的な結論

最も興味深いのは、このセッションが示した逆説的な結論です：

「C言語で実装した方が速い」という常識は、言語境界の負荷を考慮していない。同じセマンティクスであれば、Ruby（YARV）で実装した方が速いこともある。

この洞察は、Ruby が単なるスクリプト言語から、パフォーマンスも兼ね備えたシステムプログラミング言語へと進化していることを示しています。

技術的な教訓

  • 言語境界のコスト：最適化を考える際、言語間の呼び出しオーバーヘッドは常に念頭に置くべき
  • インラインキャッシュの力：YARV VM 内のキャッシングメカニズムはきわめて強力
  • 互換性の大切さ：Ruby の柔軟性（monkeypatch など）を損なわずに最適化する工夫

Railsエコシステムへの期待

Rails コミュニティにとって、この最適化は特に大きな意味を持ちます：

  • モデルインスタンス生成時の 6.2x 高速化
  • 関連オブジェクト（association proxy など）の生成の高速化
  • システム全体のレイテンシー削減

Ruby on Rails のようなフレームワークこそが、言語レベルの最適化により大きな恩恵を受けるのです。
