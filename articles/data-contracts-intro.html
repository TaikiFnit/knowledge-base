Part I Overview

Part I 概要

Part I（第1〜4章）は、データコントラクトの「なぜ」と「何か」を確立する。データ品質を劣化させてきた歴史的な力を追跡し、データ品質が本当に何を意味するかを定義し、データインフラをスケールさせる際の組織的・技術的課題を検討し、解決策としてデータコントラクト・アーキテクチャを導入する。

// Part I の中心的主張

「Data contracts—agreements between data producers and consumers that are established, updated, and enforced via an API—are necessary for scaling and maintaining data-related change management within an organization.」

データ問題は本質的に複雑なシステム内の変更管理問題である。データコントラクトは、組織がスケールしてもその変更管理を維持するために必要なメカニズムだ。

01
Ch.1 / GIGO Cycle

Ch.1: Garbage-In, Garbage-Out サイクル

第1章の出発点は、業界がいかにしてデータ負債（data debt）という慢性疾患を蓄積してきたか、という問いだ。ビッグデータ・クラウドコンピューティング・モダンデータスタックの台頭は、データの利活用を加速させた反面、制約の欠如による深刻な負の遺産を生んだ。

データ負債（Data Debt）の定義

"Data debt refers to the outcome of incremental technology choices made to expedite the delivery of data infrastructure at the expense of its long-term trustworthiness." — Chad Sanderson et al., Data Contracts (O'Reilly, 2025) Ch.1

技術的負債（tech debt）に類似しているが、データ負債には2つの重要な違いがある：

Tech Debt（技術的負債）
速度 vs 保守性のトレードオフ
個別マイクロサービスを独立してリファクタリング可能
コードは可視化・差分確認が容易
影響範囲の特定が比較的容易
Data Debt（データ負債）
速度 vs 信頼性のトレードオフ ← より深刻
緊密に結合した依存関係により解消が困難
サイレントな失敗（誤った出力が検出されない）
広範な組織的合意なしには変更不可能
// データ負債の悪循環

「Data debt is a vicious cycle. It depreciates trust in the data, which attacks the core value proposition of the data team. And unwinding data debt is not easy—not even close.」

データ負債は信頼を侵食し、信頼の低下はデータチームの存在意義を脅かす。解消はソフトウェアのリファクタリングより遥かに困難だ。

GIGO（Garbage-In, Garbage-Out）サイクル

GIGO とは、ステークホルダーの期待を満たさないデータがパイプラインに入り込む状態を指す。GIGOが特に問題なのは、事後的にしか対処できないという点だ。つまり問題が発生した後に必ずコストが生じる。

GIGOの影響は時間とともに指数関数的に増大する。修正が積み重なるにつれ、特定の個人にinstitutional knowledge hotspot（制度的知識の集積点）が形成される。この個人が退職すると、そのコンテキストを取り戻すことは極めて困難になる。ソフトウェア開発者に対するデータ開発者の比率が増加するにつれ、GIGOによるデータ負債は指数関数的に増加する。

// ケーススタディ：Convoy（物流スタートアップ）

Convoyのエンジニアリングチームは、Spotifyの組織モデルから採用した「ポッド（pod）」に分割されていた。各ポッドは独立して意思決定し、異なる問題に集中していた：

• ML モデルでカスタマーサービス問題の優先順位付けを行うポッド
• コア価格モデルを担当するポッド
• 最大パートナーへのリアルタイム出荷ETAアナリティクスを担当するポッド

ポッド間のコミュニケーションは最小限だった。その結果、あるポッドのソフトウェアエンジニアがカラム名の変更・カラムの削除・ビジネスロジックの変更・イベントの停止などを行っても、データコンシューマーには何も伝わらなかった。

重要な観察：スタートアップが約20名以下のエンジニアの段階では、データエンジニアはまだ複数のサイロ化されたドメインに分裂していない。20名を超え始めると、このコミュニケーション構造は機能しなくなる。

02
Ch.1 / Evolution

Ch.1: データアーキテクチャの進化

現代のデータ品質危機を理解するには、どのように今日の状況に至ったかを把握する必要がある。第1章は、プレモダン時代からデータ中心AI時代まで、アーキテクチャの歴史的変遷を丁寧に辿る。

// データアーキテクチャの歴史的進化
  • 01
    Pre-2000s — プレモダン時代

    オンプレミス データウェアハウス時代

    Oracle・IBM DB2・Microsoft SQL Server などのオンプレミスDWH。高価な ETL ツール（Informatica, SSIS）。データアーキテクトが ERD（Entity Relationship Diagram）を設計し、論理的・意味的なビジネス構造を表現していた。Bill Inmon の4原則（主題指向・統合・時系列・非揮発性）に基づくデータウェアハウスが単一の真実ソース（SSOT）として機能した。

  • 02
    2000s–2010s — Software Eats the World

    アジャイル・クラウド変革時代

    Marc Andreessen が「ソフトウェアが世界を食べている（Software Is Eating the World）」と宣言（WSJ, 2011）。Thoughtworks が普及させたアジャイル手法、Mark Zuckerberg の「Move fast and break things」文化。AWS・Azure・GCPがクラウドインフラを民主化。データアーキテクチャは意図的に軽視された——ERD設計に数ヶ月を費やすことはアジャイルの「移動速度」と相容れなかったのだ。

  • 03
    2010s — マイクロサービス時代

    モノリス → マイクロサービスへの分解

    「Microservices」は Thoughtworks コンサルタントが初めて使用し、Martin Fowler のブログで広まった。小規模・自律的・独立してデプロイ可能なサービスへの分解。データアーキテクチャのサイレントな犠牲者：マイクロサービス全体にデータが分散し、ガバナンスのないデータレイクが台頭。これが「data swamp（データ沼）」を生んだ。

  • 04
    2012–2022 — モダンデータスタック時代

    Snowflake・dbt・Fivetran の時代

    Modern Data Stack (MDS)：Snowflake（クラウドベースSQL変換・ストレージの市場制覇）、Google BigQuery・Amazon Redshift・Databricks、ELTツール（Fivetran・Airbyte）、dbt（バージョン管理・テスト・ドキュメント付きSQL変換CLI）。プロダクト主導型成長（Product-Led Growth, PLG）とVC投資がMDSの急速な普及を促した。

    問題の顕在化：①単一の真実ソースなし ②プロデューサーとコンシューマーの断絶 ③データへの信頼の喪失 ④コストの急騰。「Data teamはかつてないほど大規模になり、かつてないほど高コストになり、そして議論の余地なくかつてないほど信頼されなくなった。」

  • 05
    Post-2020 — データ中心AI時代

    Andrew Ng の Data-Centric AI とシフトレフト

    Andrew Ng の 2021 年キャンペーンによりData-Centric AIが広く受容される。モデル中心アプローチ（モデルのチューニング・計算資源の増強）では限界があり、データの品質と選定を改善することでMLモデルの性能を向上させるアプローチが優位に。MLベンダーは 2014 年の約5社から 2024 年には 200 社超へ急増（Matt Turck 年次データベンダーレポート）。競争優位は今やデータの品質と独自性に集中する。

「アクティブ顧客」問題 ——SQLの実例

アーキテクチャ進化の実害を示す典型例として、本書は「今月のアクティブ顧客数は？」というシンプルなクエリが、上流データの変更によりどれほど複雑になるかを描写する。

初期クエリ（シンプル）：「訪問（visit）」はユーザーがアプリを開いた回数として定義されていた。

WITH visit_counts AS (
  SELECT customer_id, COUNT(*) AS visit_count
  FROM visits
  WHERE DATE_FORMAT(visit_date, '%Y-%m') = DATE_FORMAT(CURDATE(), '%Y-%m')
  GROUP BY customer_id
)
SELECT COUNT(DISTINCT customers.customer_id) AS active_customers
FROM customers
LEFT JOIN visit_counts ON visit_counts.customer_id = customers.customer_id
WHERE COALESCE(visit_counts.visit_count, 0) >= 3;

その後、上流のエンジニアリングチームが仕様を変更：「impression（インプレッション）」＝ 任意の画面への遷移、「visit（訪問）」＝ 10秒以上の活動期間。クエリは複雑な CASE WHEN ロジックに膨れ上がる：

WITH impressions_counts AS (
  SELECT customer_id,
    SUM(CASE WHEN duration_seconds >= 10 THEN 1 ELSE 0 END) AS visit_count,
    SUM(CASE WHEN duration_seconds < 10 THEN 1 ELSE 0 END) AS impression_count
  FROM impressions
  WHERE DATE_FORMAT(impression_date, '%Y-%m') = DATE_FORMAT(CURDATE(), '%Y-%m')
  GROUP BY customer_id
  HAVING (visit_count + impression_count) >= 3
)
SELECT COUNT(DISTINCT customers.customer_id) AS active_customers
FROM customers
LEFT JOIN impressions_counts ON impressions_counts.customer_id = customers.customer_id
WHERE COALESCE(impressions_counts.visit_count, 0) >= 3;

上流変更が重なるごとにクエリは肥大化し、CASE文や WHERE句が存在する理由のコンテキストはすべて失われる。新しいデータ開発者は独自の目的のために「車輪の再発明」を行い、コアメトリクスの定義が組織内で増殖し続ける。これがデータ負債の典型的な発生メカニズムだ。

Shift Left Data — 解決策への予兆

Shift Left Dataとは、データパイプラインにおいて、品質・ガバナンス・コントラクトの適用をより上流（データが生産されるソフトウェアエンジニアに近い側）に移動させる動きを指す。この概念が第4章のデータコントラクト・アーキテクチャの前提となる。

03
Ch.2 / Data Quality

Ch.2: データ品質の再定義

第2章の核心的洞察は、データ品質の根本的な誤解を解くことにある。多くのデータキャリアの初期の過ちは、「完璧なデータ」の価値でデータ品質を売り込もうとすることだ。データ実践者以外のほとんどの人々はデータ自体には関心がない——彼らが関心を持つのは、データで何ができるかだ。

データ品質の定義の変遷

データの歴史的初記録は紀元前19,000年まで遡るが、本書は1970年に Edgar F. Codd の論文「A Relational Model of Data for Large Shared Data Banks」によってリレーショナルデータベースの規律が生まれた時点から論じ始める。

// データ品質の定義の進化
Wang et al. (1996)
「データコンシューマーが使用するのに適したデータ」
MIT データ品質研究グループ、最も引用される定義
→
拡張
本書の定義 (2025)
「データライフサイクル全体において、データ資産の正確性の度合いを理解し、コンシューマーの使用目的に適合する度合いで、そのデータを運用する際のトレードオフを把握する組織の能力」

本書の定義の重要なフレーズを分析しよう：

①
Key Phrase 1

「様々な正確性の度合いでのトレードオフ」

NoSQL の台頭（1998年 Carlo Strozzi が造語、2009年 Johan Oskarsson が普及）以来、リレーショナルデータベース以外のデータ保存方法が急増し、複雑さとトレードオフが増大した。データチームが行うトレードオフを意識することが個人の行動変容の鍵となる。

→ 完璧主義より戦略的トレードオフ意識
②
Key Phrase 2

「正確性の度合いを理解する能力」

完璧なデータが必要だという信念が陥穽だ。データは絶えず劣化しており、永遠に完結することのない継続的なモニタリングと反復が必要。健康指標の計測（血圧計・体重計・体温計）は本質的に不正確だが、予防医療においては極めて有用だ。

→ 完璧より「十分に良い」を目指す
// 核心的洞察

「Central to this chapter is the idea that data quality is not about data being pristine, but instead being 'fit for use' for your stakeholders. At the end of the day, technical and data debt should ideally be a strategic trade-off where you are balancing effort and robustness with ROI.」

データ品質問題のフレーミングを「技術的完璧さ」から「重要なビジネスワークフロー（多くは収益に直結する）へのリスク」へと転換することで、経営層の関心を引きつけることができる。

OLTP と OLAP の断絶 ——翻訳問題の根源

多くのデータ品質問題の根底には、OLTP（Online Transaction Processing）とOLAP（Online Analytical Processing）のサイロが存在する。

観点 OLTP（オンライントランザクション処理） OLAP（オンライン分析処理）
最適化対象 高速データトランザクション・現在状態の維持 大量データのスキャン・統計計算
主要操作 CRUD + ACID準拠 + 第三正規形（3NF） 非正規化データ・複雑なビジネスロジック・反復的探索
主要ペルソナ ソフトウェアエンジニア（データの生産者） データアナリスト・データサイエンティスト・MLエンジニア
限界 複雑な分析クエリで本番DBが停止するリスク OLTPからのレプリケーションで複雑性と誤解が増大

OLTPからOLAPへのデータレプリケーションは、組織のデータ理解において大きなメリットをもたらす一方、それぞれの「データ世界観」にサイロを生み出し、誤解を招く。

// ケーススタディ：Kelp（架空の水族館レビューサイト）

Kelpのエンジニアリングチームが「レビューの更新」機能をリリース。データアナリストはこの機能がユーザーセッション時間を増加させるかを分析しようとした。

中心的な問い：「平均レビュー星数はどう計算するか？」

OLTP（3NF）では Aquarium テーブル・User テーブル・Reviews テーブルが分離されている。アナリストがOLAP用の非正規化ワイドテーブルを作成した時点で、解決不能な問いが生まれる：

• 近接した複数セッションをまたぐ場合、セッション時間はどう計算するか？
• 一人のユーザーが複数アカウントを持つ場合は？
• ビジネスにとって関連するセッション時間の変化はどの程度か？
• セッション時間変化を引き起こすのはユーザーレビューそのものか、それとも複合的な属性か？

これらの問いは、OLTP データにビジネスロジックが適用されなくなった状態を示す。正解は機能を最初に構築したソフトウェアエンジニアとのコミュニケーションなしには判断できない。これがデータコントラクトが重要な理由の核心だ。

04
Ch.2 / Measuring Quality

Ch.2: データ品質の測定

データ品質を測定するには、leading indicators（先行指標）とlagging indicators（後行指標）の両方が必要だ。先行指標は将来の問題を予測し、後行指標は損害を計測する。

1
Leading / 先行指標

データ負債（Data Debt）

データ環境の複雑性とスケール可能性の指標。測定ヒューリスティクス：ドキュメントが存在するデータ資産の割合、1データセットあたりの平均依存数、直近1年のバックフィルジョブ数、クエリ1件あたりの平均フィルタ数。

→ スケール可能性の予測指標
2
Leading / 先行指標

信頼性（Trustworthiness）

下流チームがデータ資産を意思決定やモデルに使用する可能性と強く相関。四半期サーベイで「会社のデータを信頼しているか」「使用中のデータが現実を正確に反映しているか」等を計測。複製されたデータ資産数も信頼の代理指標。

→ データ活用率への影響が大
3
Leading / 先行指標

オーナーシップ（Ownership）

上流のエラーを解決する可能性と速度を予測する指標。オーナーシップの欠如がビジネスインパクトを生み出す場合に特に有効。Ops ミーティングで提起すると議論を呼ぶ指標だ。

→ インシデント解決速度の予測
4
Leading / 先行指標

期待値の違反（Violated Expectations）

データコンシューマーが上流データシステムに期待することの測定。4形態：①スキーマ（データ構造・型・カラム名）②セマンティクス（ビジネスロジック）③SLA（レイテンシ・ボリューム要件）④PII（個人識別情報のマスク要件）

→ ブレイキングチェンジの予防
5
Lagging / 後行指標

データダウンタイム（Data Downtime）

データエンジニアリングチームでデータ品質を定量化する最も一般的な指標。ログ：全インシデントの記録（期間・原因・解決プロセス）。計算：平均ダウンタイム時間・頻度・ビジネスオペレーションへの影響（遅延レポート・誤分析・誤意思決定）。

→ 損害の可視化・過去分析
6
Lagging / 後行指標

四半期インシデント数

時系列で「ビジネスインパクト」が最も高いインシデントを追跡。ソフトウェアと同様に重大度スケール（P0〜P3）で分類。実世界への影響があるインシデントが最も意味のある種類。

→ ビジネスリスクの可視化

誰がデータ品質問題の影響を受けるか

ペルソナ 影響
データエンジニア 全データ問題のデフォルト「オーナー」として機能。オンコールのバックログが積み上がる。アナリティクス/データサイエンスチームとの緊張が高まる。
データサイエンティスト 欠陥データでMLモデルを数ヶ月訓練してから問題を発見。学術的バックグラウンドがデータの信頼性への過信につながる。
データアナリスト ビジネスロジックが混乱。信頼していたデータソースが予期せず変化。ステークホルダーへの誤った推奨を行う。
ソフトウェアエンジニア 通常は上流の原因（意図的ではない）。変更が適切なデータコンシューマーに届かない。
ビジネスチーム 最終的な消費者。問題の特定・解決への主体性がほぼない。上流のエラーの帰結を最も大きく受ける。
05
Ch.3 / Scale Challenges

Ch.3: スケーリングの課題

第3章は、データ開発がソフトウェア開発とどのように異なるかを詳しく解説し、その違いが組織のデータ価値創出能力にどのような意味を持つかを論じる。また、スケールに伴いデータシステムが複雑化する際の根本的な課題を掘り下げる。

ソフトウェア開発 vs データ開発

SOFTWARE DEVELOPMENT
既知のユーザー体験から開始
要件定義 → 技術仕様 → ビルド
最重要事前作業：要件収集・アーキテクチャ設計
チーム内変更管理：バージョン管理・PR・コードdiff が確立
DATA DEVELOPMENT
ビジネスの問いから開始（「今月のアクティブ顧客数は？」）
データ探索 → クエリ構築 → 検証 → 安定化
最重要事前作業：仮説の定式化・データの理解
クロスチーム変更管理：確立されたシステムが存在しない

データ開発では、データが存在するか・どこにあるか・誰がオーナーか・何を意味するか・アクセス権限があるか、といったdata discovery（データ発見）の問いに答えるだけで、1つの問いあたり数時間〜数日を費やすことがある。

大規模リファクタリングのコスト

Martin Fowler のリファクタリング定義：「観察可能な動作を変えずに、より理解しやすくより変更しやすくするためにソフトウェアの内部構造を変える変更。」

// リファクタリングコストの統計（James Ivers et al. の調査）
1,500
大規模リファクタリング1件あたりの平均スタッフ日数
$663K
平均給与 $115,000・260 勤務日/年で換算した労務費
5年
Alan社がリファクタリングを先延ばしにした期間（その間、欠陥データモデルの上に技術的負債が積み上がった）
// ケーススタディ：Alan（フランスの医療保険会社）の大規模リファクタリング

Alan は 2016 年創業の医療保険会社。データモデルの中心的な前提：ユーザーと健康保険契約が1対1の関係（フランスでは標準だが、国際展開では成立しない）。国際展開のため、複数契約ユーザーをサポートするリファクタリングが必要になった。

Alan の7ステップ大規模リファクタリング：
1. 変更が必要な箇所の特定（コードベース全体から1対1前提を探索）
2. 変更内容の決定（コアデータモデルの変更が最重要と判断）
3. 変更の実装（製品ロードマップへの優先付けと専任クロスファンクショナルチームの結成）
4. 新規テストの生成と既存テストの移行
5. リファクタリングコードの検証
6. 再認定（ヘルスケアコンテキストでのHIPAA等コンプライアンス要件）
7. ドキュメントの更新（技術的問題ではなく、エンジニアリングチームを超えるビジネス問題）

欠陥データモデルの上に5年間の技術的負債が積み重なり、リファクタリングは極めて困難なプロジェクトとなった。

コンセプトドリフトの4類型

データとその理解の変化（concept drift）がモデルパフォーマンスとデータ品質に影響を与える。Jie Lu et al.「Learning Under Concept Drift: A Review」の分類体系：

TYPE 01

突発的ドリフト（Sudden Drift）

abrupt change in data distribution

例：GDPR コンプライアンス対応でユーザーデータの削除・変更が必要になり、EU加盟国ユーザーがトラフィックから消失。

TYPE 02

漸進的ドリフト（Gradual Drift）

slow shift in distribution over time

例：Netflix が DVD販売からストリーミングへ移行し、最終的にDVDを廃止（製品ターゲット顧客が徐々に変化）。

TYPE 03

段階的ドリフト（Incremental Drift）

step-by-step change in distribution

例：Roblox の当初のユーザー層は幼い子供だったが、現在の中央年齢は13歳超（ユーザーベースが段階的に高齢化）。

TYPE 04

周期的ドリフト（Recurring Drift）

periodically recurring patterns

例：景気後退は定期的に発生するが、正確な予測は不可能。消費者組織データにおける任意消費支出の減少は周期的なコンセプト。

Dunbar の数と Conway の法則

組織がスケールするにつれてデータシステムがなぜ機能不全に陥るかを、人類学と組織論の2つの原理が説明する。

Dunbar の数（~150）

人類学者 Robin Dunbar が提唱。人間が維持できる有意義な関係の認知上限は約150。Malcolm Gladwell の『The Tipping Point』でも引用。

Meta の CPO Chris Cox（2016）：「150人を超えると、おかしなことが起き始める」。150名の組織では11,175の潜在的関係が存在し、1名追加するごとに150の新たな関係が生まれる。Martin Fowler：「コードの著者と容易に話せれば、コードへの豊かな理解を構築しやすい」。

Conway の法則

Melvin Conway「How Do Committees Invent?」：「システムを設計するいかなる組織も、その組織のコミュニケーション構造のコピーである設計を不可避的に生み出す。」

データシステムは組織のサイロを反映する。これらの法則が共に示すのは、複雑なデータシステムのスケールは人的コーディネーションだけでは達成できないということだ。データコントラクトはこの限界を技術的に克服するメカニズムとなる。

// スケーリング問題への5つの要件

効果的なデータ変更管理フレームワーク（design surface）には4つの要件が必要だ：①データの存在と使用方法の可視性、②データプロデューサーとコンシューマーの接続、③変更通知のタイムリーな伝達（「ブレイキングチェンジ直前」が最適）、④適切なステークホルダーの特定。

そして5要件のデータ変更管理：①ドメイン知識のコード化、②意味のある制約の作成、③データドリフトの識別、④閾値超過時のアラート、⑤ドリフト後のデータ品質修正またはビジネスロジックの更新。

「これら5つの要件こそが、データコントラクトがスケールにおけるデータ品質を実現するために必要なメカニズムだと我々が強く信じる理由だ。」

06
Ch.4 / What Are Data Contracts?

Ch.4: データコントラクトとは何か

前3章が「なぜ」を論じてきたのに対し、第4章はデータコントラクト・アーキテクチャが具体的に何であり、どのように機能するかを定義する。本書の技術的基盤となる章だ。

ステークホルダーの定義

DATA PRODUCERS（データプロデューサー）
データの収集・保存を担当するエンジニア。最も一般的な形態はソフトウェアエンジニア
トランザクショナルイベント（PostgreSQL・MySQL・Kafka）を生成
クリックストリームイベント（Segment・Rudderstack・Amplitude）を生成
リレーショナルDBをアプリケーションの延長として扱う
現代のデータ環境では、データが収集・処理・保存された後、役割が終了
DATA CONSUMERS（データコンシューマー）
データエンジニア：データの抽出・移動・パイプライン構築
データサイエンティスト：ML・統計モデリング
データアナリスト：SQL専門家、ダッシュボード・レポート
アナリティクスエンジニア：ソフトウェア開発の最善実践をデータ分析に適用（dbtモデル等）
プラットフォームエンジニア：データインフラの実装・採用・保守

コンシューマーが退職すると、データに関して積み上げられたコンテキストを取り戻すことは極めて困難だ。これがdata lineage（データリネージュ）——分析環境内でデータ資産を相互に結びつける蜘蛛の巣状の接続——が重要な理由の一つだ。

11ステップ 反応的解決ライフサイクル vs 7ステップ コントラクトワークフロー

// 現状：11ステップの反応的解決ライフサイクル

「これらの数字が少し正常ではないように見えると思います」という言葉はデータ実践者の恐怖だ。典型的な解決プロセス：

①ステークホルダーが問題を発見 → ②問題のトリアージ → ③要件スコーピング → ④問題の再現 → ⑤データプロファイリング（タイムライン・null パターン・スパイク/ドロップ・集計別カウント） → ⑥下流パイプライン調査 → ⑦上流パイプライン調査 → ⑧技術ステークホルダーへの相談 → ⑨デプロイ前：DQ修正の実装 → ⑩デプロイ：DQ修正の適用 → ⑪ステークホルダーへのコミュニケーション

「これらのステップの大多数は技術的なものではなく、チームをまたいだコミュニケーションを中心に展開する。」このプロセスは依然として手動的・反応的だ。

// 代替案：7ステップ データコントラクト ワークフロー（プロアクティブ）

①データコンシューマーがデータ制約を特定 → ②データコンシューマーがデータ資産のコントラクトをリクエスト → ③データプロデューサーがコントラクトの実現可能性を確認 → ④コントラクトがコードとして確認（通常 YAML ファイル、バージョン管理） → ⑤データプロデューサーがデータ資産変更のプルリクエストを作成 → ⑥CI/CD チェックが要求変更がデータコントラクトに違反するか自動検証 → ⑦（7a）CI/CD 失敗：データ資産オーナーに違反通知・失敗プロトコルに従う（7b）CI/CD 成功：変更が下流プロセスに適用

鍵となるシフト：データコントラクトワークフローは、データ品質解決プロセスを反応的からプロアクティブへ移行させ、各ステークホルダーのインセンティブと責任を根本的に変える。

データコントラクト vs データオブザーバビリティ

観点 データコントラクト データオブザーバビリティ
主目的 特定のデータ品質問題の防止 データ品質トレンドの検出・可視化
CI/CDとの関係 CI/CD ワークフローに組み込まれる（デプロイ前にブレイキングチェンジを防止） CI/CD ワークフローを補完する（開発者ワークフロー外でデータプロセスを測定）
ビジネスロジック ビジネスコンテキストが入力（プロデューサーとコンシューマー間のコミュニケーションをスケールさせる技術） ビジネスロジックがデータにどう捉えられるかを反映する（出力の測定）
可視性の範囲 ターゲット可視性：最重要データ資産のみ 広範な可視性：データスタック全体（インジェスト〜消費まで）
アラートのタイミング 変更前にアラート：プロデューサーとコンシューマーの合意を暗黙から明示へ（本番デプロイ前） 変更後にアラート：イベントが観測可能になった後（発生後の検出が必須）
アナロジー スポットライト（重要な資産に焦点を当てた光） 懐中電灯（データシステム全体を照らす）
"The key difference between data contracts and data observability is that contracts emphasize the prevention of data quality issues, while observability emphasizes detection." — Chad Sanderson et al., Data Contracts (O'Reilly, 2025) Ch.4

データコントラクトの成熟度曲線

データコントラクトの採用は本質的に社会的・組織的課題であり、純粋に技術的なものではない。成熟度は組織間だけでなく、組織内のチーム間でも異なる。

  • P1
    Stage 1 — Awareness（認識）

    目標：下流チームが上流データをどう使用するかの可視性を確立

    データプロデューサーは自分の変更がコンシューマーに害を与えることを理解する必要がある。プロデューサーはアウテージを起こしたいわけではない——デプロイ前に何が壊れるかを知るコンテキストがないだけだ。

  • P2
    Stage 2 — Collaboration（協働）

    目標：コントラクトによるソースでのデータ保護

    「この協働がデプロイ前、コンテキストによって促進される形で行われることの重要性はいくら強調してもし足りない。プロデューサーが事前にコミュニケーションし、コンシューマーが何が変わるかの可視性を持てば、ほとんどのデータ品質問題は回避できた。」

  • P3
    Stage 3 — Ownership（オーナーシップ）

    目標：データ資産の明確なオーナーシップとアカウンタビリティの確立

    プロデューサーがユースケースとコンシューマーを理解し、アクセス可能なフィールドを明示的に定義し、変更管理・コントラクトバージョン管理・コントラクト進化のプロセスを確立した段階。「Visibility creates culture change.」

07
Part I Summary

まとめ — Part I の要点

Part I を通じて明らかになった核心的な洞察を整理する。

  • 01
    データ問題は変更管理問題である

    データの品質問題は表面上は技術的に見えるが、その根本はすべてchange management within complex systems（複雑なシステム内の変更管理）に行き着く。コードの変更がデータコンシューマーに届かないこと、届く仕組みがないことが問題の本質だ。

  • 02
    データ負債はテック負債より解消が困難

    テック負債は個別マイクロサービスを独立してリファクタリングできる。しかしデータ負債は相互に依存した事業体の上に構築されており、広範な組織合意と多大なコーディネーションなしには変更できない。Alan 社の事例は、欠陥データモデルの上に5年間の負債が積み上がり、約663,000ドルのリファクタリングコストが発生することを示す。

  • 03
    データ品質 ≠ 完璧なデータ

    Wang et al. の「利用目的に適合すること（fit for use）」は今なお正しい。本書はこれを「様々な正確性の度合いでデータ資産を運用する際のトレードオフを理解し管理する組織の能力」へと拡張する。データは常に劣化しており、完全な完璧さは非現実的な目標だ。

  • 04
    OLTP/OLAP の断絶が翻訳問題を生む

    OLTPの3NFとOLAPの非正規化された世界観の間に存在するギャップは、データチームが単独では解決できない解釈の問題を生み出す。Kelp のケースが示すように、「平均レビュー星数をどう計算するか」という問いは、最初にその機能を構築したソフトウェアエンジニアとのコミュニケーションなしには答えられない。

  • 05
    Dunbar の数と Conway の法則がスケールを阻む

    150人を超えると人間の関係維持能力は限界を迎え（Dunbar の数）、組織のシステムはその通信構造を反映する（Conway の法則）。これらの組み合わせは、スケールした組織ではデータシステムの変更管理が人的コーディネーションだけでは維持できないことを意味する。

  • 06
    コントラクトはオブザーバビリティを置き換えない、補完する

    データオブザーバビリティ（懐中電灯）はデータスタック全体を広く照らすが、イベントが発生した後にのみアラートを出す。データコントラクト（スポットライト）は最重要データ資産に焦点を当て、本番デプロイ前に問題を防ぐ。組み合わせることで最大の効果を発揮する。

  • 07
    シフトレフトは文化変革である

    データコントラクトの採用は技術的な実装だけではなく、データプロデューサーとコンシューマーの間のコラボレーション文化の変革を要求する。可視性が文化変革を生む（"Visibility creates culture change."）。成熟度は認識 → 協働 → オーナーシップと段階的に進む。

// Part I の本質的なメッセージ

データ品質問題は検査（inspection）の問題ではなく、予防（prevention）の問題だ。業界は反応的（オブザーバビリティ）なアプローチからプロアクティブ（コントラクト）なアプローチへとシフトする必要がある。

「Most data quality problems are code quality problems.」

上流のコードの変更が、通知なしに下流のデータ消費者に影響を与える。データコントラクトはこの暗黙の依存関係を明示的・強制可能なAPI契約へと変換することで、ソフトウェア指向の変更管理をデータチームへと拡張する。これがデータコントラクトの存在意義だ。
